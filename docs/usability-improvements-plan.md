# PlexLoader - UX improvement plan

Цель плана - не потерять UX-направление: сделать PlexLoader не просто мощным
ботом, а понятным ассистентом, который ведёт пользователя и всегда объясняет,
что происходит.

## Главный принцип

В любой момент пользователь должен понимать:

1. Что я попросил у бота.
2. Что бот сейчас делает.
3. Что будет дальше.
4. Нужно ли действие от меня.

Если экран или уведомление не отвечает хотя бы на эти вопросы, UX считается
недоделанным.

## Приоритеты

### В работе: первый UX-пакет

В работу берём первые четыре направления:

1. Центр запросов.
2. Стартовый экран с действиями.
3. Единый шаблон ошибок и восстановления.
4. Экран конкретного результата поиска.

Главная логика порядка: сначала создаём единое место, куда можно вести
пользователя после любого действия или ошибки; потом улучшаем вход в продукт;
потом приводим ошибки к одному recovery-паттерну; затем делаем выбор раздачи
понятным через детальный экран.

#### 1. Центр запросов

Проблема:

- пользователь видит загрузки в `/status`, подписки в `/subs`, уведомления в
  отдельных push-сообщениях, pending retry только косвенно;
- из-за этого непонятно, продолжает ли бот работать над просьбой или уже ждёт
  действия пользователя;
- технические сущности Download Station / Jackett / Plex не совпадают с тем,
  как пользователь думает о задаче.

UX-цель:

- дать один экран "Мои запросы", где видно всё, что пользователь поручил боту;
- каждый запрос должен отвечать на вопросы:
  - что это;
  - текущий статус;
  - что бот делает дальше;
  - когда была последняя активность;
  - нужна ли реакция пользователя.

Минимальный пользовательский сценарий:

1. Пользователь скачивает фильм из поиска.
2. Бот создаёт запрос.
3. Пользователь открывает "Мои запросы".
4. Видит фильм в статусе "Скачивается".
5. После завершения видит "Проверяем Plex".
6. После нахождения в Plex видит "Добавлено в Plex".
7. Может убрать запрос из активного списка.

Первый технический scope:

- новый state-файл `requests.json`;
- новый модуль с чистой логикой request-модели и статусов;
- state API в `state_store.py`;
- команда или callback-экран "Мои запросы";
- создание request при обычном скачивании из поиска;
- обновление request при финальном уведомлении Download Station;
- первичная связь с Plex polling.

Что пока не тащить:

- миграцию всех старых задач и подписок;
- полную замену `/status`;
- сложную историю на месяцы;
- редактирование всех видов подписок на первом шаге.

Критерии готовности первого шага:

- новый request создаётся при скачивании из поиска;
- request показывается в списке;
- есть детальный экран;
- статус меняется после завершения Download Station задачи;
- завершённый request можно скрыть;
- `/status` продолжает работать как раньше.

Риски:

- дублирование состояния с `task_meta` и `notified_tasks`;
- слишком ранняя попытка заменить `/status`;
- разрастание модели request до "идеального workflow engine".

Как снижать риск:

- request в первом релизе должен быть слоем поверх текущих state-файлов;
- статус можно пересчитывать из связанных сущностей;
- старые экраны не ломать, а связывать постепенно.

#### 2. Стартовый экран с действиями

Проблема:

- `/start` хорошо объясняет возможности, но пользователь всё равно должен
  помнить команды или писать текст вручную;
- для нового пользователя нет явного маршрута "что нажать дальше";
- админ и обычный пользователь видят похожий текст, хотя ожидаемые действия
  разные.

UX-цель:

- превратить `/start` в стартовую панель;
- оставить короткое объяснение, но добавить действия;
- показывать только доступные кнопки по текущим feature flags.

Предлагаемый экран:

```text
👋 Готов к работе!

Что хотите сделать?

```

Кнопки для обычного пользователя:

- `🔍 Искать`;
- `🎬 Новинки` - только если доступен `/new`;
- `📋 Мои запросы` - после появления Центра запросов;
- `📋 Загрузки` - fallback до Центра запросов или рядом с ним;
- `🔔 Подписки` - если доступен поиск/подписки;
- `❓ Помощь`.

Дополнительная кнопка для администратора:

- `🛠 Админка`.

Поведение кнопки `🔍 Искать`:

- лучший UX - открыть короткий экран "Напишите название фильма или сериала";
- на первом этапе допустимо отправить подсказку и оставить пользователя в
  обычном текстовом сценарии поиска.

Критерии готовности:

- `/start` имеет inline-кнопки;
- недоступные функции не показываются;
- нажатие каждой кнопки ведёт в рабочий экран или понятную подсказку;
- есть тесты на состав кнопок для обычного пользователя, админа и отключённых
  интеграций.

Риски:

- перегрузить стартовый экран;
- показать кнопку функции, которая не настроена;
- смешать пользовательские и админские действия.

Как снижать риск:

- максимум 4-5 основных кнопок;
- кнопки строить из тех же feature flags, что и `/help`;
- админские действия отделить последней строкой.

#### 3. Единый шаблон ошибок и восстановления

Проблема:

- ошибки сейчас возникают в разных местах: поиск, скачивание, Jackett,
  Rutracker, pending downloads, Plex, подписки;
- часть экранов говорит "не удалось", но не объясняет, что бот уже попробовал и
  что можно сделать дальше;
- recovery-кнопки отличаются от места к месту.

UX-цель:

- любая ошибка должна быть развилкой, а не тупиком;
- пользователь должен видеть человеческое объяснение, текущий план бота и
  безопасные действия.

Базовый шаблон:

```text
Не получилось скачать раздачу

Что произошло:
Jackett не отдал torrent-файл. Такое бывает, если ссылка устарела или трекер
временно недоступен.

Что я уже попробовал:
Повторно запросил раздачу через Jackett.

Что можно сделать:
Попробовать ещё раз, поставить в очередь или поискать через другой трекер.
```

Типовые кнопки:

- `🔄 Повторить`;
- `🕓 В очередь`;
- `🌐 Другой трекер`;
- `🔗 Rutracker напрямую`;
- `🔎 Новый поиск`;
- `📋 К запросу` - после появления Центра запросов;
- `✖️ Закрыть`.

Первый scope:

- описать helper для сборки текстов ошибок по одному формату;
- начать с download/search ошибок, потому что они самые заметные;
- привести `_search_error_keyboard`, `_download_error_keyboard` и места с
  Jackett/Rutracker fallback к одной логике;
- после Центра запросов добавлять кнопку `📋 К запросу`.

Критерии готовности:

- ключевые ошибки поиска и скачивания имеют одинаковую структуру;
- у каждой ошибки есть retry или альтернативный путь;
- экран ошибки всегда имеет `✖️ Закрыть`;
- тексты не показывают пользователю сырой traceback/API-мусор;
- тесты проверяют кнопки и один-два типовых текста.

Риски:

- попытаться сразу унифицировать все ошибки во всём боте;
- потерять полезные технические детали для админа;
- сделать тексты слишком длинными.

Как снижать риск:

- сначала покрыть только search/download;
- подробности оставлять в логах и `/admin` diagnostics;
- пользовательский текст держать коротким: причина, попытка, действие.

#### 4. Экран конкретного результата поиска

Проблема:

- текущая схема `⬇️ N` и `🔔 N` компактна, но пользователь должен сопоставить
  номер кнопки с номером результата в тексте;
- для сериала особенно сложно понять, что именно произойдёт: скачать сейчас,
  скачать и следить, ждать финал, только уведомлять;
- решение принимается в общем списке, где мало контекста.

UX-цель:

- пользователь сначала открывает конкретную раздачу, потом выбирает действие;
- карточка результата должна быть местом принятия решения.

Предлагаемый flow:

1. Пользователь вводит запрос.
2. Бот показывает список результатов.
3. Кнопки под списком открывают карточку результата: `1`, `2`, `3`, `4`, `5`.
4. В карточке показываются детали раздачи.
5. На карточке доступны действия.

Карточка результата должна показывать:

- полное название;
- трекер;
- размер;
- сиды;
- качество/source, если распознано;
- сезон и прогресс эпизодов, если это сериал;
- ссылку на тему, если есть;
- предупреждение Plex, если контент уже есть;
- короткий вывод: "похоже на полный сезон", "неполный сезон", "качество ниже
  текущего Plex" и т.п.

Действия для обычного фильма:

- `⬇️ Скачать`;
- `▶️ Открыть на трекере`;
- `⬅️ К результатам`;
- `✖️ Закрыть`.

Действия для неполного сезона:

- `⬇️ Скачать сейчас`;
- `⬇️ Скачать сейчас + обновлять`;
- `📦 Скачать полный сезон, когда выйдет`;
- `🔔 Только уведомлять`;
- `🎯 Сообщить о финале`;
- `⬅️ К результатам`;
- `✖️ Закрыть`.

Первый scope:

- не удалять сразу старые `⬇️ N` / `🔔 N`;
- добавить карточку результата как промежуточный экран;
- для первого шага можно оставить быстрые кнопки скачивания в списке, если это
  ускоряет привычный сценарий;
- после проверки UX решить, заменять ли список полностью.

Критерии готовности:

- результат можно открыть отдельной карточкой;
- с карточки можно скачать обычный фильм;
- с карточки можно выбрать все варианты для неполного сезона;
- есть возврат к результатам;
- callback-data покрыты тестами;
- старый быстрый сценарий не сломан.

Риски:

- добавить лишний тап для опытного пользователя;
- разрастить карточку до слишком длинного текста;
- усложнить состояние `ConversationHandler`.

Как снижать риск:

- быстрые кнопки можно оставить на переходный период;
- карточку делать компактной и показывать только важное;
- callback namespace держать в `srch:*`, как сейчас.

### P0. Центр запросов

Документ направления: `docs/request-center-v1.5.md`.

Смысл: заменить разрозненную модель "загрузки / подписки / pending / Plex
polling" на пользовательскую модель "мои запросы".

Что должно появиться:

- команда или экран `Мои запросы`;
- список активных запросов пользователя;
- детальный экран запроса;
- понятные статусы: скачивается, ждём новые серии, ждём финал сезона, ждём
  повторной попытки, проверяем Plex, добавлено в Plex, нужно действие, ошибка;
- связь с текущими `task_id`, `pending_downloads`, `topic_subscriptions` и Plex
  polling;
- возможность убрать завершённый запрос из активного списка.

Критерий готовности:

- пользователь может открыть один экран и понять состояние всех своих просьб к
  PlexLoader без знания внутренних механизмов.

### P1. Стартовый экран с действиями

Сейчас `/start` и `/help` хорошо объясняют возможности, но не ведут в следующий
шаг.

Что допилить:

- добавить inline-кнопки в `/start`;
- основные кнопки:
  - `🔍 Искать`;
  - `🎬 Новинки`;
  - `📋 Мои запросы` или `📋 Загрузки`;
  - `🔔 Подписки`;
  - для администратора: `🛠 Админка`.
- показывать только те кнопки, которые реально доступны по включённым
  интеграциям.

Критерий готовности:

- новый пользователь после `/start` может начать основной сценарий без чтения
  справки и ручного ввода команды.

### P1. Экран конкретного результата поиска

Сейчас результаты поиска работают через номера: `⬇️ N` и `🔔 N`. Это компактно,
но требует сопоставлять номер в тексте с кнопкой.

Великолепный UX лучше строить так:

1. Пользователь открывает конкретный результат.
2. Видит карточку раздачи.
3. Выбирает действие на этой карточке.

Что допилить:

- добавить кнопку открытия результата, например `1`, `2`, `3` или карточку
  результата;
- на экране результата показать:
  - название;
  - трекер;
  - размер;
  - сиды;
  - качество;
  - найденный сезон/эпизоды, если это сериал;
  - Plex-проверку, если доступна;
  - понятные действия.
- действия:
  - скачать сейчас;
  - скачать сейчас и обновлять;
  - ждать полный сезон;
  - только уведомлять;
  - сообщить о финале;
  - назад к результатам.

Критерий готовности:

- пользователь принимает решение на экране конкретной раздачи, а не по номеру
  из общего списка.

### P1. Единый шаблон ошибок и восстановления

Ошибки должны быть не техническим тупиком, а развилкой.

Шаблон текста:

```text
Что произошло:
...

Что я уже попробовал:
...

Что можно сделать:
...
```

Типовые recovery-кнопки:

- `🔄 Повторить`;
- `🕓 В очередь`;
- `🌐 Другой трекер`;
- `🔎 Новый поиск`;
- `📋 К запросу`;
- `✖️ Закрыть`.

Критерий готовности:

- после ошибки пользователь понимает причину человеческим языком и видит
  безопасные следующие действия.

### P2. Подписки как понятные правила

`/subs` должен быть экраном "что я поручил боту отслеживать".

Текущий статус:

- список подписок уже показывает источник, прогресс, уведомления, скачивание,
  статус и общую следующую проверку;
- `/new` показывается отдельно как подписка на уведомления, без выдуманного
  режима скачивания;
- на первом экране оставлены точечная отписка и настройка конкретной
  сериальной подписки;
- экран настройки позволяет менять уведомления и скачивание, но запрещает
  комбинацию "не уведомлять + не скачивать".

Что допилить:

- добавить детальный экран подписки;
- в будущем можно расширить настройки, если появятся новые реальные режимы.

Критерий готовности:

- пользователь может открыть `/subs` и понять не только наличие подписки, но и
  её поведение.

### P2. `/new` как персональная лента

`/new` уже полезен, но следующий UX-уровень - дать пользователю управлять своей
лентой.

Возможные действия:

- `🙈 Не интересно`;
- `✅ Уже видел`;
- `🔕 Не присылать похожее`;
- жанровые предпочтения;
- порог рейтинга для пользователя.

Критерий готовности:

- `/new` становится не просто топом, а лентой, которая постепенно лучше
  подстраивается под пользователя.

### P2. Plex duplicate flow

Перед скачиванием Plex-проверка должна объяснять последствия простым языком.

Что допилить:

- явно различать:
  - скачать дубликатом;
  - заменить/улучшить качество;
  - отменить;
  - открыть существующий файл в Plex.
- показывать текущее качество в Plex и качество выбранной раздачи.

Критерий готовности:

- пользователь понимает, создаст ли действие дубль или улучшит существующий
  контент.

### P3. Админ-панель по задачам, а не по сервисам

Сейчас `/admin` полезен технически. Для великолепного опыта его можно
перегруппировать.

Возможные разделы:

- здоровье системы;
- контент и загрузки;
- пользователи;
- интеграции;
- диагностика;
- обслуживание.

Критерий готовности:

- админ открывает панель и видит не список внутренних сервисов, а ответы на
  рабочие вопросы: всё ли здорово, что требует внимания, где настроить.

## Рекомендуемый порядок

1. Центр запросов: модель состояния и `/requests`.
2. Стартовый экран с действиями.
3. Единый шаблон ошибок.
4. Детальный экран результата поиска.
5. Улучшенный `/subs`.
6. Персонализация `/new`.
7. Plex duplicate flow.
8. Перегруппировка `/admin`.

## Что не делать сейчас

- Не начинать с визуальной косметики кнопок без изменения модели состояния.
- Не переписывать весь поиск ради UX.
- Не объединять `/status`, `/subs` и `/new` насильно до появления Центра
  запросов.
- Не делать веб-интерфейс как замену Telegram UX.
- Не тащить новую базу данных только ради плана v1.5.

## Definition of done для UX-задачи

Любая UX-правка считается законченной, если:

- пользовательский сценарий описан до правки;
- экран отвечает на вопросы "что происходит" и "что дальше";
- есть кнопка выхода или возврата по правилам `AGENTS.md`;
- ошибки имеют recovery-путь;
- тексты не используют внутренние термины без необходимости;
