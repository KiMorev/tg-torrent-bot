# Единый экран действия после добавления загрузки

Цель документа - зафиксировать, как привести post-download экраны PlexLoader к одному смысловому формату: пользователь должен понимать, что уже произошло, что бот сделает сам и какие действия доступны дальше.

## Граница задачи

Речь про экран сразу после того, как бот отправил задачу в Synology Download Station.

Не смешивать с финальным уведомлением, когда загрузка уже завершилась и/или контент появился в Plex. Финальный Plex-экран можно привести к тем же принципам, но это другой момент жизни задачи.

## Критерий успеха

Задача решена, когда:

- все одиночные добавления задачи показывают одинаковый смысловой экран;
- кнопки не описывают внутреннюю механику Telegram, а говорят, куда попадёт пользователь;
- magnet без `task_id` имеет отдельный понятный сценарий;
- bulk-сценарии не пытаются притворяться одиночной задачей;
- тесты фиксируют состав кнопок для каждого типа экрана.

## Термины

| Термин | Смысл |
|---|---|
| Success screen | Первое сообщение после успешного добавления в Download Station. |
| Task card | Карточка конкретной задачи из `/status`: статус, прогресс, кнопки управления. |
| Auto-refresh | Фоновый цикл, который каждые 30 секунд перерисовывает success screen в task card, если есть `task_id`. |
| Unknown task id | Download Station принял magnet, но ещё не вернул ID задачи. |
| Bulk screen | Итог скачивания нескольких сезонов или пака сезонов. |

## Текущая механика

После успешного добавления одиночной задачи бот показывает success screen и, если есть `task_id`, запускает `_start_task_card_refresh`.

Фоновый цикл:

- ждёт `PROGRESS_UPDATE_INTERVAL_SECONDS = 30`;
- получает список задач Download Station;
- ищет задачу по `task_id`;
- редактирует это же Telegram-сообщение в task card;
- повторяет, пока задача активна;
- останавливается после финального статуса.

Технически пользователь не переходит в новый экран: то же сообщение меняет текст и клавиатуру.

## Текущие входы в post-download экран

| Флоу | Где в коде | Сейчас |
|---|---|---|
| Поиск -> скачать фильм/раздачу | `_download_and_add` | `_task_added_message` + `_task_reply_markup` |
| Сериал -> скачать сейчас | `_download_and_add` | `_task_added_message` + `_search_after_add_keyboard` |
| Неполный сезон -> скачать сейчас | `_download_and_add` | То же, без подписочного suffix |
| Неполный сезон -> скачать сейчас + следить | `_download_and_add(subscribe=True)` | То же + строка про подписку |
| Plex duplicate confirm | `plex_confirm_download`, `plex_upgrade_download` | Возвращается в `_download_and_add` |
| `/new` -> скачать | search-flow | Возвращается в `_download_and_add` |
| Пользователь прислал magnet | `_process_magnet_uri` | `_task_added_message` + `_task_reply_markup` или список загрузок |
| Пользователь прислал `.torrent` | `_do_process_torrent` | `_task_added_message` + `_task_reply_markup` |
| Bulk: скачать уверенные | `search_series_bulk_run` | `_series_bulk_done_text` + `_series_bulk_done_keyboard` |
| Bulk: ручной кандидат | `search_series_bulk_candidate_download` | Возврат к bulk-review с notice |
| Bulk: pack сезонов | `search_series_bulk_pack_run` | Возврат к bulk-плану с notice |
| Фоновая pending-очередь | `_run_pending_downloads_once` | Отдельный push-результат |
| Фоновая подписка | `_check_jackett_subscriptions`, `_check_subscriptions` | Отдельный push-результат |

## Проблема текущего UI

Сейчас кнопка `🔄 Обновить статус` на success screen технически делает полезное действие: она сразу перерисовывает сообщение в task card, не дожидаясь auto-refresh.

Но для пользователя формулировка спорная:

- на success screen это не просто "обновить строку";
- фактически пользователь попадает в карточку задачи;
- внутри самой task card кнопка `🔄 Обновить статус` уже уместна и не должна конкурировать с success screen.

Поэтому success screen лучше формулировать как "показать задачу сейчас", а не как "обновить".

## Решения после обсуждения

### Пользовательский язык

В пользовательских success screens не писать "Download Station" как основную формулировку. Для пользователя это внутреннее место назначения, а не действие.

Базовая формулировка:

```text
✅ Задача добавлена в очередь скачивания
```

Для magnet:

```text
✅ Magnet отправлен в очередь скачивания
```

Где можно оставлять "Download Station":

- README и установка;
- `/admin` и диагностика;
- технические ошибки, где источник проблемы именно Download Station;
- task card, если она уже является техническим экраном статуса.

### Task ID

В коротком success screen ID задачи пользователю не нужен. Он полезен для диагностики, логов, callback-data и карточки задачи, но на первом экране добавления создаёт лишний технический шум.

Правило:

- не показывать `ID: dbid_...` в success screen после добавления;
- продолжать хранить `task_id` внутри callback-data и state;
- показывать ID в task card `/status`, если он уже там есть и помогает управлять задачей;
- для magnet без ID не писать "ID пока не появился", а объяснять человечески: задача может появиться в списке загрузок через несколько секунд.

### Кнопки и автообновление

Принятые названия:

- основная кнопка одиночной задачи: `📋 Показать задачу`;
- кнопка списка: `📚 К списку загрузок`;
- кнопка подписки, если появится: `⚙️ Изменить подписку`.
- кнопка закрытия: `✖️ Закрыть` на всех финальных экранах.

Auto-refresh success screen в task card через 30 секунд оставляем. После автообновления success screen уже становится task card, и там остаётся уместная кнопка `🔄 Обновить статус`.

### Public-трекеры

На обычном пользовательском success screen не показывать строки про успешное добавление public-трекеров. Пользователю это не даёт полезного следующего действия.

Показывать только отклонения, если они важны:

- трекеры не добавлены, потому что torrent приватный;
- ID задачи пока неизвестен, поэтому трекеры не добавлялись;
- добавление трекеров завершилось ошибкой.

## Magnet без task_id

### Что происходит сейчас

Когда Download Station принимает magnet, но не возвращает `task_id`, бот пытается найти задачу сразу:

- перед добавлением запоминает текущие task id;
- вызывает `create_magnet`;
- если ID пустой, запускает `_wait_for_magnet_task_id`;
- по умолчанию ждёт `MAGNET_POLL_ATTEMPTS = 8` попыток с интервалом `MAGNET_POLL_INTERVAL_SECONDS = 1.5`;
- если ID так и не найден, показывает сообщение "ID пока не появился" и кнопку списка загрузок.

Важная тонкость: без `task_id` нельзя запустить обычный auto-refresh task card, потому что `_start_task_card_refresh` требует конкретный ID.

### Как это должно выглядеть в UI

Если `task_id` найден:

```text
✅ Задача добавлена в очередь скачивания

Magnet принят. Статус обновится автоматически.
Можно открыть задачу сейчас.
```

Кнопки:

```text
📋 Показать задачу
📚 К списку загрузок
✖️ Закрыть
```

Если `task_id` не найден:

```text
✅ Magnet отправлен в очередь скачивания

Задача может появиться в списке загрузок через несколько секунд.

Откройте список загрузок, чтобы проверить её вручную.
```

Кнопки:

```text
📚 К списку загрузок
✖️ Закрыть
```

Не показывать `📋 Показать задачу`, пока нет `task_id`.

### Возможная модернизация magnet-flow

MVP: оставить текущий короткий поиск ID и улучшить текст/кнопки.

Долгий background lookup после финального сообщения в MVP не делаем: текущего короткого ожидания хватает. Если когда-нибудь окажется, что magnet без ID часто раздражает, можно вернуться к отдельной доработке:

- сохранить `chat_id`, `message_id`, `magnet_uri`, `known_task_ids`;
- запустить отдельный lookup-loop на 2-3 минуты;
- если ID найден, запомнить owner/meta, перерисовать сообщение в task card и включить обычный auto-refresh;
- если ID не найден, оставить экран со списком загрузок.

Это отдельная доработка, потому что без `task_id` текущий task-card refresh использовать нельзя.

### 6. Финальное уведомление после завершения/Plex

Это отдельный экран, не success screen после добавления.

#### Загрузка завершилась, Plex ещё не подтвердил

```text
✅ Скачивание завершено

Проверяю Plex. Если файл появится в библиотеке, пришлю кнопку просмотра.
```

Кнопки:

```text
📋 Показать задачу
✖️ Закрыть
```

Не показывать generic `▶️ Открыть Plex`, пока Plex не нашёл конкретный фильм/сезон.

#### Plex нашёл фильм/сезон

```text
✅ «Название» появилось в Plex
```

Кнопки:

```text
▶️ Смотреть в Plex
✖️ Закрыть
```

`📋 Показать задачу` здесь не нужен: главный следующий шаг пользователя - смотреть контент.

#### Plex не нашёл за время ожидания

```text
⚠️ Скачивание завершено, но Plex пока не подтвердил появление

Файл может индексироваться дольше обычного. Проверьте задачу или Plex позже.
```

Кнопки:

```text
📋 Показать задачу
✖️ Закрыть
```

#### Plex был недоступен

```text
⚠️ Скачивание завершено, но проверить Plex не удалось

Сервер Plex был недоступен. Файл, возможно, уже в библиотеке.
```

Кнопки:

```text
📋 Показать задачу
✖️ Закрыть
```

Правила:

- `▶️ Смотреть в Plex` показывать только при точном Plex match;
- generic `▶️ Открыть Plex` на `finished/seeding` убрать;
- `📋 Показать задачу` показывать только в проблемных Plex-сценариях;
- `🗑️ Удалить задачу` оставить внутри task card, не на push-уведомлении.

## Общие правила кнопок

| Кнопка | Когда показывать | Не показывать |
|---|---|---|
| `📋 Показать задачу` | Есть один конкретный `task_id`, экран ещё не task card | Если `task_id` неизвестен или экран уже является task card |
| `📚 К списку загрузок` | Есть несколько задач или нет конкретного `task_id`; нужен переход в общий список | Если экран финальный Plex и список не нужен |
| `🔎 Другой сезон` | Только search-flow, сериал, есть `series_base` и `task_id` | Magnet/torrent uploads, bulk, неизвестный `task_id` |
| `⚙️ Разобрать ошибки` | Bulk finished with failures | Одиночные задачи |
| `▶️ Смотреть в Plex` | Plex уже нашёл конкретный фильм/сезон | До точного Plex match |
| `🗑️ Удалить задачу` | Task card | Success screen и push-уведомления |
| `✖️ Закрыть` | Каждый финальный экран | Промежуточные WAITING-экраны с активной отменой |

## Предлагаемая реализация

### Шаг 1. Унифицировать одиночные клавиатуры

Заменить `_new_task_keyboard` и `_search_after_add_keyboard` на общий helper:

```python
def _download_added_keyboard(
    task_id: str | None,
    *,
    can_show_other_season: bool = False,
    task_id_pending: bool = False,
) -> InlineKeyboardMarkup:
    ...
```

Правила helper:

- если `can_show_other_season` и есть `task_id` - добавить `🔎 Другой сезон`;
- если есть `task_id` - добавить `📋 Показать задачу`;
- всегда добавить `📚 К списку загрузок`;
- всегда добавить `✖️ Закрыть`;
- если `task_id` пустой - не добавлять кнопку статуса.

Тесты:

- одиночная задача с `task_id`;
- сериальная задача с `task_id`;
- magnet без `task_id`;
- порядок кнопок.

### Шаг 2. Унифицировать текст одиночного success screen

Сделать helper поверх `_task_added_message` или заменить его:

```python
def _download_added_text(
    *,
    method: str,
    title: str = "",
    task_id: str = "",
    tracker_result: TrackerApplyResult | None = None,
    accepted_without_task_id: bool = False,
    subscription_summary: str = "",
    can_auto_refresh: bool = False,
) -> str:
    ...
```

Текст должен явно говорить:

- что задача добавлена в очередь скачивания или magnet принят;
- будет ли автообновление;
- что делать, если задача ещё не появилась в списке;
- не показывать `task_id` в коротком success screen.

Тесты:

- torrent с ID;
- magnet с ID;
- magnet без ID;
- подписка создана;
- успешные tracker-result строки скрываются на обычном success screen;
- важные tracker-result отклонения остаются видимыми.

### Шаг 3. Подключить helper в одиночных флоу

Заменить локальные сборки текста/клавиатур в:

- `_download_and_add`;
- `_process_magnet_uri`;
- `_do_process_torrent`.

Проверить, что `_start_task_card_refresh` вызывается только при наличии `task_id`.

### Шаг 4. Bulk привести к тем же словам, но не к той же клавиатуре

Bulk не должен использовать одиночный helper, потому что у него несколько задач.

Нужно отдельно уточнить:

- `_series_bulk_done_text`;
- `_series_bulk_done_keyboard`;
- notice после `search_series_bulk_candidate_download`;
- notice после `search_series_bulk_pack_run`.

Тесты:

- success only;
- success + pending;
- failures only;
- success + failures;
- pack downloaded.

### Шаг 5. Magnet без task_id

Долгий фоновый lookup не делаем. Оставляем текущий короткий `_wait_for_magnet_task_id`, а если ID не нашёлся - показываем понятный экран с переходом к списку загрузок.

Тесты:

- короткое ожидание нашло ID - показывается обычный экран с `📋 Показать задачу`;
- короткое ожидание не нашло ID - показывается экран без кнопки задачи;
- в no-id сценарии есть `📚 К списку загрузок`.

### Шаг 6. Финальное Plex-уведомление

После одиночных success screens меняем финальные уведомления:

- `_notification_keyboard` для `finished/seeding` больше не даёт generic Plex-link;
- `_plex_poll_after_finish` при точном match отправляет `▶️ Смотреть в Plex` + `✖️ Закрыть`;
- `_plex_poll_after_finish` при timeout/unreachable отправляет `📋 Показать задачу` + `✖️ Закрыть`;
- `🗑️ Удалить задачу` остаётся только в task card.

Тесты:
