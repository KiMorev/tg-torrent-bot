# План визуального UX PlexLoader

Цель документа - зафиксировать широкий roadmap улучшения Telegram-интерфейса PlexLoader: от быстрых правок текста и кнопок до больших сценариев будущего. Это не план переписывания бота. Сначала улучшаем понятность текущих экранов, потом беремся за более крупные интерфейсные модели.

## Критерий успеха

UX-улучшение считается полезным, если пользователь в каждом основном сценарии понимает:

1. Что он попросил у бота.
2. Что бот сейчас делает.
3. Что произойдет дальше автоматически.
4. Нужно ли действие от пользователя.
5. Какая кнопка ведет к какому следующему экрану.

## Приоритеты

| Приоритет | Смысл | Горизонт |
|---|---|---|
| P0 | Быстрые улучшения, которые заметно снижают путаницу без смены архитектуры | ближайшие задачи |
| P1 | Средние улучшения отдельных экранов и флоу | после P0 |
| P2 | Новые пользовательские панели и более цельные сценарии | отдельные фичи |
| P3 | Крупные интерфейсные направления, которые могут потребовать WebApp или новой модели данных | дальняя перспектива |

## P0. Быстрые улучшения

### P0.1. Единый шаблон пользовательских сообщений

**Проблема**

Сейчас разные экраны могут говорить на разном языке: где-то видны внутренние сущности вроде Download Station, `task_id`, Jackett или Plex polling, а где-то сообщение уже человеческое. Из-за этого пользователь периодически видит не один продукт, а набор технических подсистем.

**Цель**

Привести основные сообщения к одному смысловому каркасу:

```text
Короткий итог

Что происходит:
...

Что дальше:
...
```

Не обязательно использовать все блоки каждый раз. Для короткого success screen достаточно 1-2 строк, для ошибки или ожидания нужен более подробный текст.

**MVP**

Ввести правило для новых и редактируемых экранов:

- первая строка - результат или текущий статус простым языком;
- технические детали показывать только если они помогают принять решение;
- не писать `task_id`, `dbid`, `polling`, `indexer`, `cache`, если пользователь не находится в админке или диагностике;
- вместо "добавлено в Download Station" для обычного пользователя писать "задача добавлена в очередь скачивания";
- вместо "пришлем ссылку" писать "сообщим", потому что Plex может дать не только ссылку, но и сам факт появления.

**Где применять первым делом**

- success screen после добавления загрузки;
- ошибки скачивания;
- `/continue`;
- `/subs`;
- `/status`;
- финальные Plex-уведомления.

**Граница**

Не переписывать все тексты в проекте одним большим коммитом. Идти по экранам: один экран или один флоу за задачу.

**Проверка**

- скрин/текст экрана читается без знания внутренних сервисов;
- в тестах обновлены ожидаемые тексты, если менялся интерфейс;
- README обновляется только если изменился пользовательский сценарий, а не просто формулировка.

### P0.2. Более полезные пустые состояния

**Проблема**

Пустой экран часто выглядит как "ничего нет", хотя для пользователя важнее понять, почему пусто и что можно сделать дальше. Особенно это критично для `/continue`, `/subs`, `/new`, `/status`: пустота может значить "все хорошо", "еще нет данных", "не настроена интеграция" или "бот пока не уверен".

**Цель**

Каждое empty state должно отвечать на три вопроса:

1. Что именно сейчас пусто.
2. Почему это может быть пусто.
3. Какой следующий полезный шаг доступен.

**MVP-шаблон**

```text
Здесь пока пусто

Бот показывает тут только ...
Если подходящие варианты появятся, они будут здесь.

Что можно сделать:
...
```

**Первый список экранов**

- `/continue`: объяснить, что бот сверяет неполные сезоны Plex с актуальными раздачами и показывает только уверенные варианты.
- `/subs`: если подписок нет, объяснить, где они появляются: после выбора "следить", "докачать и следить", уведомлений о новых сериях или `/new`.
- `/status`: если загрузок нет, написать, что активных задач сейчас нет, а не "задач 0".
- `/new`: если кэш пустой, различать "еще идет прогрев" и "источники недоступны".

**Кнопки**

Пустой экран не должен быть тупиком. Минимальный набор:

- `Обновить`, если данные могут появиться после refresh;
- `Закрыть`, если это финальный экран;
- профильная кнопка действия, если она очевидна: например, "Искать" или "К подпискам".

**Граница**

Не добавлять новые фоновые механики ради красивого empty state. Экран должен честно описывать текущие возможности.

**Проверка**

- пользователь не делает ложный вывод "у меня точно нет неполных сезонов", если бот просто не нашел уверенных кандидатов;
- для каждого измененного empty state есть тест на текст или хотя бы на наличие ключевой кнопки, если экран собирается helper'ом.

### P0.3. Единая грамматика кнопок

**Проблема**

Кнопки должны быть предсказуемыми. Если похожие действия называются по-разному, пользователь вынужден каждый раз заново догадываться, что произойдет после нажатия.

**Цель**

Закрепить словарь кнопок и использовать его во всех новых правках.

**Базовый словарь**

| Действие | Предпочтительная кнопка | Когда использовать |
|---|---|---|
| Закрыть финальный экран | `Закрыть` | финальный экран без следующего шага |
| Вернуться на предыдущий экран | `Назад` | внутри одного флоу |
| Повторить сетевое действие | `Попробовать снова` | ошибка поиска, скачивания, запроса к сервису |
| Обновить текущие данные | `Обновить` | список, статус, диагностика |
| Открыть конкретную загрузку | `Показать задачу` | есть один конкретный `task_id` |
| Открыть общий список | `К списку загрузок` | нет одной конкретной задачи или их несколько |
| Скачать немедленно | `Скачать сейчас` | действие начнет загрузку сразу |
| Скачать и продолжить следить | `Скачать и следить` | загрузка плюс подписка на продолжение |
| Только уведомления | `Только уведомлять` | без скачивания |
| Изменить правила подписки | `Изменить подписку` | пользователь уже создал подписку |

**Правила**

- Не показывать кнопку ради симметрии, если у нее нет понятного пользовательского сценария.
- Не использовать внутренние слова вроде "callback", "task id", "polling".
- Если кнопка ведет в список, в названии должен быть список.
- Если кнопка ведет к конкретной сущности, в названии должна быть эта сущность: задача, подписка, раздача, сезон.
- `Закрыть` единообразно использовать на финальных экранах.

**Граница**

Не переименовывать все кнопки разом. При каждой UX-задаче проверять только кнопки затронутого экрана и соседних переходов.

**Проверка**

- тесты keyboard helper'ов обновлены;
- нет экрана с ошибкой без recovery-кнопки;
- нет финального экрана без `Закрыть`, если пользователь остается с inline-клавиатурой.

### P0.4. Понятные статус-маркеры

**Проблема**

Статусы сейчас могут быть слишком техническими или пугающими. Пример: Download Station показывает `error`, но прогресс 100%. Пользователю не обязательно видеть резкое "ошибка", если бот сам понимает, что файл, вероятно, скачан и уже проверяет Plex.

**Цель**

Разделить технический статус и пользовательский статус.

**Предлагаемые пользовательские статусы**

| Ситуация | Пользовательский статус |
|---|---|
| Задача добавлена | `В очереди скачивания` |
| Идет загрузка | `Скачивается` |
| 100%, но DS показывает error | `Скачивание дошло до 100%, проверяем Plex` |
| Загрузка завершена, Plex еще не нашел | `Проверяем Plex` |
| Plex нашел | `Добавлено в Plex` |
| Бот ждет новые серии | `Ждем новые серии` |
| Бот ждет финал сезона | `Ждем завершения сезона` |
| Нужен выбор пользователя | `Нужно решение` |
| Временный сбой | `Временно недоступно` |
| Постоянная ошибка | `Не получилось` |

**MVP**

Сначала применить к пользовательским экранам:

- карточка загрузки;
- финальное уведомление о загрузке;
- `/subs`;
- `/continue`;
- `/admin` оставить техническим, но сделать верхний summary человеческим.

**Тон для спорных статусов**

Для случая `error + 100%`:

```text
Скачивание дошло до 100%. Download Station показывает ошибку, но файл скачан полностью. Проверяем Plex и сообщим, когда он появится в библиотеке.
```

В короткой карточке:

```text
Статус: скачано, проверяем Plex
```

**Граница**

Не скрывать реальную ошибку полностью. Техническую деталь можно оставить в админке, диагностике или подробностях задачи.

**Проверка**

- пользователь не видит категоричное "ошибка" в сценарии, где бот продолжает успешный путь;
- диагностика для администратора все еще показывает исходный статус сервиса;
- тесты покрывают `error + progress 100%` как особый пользовательский статус.

### P0.5. Живые экраны ожидания

**Проблема**

Долгие операции выглядят одинаково: пользователь не всегда понимает, бот завис или работает. Особенно это касается поиска, массового подбора сезонов, прогрева Jackett, Plex-проверки и ожидания появления задачи по magnet.

**Цель**

Сделать ожидание честным и спокойным: показать, что процесс может занять время, какие этапы уже идут, и что пользователю не нужно нажимать кнопку несколько раз.

**MVP-шаблон**

```text
Ищу подходящие раздачи

Это может занять несколько минут.

Сейчас:
1. Проверяю Plex
2. Ищу раздачи
3. Отбираю уверенные варианты
```

Для коротких операций список этапов не нужен. Достаточно:

```text
Проверяю статус...
```

**Где применить**

- массовый подбор сезонов;
- `/continue` refresh, если появится долгий внешний запрос;
- поиск через Jackett/Rutracker;
- ожидание magnet `task_id`;
- Plex polling в финальных уведомлениях.

**Анимация**

Если в проекте уже есть gif для поиска, использовать его только в долгих сценариях, где Telegram-сообщение реально может висеть больше нескольких секунд. Не вставлять анимацию в каждый маленький refresh.

**Кнопки**

- Во время активного сетевого действия не оставлять экран без выхода.
- Если отмена безопасна - показывать `Отмена`.
- Если отмена не останавливает фоновую работу, не обещать отмену. Лучше оставить `Закрыть` и текст "процесс продолжится в фоне".

**Граница**

Не строить полноценный progress engine. В P0 достаточно честного текста и аккуратного обновления сообщения на ключевых этапах.

**Проверка**

- пользователь видит предупреждение, что операция может занять несколько минут;
- повторные нажатия не запускают параллельные одинаковые процессы, если для флоу уже есть single-flight;
- тесты покрывают callback ожидания, если меняются клавиатуры.

### P0.6. Единый финальный экран после скачивания

**Статус**

Частично уже проработано отдельно в `docs/download-action-screen.md`. Этот пункт остается в P0 как эталон для остальных экранов: все новые финалы должны быть такими же понятными.

**Проблема**

После действия "скачать" пользователь должен получить не технический отчет, а ответ: задача принята, дальше бот сам следит за загрузкой, потом проверит Plex и сообщит результат.

**Цель**

Финальный экран после добавления загрузки должен отвечать:

1. Задача принята или нет.
2. Что бот будет делать дальше.
3. Куда нажать, если хочется посмотреть состояние.
4. Как закрыть экран.

**MVP-вид**

```text
Задача добавлена в очередь скачивания

Раздача: ...

Что дальше:
статус будет обновляться автоматически. Когда загрузка завершится, бот сообщит об этом, затем проверит Plex и сообщит, когда файл появится в библиотеке.
```

Кнопки:

```text
Показать задачу
К списку загрузок
Закрыть
```

Если `task_id` неизвестен:

```text
Magnet отправлен в очередь скачивания

Ссылка принята, но бот пока не видит созданную задачу. Иногда Download Station показывает такие задачи с задержкой.

Что можно сделать:
через минуту откройте список загрузок и нажмите «Обновить». Если задачи там нет, проверьте Download Station или попробуйте добавить magnet ещё раз.
```

Кнопки:

```text
К списку загрузок
Закрыть
```

**Граница**

- Не показывать кнопку `Показать задачу`, если нет конкретного `task_id`.
- Не показывать Plex-кнопку до точного Plex match.
- Не показывать удаление задачи в push-уведомлениях, оставить его внутри карточки задачи.

**Проверка**

- одиночная загрузка, magnet, `.torrent`, `/new`, сериал и bulk не противоречат друг другу по словам;
- для bulk не использовать одиночную клавиатуру, потому что задач может быть несколько;
- tests покрывают состав кнопок.

## P1. Средние улучшения

### P1.1. Карточка конкретного результата поиска

Пользователь сначала открывает конкретную раздачу, видит ее детали и только потом выбирает действие. Это убирает режим "нажми номер N" и снижает риск ошибочного скачивания.

### P1.2. Разделение главных и вторичных действий

На экранах с несколькими действиями первым должен быть один главный путь. Вторичные действия уходят ниже: "следить", "подписка", "другой сезон", "назад".

### P1.3. `/subs` как панель правил

Подписки должны выглядеть не как список записей, а как список правил: что отслеживаем, что скачиваем, какие уведомления включены, когда была последняя проверка.

### P1.4. `/status` как центр загрузок

Список загрузок должен показывать не только Download Station status, а пользовательскую стадию: скачивается, проверяем Plex, добавлено в Plex, нужна реакция.

### P1.5. Plex lifecycle screen

Отдельный понятный экран жизненного цикла после скачивания: "скачано", "ждем индексацию Plex", "нашли", "не нашли за 10 минут".

### P1.6. Двухуровневая диагностика

В `/admin` сначала короткий summary: что работает, что требует внимания. Детали по сервисам открываются отдельными кнопками.

## P2. Крупные пользовательские панели

### P2.1. `/menu`

Главное меню с основными действиями: поиск, новинки, загрузки, подписки, докачать сезон, помощь, админка для администратора.

### P2.2. Личный домашний экран

Персональная панель: активные загрузки, подписки, новые рекомендации, кандидаты на докачку, последние действия.

### P2.3. История как пользовательский инструмент

Не просто лог загрузок, а полезная история: что скачивал пользователь, какие сезоны можно продолжить, какие раздачи уже были выбраны.

### P2.4. Guided-мастера

Пошаговые сценарии для сложных действий: скачать все сезоны, выбрать перевод, настроить подписку, подключить Plex.

Первый подробный кандидат: guided-мастер
[`📚 Скачать сериал целиком`](series-guided-master.md), который ведёт пользователя
через выбор эталонной раздачи к текущему bulk-плану сезонов.

### P2.5. Метки уверенности

Для автоматических решений показывать понятные уровни: "уверенно", "похоже", "нужно проверить". Особенно полезно для сериалов и альтернативных раздач.

## P3. Дальняя перспектива

### P3.1. Telegram WebApp

Веб-интерфейс внутри Telegram для сложных списков, фильтров, истории и массовых действий. Имеет смысл только когда Telegram-кнопок уже явно не хватает.

### P3.2. Визуальный каталог PlexLoader

Карточки фильмов и сериалов с постерами, статусами, качеством, кнопками скачивания и подписки.

### P3.3. Персональные рекомендации интерфейса

Бот запоминает предпочтения пользователя: качество, озвучки, оригинальная дорожка, субтитры, любимые действия. Использовать осторожно, с прозрачным подтверждением.

## Рекомендуемый порядок работ

1. P0.4 - статус-маркеры: особенно важны для `error + 100%` и Plex-проверки.
2. P0.2 - пустые состояния: дешево и сразу снижает путаницу.
3. P0.3 - словарь кнопок: закрепить, чтобы новые экраны не расходились.
4. P0.1 - шаблон сообщений: применять постепенно к каждому флоу.
5. P0.5 - ожидание: начать с долгих операций.
6. P0.6 - финальные экраны: сверять новые флоу с уже сделанным планом.

## Что не делать сейчас

- Не начинать с Telegram WebApp.
- Не переписывать весь бот ради единого UI-слоя.
- Не создавать новые сущности состояния только ради красивого текста.
- Не менять все кнопки во всех флоу одним коммитом.
- Не скрывать технические ошибки от администратора.

## Definition of done для UX-задачи

- Описан пользовательский сценарий до правки.
- Экран отвечает на вопросы "что происходит" и "что дальше".
- Кнопки соответствуют словарю из этого документа и правилам `AGENTS.md`.
- Ошибка или пустое состояние не является тупиком.
- Тесты обновлены, если менялся текст helper'а, callback-data или клавиатура.
- README обновлен, если изменился пользовательский сценарий.
- `docs/bot-structure.md` обновлен, если менялись команды, callback namespace, state или структура флоу.
