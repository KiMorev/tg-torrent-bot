# План: умный разбор поискового ввода и главный путь для неполных сезонов

Задача: уменьшить ручные действия в существующих флоу поиска, не добавляя новую команду поиска/скачивания и не выполняя скачивание молча. Отдельная команда настроек по умолчанию допустима, потому что она не запускает поиск.

План покрывает четыре изменения:

1. Разбирать намерение пользователя прямо из текста, голосовой расшифровки и запроса из Кинопоиска.
2. Для неполной сериальной раздачи сделать главным действием "скачать доступные серии и следить за новыми".
3. Показывать Plex-дубликаты раньше, прямо в выдаче поиска, чтобы пользователь реже доходил до лишнего confirm-dialog.
4. Дать каждому пользователю свои настройки поиска по умолчанию поверх стартовых глобальных дефолтов: качество, Original, субтитры и предпочитаемые переводы.

## Критерий успеха

Задача решена, когда:

- обычный пользователь может написать естественный запрос вроде `клиника все сезоны 1080 lostfilm`, `дюна 4к в оригинале`, `3 сезон с субтитрами`, `скачай доступные и следи`, и бот корректно заполнит уже существующие `srch_query`, `srch_settings`, `srch_intent` и subscription preset;
- при низкой уверенности бот не угадывает опасные действия, а показывает обычный экран выбора;
- GPT используется только как мягкий помощник для разбора неоднозначного текста, а не как единственный источник истины;
- при ошибке GPT / OpenAI бот деградирует в детерминированный разбор без технического текста для пользователя;
- для partial-сезона первый видимый путь ведёт к скачиванию доступных серий с подпиской на продолжение, но пользователь всё ещё может выбрать "только сейчас" или "ждать полный сезон";
- результаты поиска заранее показывают, что фильм/сезон уже есть в Plex или есть в лучшем качестве, но финальная проверка перед скачиванием остаётся обязательной;
- каждый пользователь может явно сохранить свои дефолты поиска через отдельную команду настроек, и новые текстовые/voice/KP-поиски стартуют с ними, если сам запрос не содержит более конкретных параметров;
- все новые правила покрыты unit-тестами.

## Не делаем

- Не добавляем новую команду для поиска или скачивания; отдельный вход `/settings` для дефолтов нужен только для управления настройками.
- Не скачиваем торрент автоматически сразу после текстового запроса.
- Не меняем правила scoring выдачи, кроме явного применения распознанных search settings.
- Не сохраняем широкий постоянный профиль пользователя в этой задаче: только явно выбранные search defaults и мягкие предпочтения по переводу.
- Не включаем скрытое "обучение" дефолтов по поведению пользователя в первом шаге: пользователь меняет их только в `/settings`.
- Не требуем OpenAI для базового сценария.
- Не удаляем и не заменяем файлы в Plex автоматически.
- Не блокируем скачивание только на основании ранней Plex-подсказки.
- Не меняем installer, `.env`, Docker и зависимости.

## Текущее состояние

Сейчас:

- `text_message_entry` и `voice_message_entry` сбрасывают `srch_intent`.
- `search_got_query` всегда показывает `SEARCH_OPTIONS`.
- `kp_link_entry` тоже стартует в режиме `одна раздача`.
- `_build_search_query` умеет только уже нормализованные настройки: quality, Original, Sub.
- `_SRCH_DEFAULT_SETTINGS` сейчас один на всех: `1080p`, без Original, без Sub.
- `srch_settings` живёт в `context.user_data` и очищается при cancel/timeout, то есть не является пользовательским дефолтом.
- `_split_query_settings` умеет снять технические токены из search query.
- partial-сезон после `⬇️ N` открывает `_download_picker_keyboard`, где первым стоит `⬇️ Скачать сейчас`.
- Plex pre-check сейчас происходит только после выбора конкретной раздачи в `_download_and_add`.

Точка улучшения: до показа `SEARCH_OPTIONS` / запуска поиска добавить слой нормализации пользовательского намерения.

## UX-принципы

1. Детерминированный разбор важнее GPT.
   Если запрос понятен регулярками и словарями, GPT не нужен.

2. GPT не принимает irreversible decisions.
   GPT может предложить intent/settings, но код применяет их только если они проходят whitelist.

3. Чем опаснее действие, тем выше порог уверенности.
   Сменить `quality` безопасно. Включить `series_master` безопасно, но меняет флоу. Скачать или подписаться без клика нельзя.

4. Пользователь должен видеть, что бот понял.
   На options/results экране показываем коротко: `Понял: сериал целиком, 1080p, LostFilm, субтитры`.

5. Ошибка разбора не должна быть ошибкой пользователя.
   Если не уверены, показываем обычные кнопки, как сейчас.

## Модель результата разбора

Нужен чистый объект, например `SearchIntentDraft`:

```text
raw_text: исходный текст
base_query: поисковая часть без командных слов
intent: one_release | series_master | unknown
quality: 4K | 1080p | 720p | any | None
audio_original: true | false | None
subs: true | false | None
season: int | None
episode: int | None
whole_series: true | false | None
partial_policy_hint: now | now_and_follow | wait_full | notify_only | None
tracker_hint: rutracker | jackett | all | indexer-name | None
voice_hints: list[str]
voice_required: true | false
confidence: high | medium | low
source: deterministic | gpt | mixed
warnings: list[str]
```

В коде это может быть dataclass в `bot.py` или отдельный чистый модуль, если тесты станут удобнее. Для первой реализации лучше отдельный модуль `search_intent.py`, потому что логика большая и хорошо тестируется без Telegram.

## Детерминированный разбор

### Нормализация

- lower/casefold;
- `ё -> е`;
- убрать лишнюю пунктуацию, но не ломать `S02E05`, `2160p`, `4K`, `WEB-DL`;
- нормализовать `сезон 3`, `3 сезон`, `s03`, `season 3` в `season=3`;
- оставить оригинальный `base_query` в человеческом виде.

### Quality

Распознавать:

- `4к`, `4k`, `uhd`, `2160`, `2160p`;
- `1080`, `1080p`, `1080р`, `1080 Р`, `fullhd`, `full hd`, `fhd`;
- `720`, `720p`, `720р`, `720 Р`, `hd`;
- `2160p` / `2160р` и другие варианты, где последняя буква `p` может быть латинской `p` или русской `р`;
- `любой`, `любое качество`, `без качества`, `неважно качество`, `что найдешь`.

Правило:

- если quality явно указан, ставим его в `srch_settings`;
- если пользователь сказал `любое качество`, ставим `any`;
- если quality не указан, используем текущий default, как сейчас.
- нормализация качества должна воспринимать латинскую `p` и кириллическую `р` одинаково, потому что пользователь часто пишет `1080р` русской раскладкой.

### Original audio

Положительные маркеры:

- `в оригинале`, `оригинальная дорожка`, `original`, `orig`, `без дубляжа`, `английская дорожка`, `eng audio`;
- `с русскими субтитрами в оригинале` = `audio_original=true`, `subs=true`.

Отрицательные маркеры:

- `без оригинала`, `не оригинал`, `дубляж`, `русская озвучка`, `на русском`.

Правило:

- отрицательный маркер не должен включать `audio_original`;
- `дубляж` сам по себе пока не превращаем в voice requirement, если нет точной студии.

### Subtitles

Положительные маркеры:

- `с субтитрами`, `сабы`, `subs`, `sub`, `forced subs`, `русские субтитры`;

Отрицательные:

- `без субтитров`, `сабы не нужны`.

Правило:

- положительный маркер включает `subs`;
- отрицательный явно выключает `subs`, даже если default когда-нибудь станет true.

### Whole-series intent

Положительные маркеры:

- `все сезоны`, `весь сериал`, `сериал целиком`, `полностью`, `all seasons`, `complete series`;
- `скачать недостающие сезоны`, `добрать сезоны`;
- `с 1 сезона по 5`, `1-5 сезоны` - для этой задачи только `series_master`, диапазон сезонов не реализуем.

Слабые маркеры:

- `сериал`, `season`, `сезон`;
- они не включают `series_master` сами по себе, потому что пользователь может искать одну раздачу сезона.

Правило:

- `все сезоны` / `целиком` -> `series_master` high confidence;
- `3 сезон` -> обычный one-release search с season filter;
- KP-ссылка на сериал + `все сезоны` рядом -> `series_master`;
- KP-ссылка на сериал без доп. слов -> options screen, не угадываем.

### Partial-season policy hints

Это не запускает скачивание из обычного поиска. Используется только когда пользователь уже открыл partial picker или нажал действие в partial context.

Распознавать:

- `скачай доступные и следи`, `качай и следи`, `новые серии по мере выхода` -> `now_and_follow`;
- `только сейчас`, `без подписки`, `не следи` -> `now`;
- `когда выйдет полностью`, `ждать полный сезон`, `после финала` -> `wait_full`;
- `только уведомлять`, `сообщи когда выйдет` -> `notify_only`.

### Tracker/indexer hints

Можно распознавать, но применять осторожно:

- `на rutracker`, `рутрекер` -> selected indexers Rutracker-only;
- `везде`, `все трекеры`, `all trackers` -> all Jackett indexers;
- конкретные indexer names -> применить только если точно есть такой indexer id/name.

Если indexer не найден, не показывать ошибку. Просто оставить дефолт и, возможно, добавить `warnings=["tracker_hint_not_found"]` для логов.

### Voice/studio hints

Если пользователь явно указал озвучку, бот должен учитывать это как реальное пожелание, а не молча игнорировать.

Распознавать:

- `lostfilm`, `лостфильм`;
- `newstudio`, `ньюстудио`;
- `кубик`, `кубик в кубе`;
- `кравец`;
- `hdrezka`, `резка`.

Явные маркеры требования:

- `в озвучке LostFilm`, `с озвучкой LostFilm`, `перевод LostFilm`;
- `только LostFilm`, `обязательно LostFilm`;
- `без LostFilm` - отрицательный маркер, не включать эту студию в wanted voices.

Применение:

- в обычном поиске не добавлять voice в tracker query как единственный путь: трекеры часто пишут студии нестабильно;
- после получения результатов разобрать title через локальный словарь voices;
- если есть результаты с нужной озвучкой, поднять/оставить их и показать строку `Показаны варианты с LostFilm`;
- если результатов с нужной озвучкой нет, не показывать пустой dead-end: показать лучшие альтернативы с баннером `LostFilm не нашёл, показываю другие озвучки`;
- если пользователь написал `только LostFilm`, всё равно можно показать альтернативы, но с более явным текстом `С LostFilm не нашёл. Ниже варианты без этого требования`;
- в bulk profile использовать wanted voices как мягкий `preferred_voices` бонус, если voice реально найдена в эталонной раздаче; не превращать это в hard-фильтр без ручного выбора в профиле.

## GPT-разбор

GPT нужен для запросов, где deterministic слой видит намерение, но не уверен в границах названия:

- `найди мне третий сезон клиники желательно лостфильм и сабы`;
- `хочу весь офис в 1080 но без оригинала`;
- `дюна новая в 4к оригинал`;
- голосовая расшифровка с мусором: `скачай клинику сезон наверное третий весь ну в хорошем качестве`.

### Когда GPT вызывается

Вызывать GPT только если:

- `GPT_ENABLED`;
- deterministic confidence не `high`;
- текст длиннее короткого title-запроса;
- текст содержит глаголы/служебные слова: `найди`, `скачай`, `хочу`, `нужно`, `покажи`, `подбери`, `сделай`;
- или voice transcription помечена как потенциально разговорная.

Не вызывать GPT:

- для magnet;
- для `.torrent`;
- для KP URL без дополнительного текста;
- для простого `Дюна`;
- для retry.

### GPT prompt

Нужен строгий JSON-only ответ:

```json
{
  "base_query": "Клиника",
  "intent": "one_release",
  "quality": "1080p",
  "audio_original": false,
  "subs": true,
  "season": 3,
  "whole_series": false,
  "partial_policy_hint": null,
  "tracker_hint": null,
  "voice_hints": ["LostFilm"],
  "voice_required": true,
  "confidence": "medium"
}
```

Whitelist после GPT:

- `intent` только `one_release`, `series_master`, `unknown`;
- `quality` только `4K`, `1080p`, `720p`, `any`, `null`;
- bool-поля только bool/null;
- season 1..99;
- `partial_policy_hint` только разрешённые значения;
- `voice_hints` только из локального словаря известных voice labels.
- `voice_required` только bool; если GPT поставил true без known `voice_hints`, сбрасывать в false.

Если JSON невалидный, поля неизвестны или ответ слишком большой - игнорировать GPT и логировать sanitized label.

### GPT ошибки

Пользовательский текст:

- не показывать `OpenAI`, `quota`, `HTTP`, stack trace для обычного поиска;
- если GPT не ответил, просто использовать обычный deterministic flow;
- отдельное предупреждение пользователю не нужно, потому что поиск может работать без GPT.

Админские сигналы:

- сохранить текущую модель диагностики `gpt_usage`;
- добавить label `intent_parse_timeout`, `intent_parse_bad_json`, `intent_parse_unavailable`, если это не раздует диагностику.

## Персональные предпочтения поиска

Ответ на вопрос "возможно / реально / нужно":

- возможно: да, у каждого Telegram-пользователя есть стабильный `chat_id`, а проект уже хранит per-user/per-chat state в JSON;
- реально: да, это небольшая инфраструктурная доработка вокруг `state_store`, без новых зависимостей и без внешних сервисов;
- нужно: да, если мы делаем умный разбор ввода. Иначе один пользователь каждый раз будет возвращать `4K`, другой `1080p`, третий `Original + Sub`, четвёртый `LostFilm`, и "умный" поиск всё равно будет начинаться с чужого дефолта.

### MVP

Хранить только явные search defaults:

```text
chat_id: int
quality: 4K | 1080p | 720p | any
audio: bool
subs: bool
preferred_voices: list[str]
updated_at: iso datetime
```

`preferred_voices` - это мягкое предпочтение, а не жёсткое требование. Если пользователь в конкретном запросе написал `в озвучке LostFilm` или `только LostFilm`, это явное указание сильнее дефолта, но оно всё равно не должно превращать выдачу в пустой экран: если нужного перевода нет, показываем альтернативы с пояснением. Если голосовой/текстовый запрос ничего не сказал про озвучку, дефолтные переводы помогают сортировать и дают мягкий бонус в bulk-profile.

Не хранить в MVP:

- `series_master` как дефолт: слишком легко случайно отправить пользователя в сложный bulk-flow;
- строгий режим "искать только эти переводы" как дефолт: он слишком легко превращает обычный поиск в пустую выдачу;
- partial policy: глобальный главный путь `скачать + следить` уже закрывает основную боль;
- tracker/indexer defaults: можно добавить позже, но это более хрупко из-за доступности Jackett indexers.

### Где хранить

Вариант MVP: новый state file `user_search_defaults.json` в `STATE_DIR`.

Формат:

```json
{
  "123456": {
    "quality": "1080p",
    "audio": false,
    "subs": true,
    "preferred_voices": ["LostFilm", "NewStudio"],
    "updated_at": "2026-05-30T18:00:00+03:00"
  }
}
```

Почему не `context.user_data`:

- он не является продуктовым persistent storage;
- cancel/timeout сейчас очищают `srch_settings`;
- после рестарта бота настройки должны сохраниться.

Почему не `.env`:

- `.env` общий для всех;
- пользовательские настройки не должны требовать админа и перезапуска контейнера.

### Как пользователь меняет дефолты

Дефолты меняются только через отдельный вход настроек. В результатах поиска, на экране конкретных раздач и в `⚙️ Доп. параметры` не показывать кнопки `сохранить по умолчанию` / `сбросить дефолты`: это смешивает настройку одного поиска и постоянные настройки пользователя.

Основной вход: команда `/settings` в Telegram menu с описанием `Предпочтения поиска`. Дополнительно можно принять текст `предпочтения поиска` и старый текст `настройки по умолчанию` как алиасы, но это не должно быть кнопкой внутри результатов поиска.

Вариант кнопок:

```text
🎞 Качество: предпочитаю 1080p
🎧 Original: не важно
💬 Субтитры: не важно
🎙 Переводы: предпочитаю LostFilm / NewStudio
↩️ Вернуть стартовые предпочтения
✖️ Закрыть
```

Правила:

- `/settings` показывает `quality/audio/subs/preferred_voices`, но без текущего поискового запроса;
- каждый toggle сразу сохраняет новое значение для текущего `chat_id` и обновляет тот же экран;
- после изменения бот коротко отвечает callback-уведомлением `Сохранено`;
- `↩️` удаляет запись пользователя и возвращает глобальный `_SRCH_DEFAULT_SETTINGS` плюс пустой список preferred voices;
- если пользователь меняет `⚙️ Доп. параметры` внутри поиска и нажимает `Искать`, это не меняет дефолты молча.

### Предпочитаемые переводы

Рекомендация: добавить в defaults уже сейчас, но только как "предпочитаю", не как "требую".

Почему это полезно:

- обычный пользователь часто всегда хочет один и тот же перевод;
- для сериалов это снижает ручной выбор в bulk-profile;
- explicit-запросы вроде `в озвучке Кубик` всё равно остаются сильнее дефолта.

Как выбирать:

- в `/settings` показывать известные студии из локального словаря распознавания озвучек;
- разрешить выбрать до двух preferred voices, чтобы не делать профиль слишком широким;
- добавить вариант `без предпочтений`;
- неизвестный ввод не сохранять как сырой текст, пока нет нормализации через словарь.

Как применять:

- в обычной выдаче поднять релизы с preferred voices выше, если такие релизы есть;
- если preferred voice не найден, показать обычную выдачу, а не пустой результат;
- в bulk-flow, если preferred voice есть в эталонной раздаче, сохранить её как `preferred_voices` для мягкого scoring-бонуса;
- если preferred voice в эталоне не найдена, оставить обычный выбор озвучки и не блокировать подбор;
- если пользователь явно написал озвучку в запросе, использовать её вместо preferred voices.

### Как дефолты применяются

При старте нового поиска:

1. взять глобальный `_SRCH_DEFAULT_SETTINGS`;
2. поверх применить personal defaults текущего `chat_id`, если есть;
3. поверх применить deterministic/GPT intent из конкретного запроса;
4. если запрос конфликтный, показать options screen с уже заполненными настройками;
5. хранить источник параметров (`default`/`explicit`): предпочтения из `/settings` дают ranking/profile hint, а явный ввод пользователя сильнее;
6. preferred voices, Original и субтитры из `/settings` не должны обнулять выдачу: если точного варианта нет, показываем альтернативы с пояснением.

Примеры:

- пользовательский default `4K`, запрос `Дюна` -> ищем `Дюна 4K`;
- default `4K`, запрос `Дюна 1080` -> ищем `Дюна 1080p`;
- default `Original + Sub`, запрос `Дюна без субтитров` -> `Original=true`, `subs=false`;
- default `1080p`, запрос `любое качество` -> `quality=any`.
- default `Original + Sub`, запрос `Дюна` -> раздачи с Original/Sub выше, остальные остаются в выдаче;
- запрос `Дюна в оригинале`, но Original не найден -> показываем другие варианты с пояснением;
- preferred voice `LostFilm`, запрос `Клиника` -> релизы LostFilm выше, но альтернативы остаются;
- preferred voice `LostFilm`, запрос `Клиника в озвучке Кубик` -> ищем/приоритизируем `Кубик`, а не LostFilm.

### UI-текст

Не выводить управление дефолтами в результатах поиска. Не добавлять туда `💾 Сделать моими предпочтениями`, `↩️ Вернуть стартовые предпочтения` или служебный блок `Ваши предпочтения`.

Текущие предпочтения показывать только в `/settings`:

```text
Ваши предпочтения: 4K, Original, Sub, LostFilm
```

Если применён конкретный запрос, на экране подтверждения можно показать только то, что бот понял из самого запроса:

```text
Понял из запроса: 1080p, без субтитров
```

В результатах поиска допустим только смысловой search-banner, если он объясняет выдачу, например: `LostFilm не нашёл, показываю другие озвучки`. Это не управление настройками и не должно вести к сохранению defaults.

### Ошибки

Если state file не читается или не пишется:

- поиск продолжает работать с глобальными дефолтами;
- пользователь видит мягкий текст только при попытке сохранить: `Не удалось сохранить настройки. Поиск можно продолжить.`;
- лог содержит подробности для админа.

### Приватность и доступ

- ключ только `chat_id`;
- не хранить поисковые запросы в defaults;
- не хранить voice transcript;
- обычный пользователь видит и меняет только свои defaults;
- админский UI для просмотра defaults в этой задаче не нужен.

## Поведение входов

### Text

`text_message_entry`:

1. magnet/KP URL existing routing remains first.
2. Для обычного текста вызвать intent parser.
3. Собрать settings: global defaults -> personal defaults -> intent draft.
4. Если parser high confidence и не нужен options screen, сразу вызвать `_run_search`.
5. Если medium/low или нужен human confirmation, показать `SEARCH_OPTIONS` с уже заполненными настройками.

Когда показывать options:

- `series_master` распознан, потому что пользователь должен понимать, что сначала выбирает эталон;
- запрос содержит конфликт: `все сезоны 3 сезон`;
- GPT confidence low;
- распознан неизвестный tracker/voice hint;
- запрос выглядит как команда с несколькими действиями.

### Voice

После Whisper:

- применить тот же parser к transcription;
- на экране показать `Услышал: ...` и `Понял: ...`;
- если parser high confidence для one-release, можно сразу искать;
- если transcription короткая и чистая, поведение как text;
- если transcription содержит много лишних слов, чаще показывать options.

### KP link

KP URL остаётся special case:

- чистая KP-ссылка -> текущий screen options;
- KP-ссылка + `4к`, `оригинал`, `субтитры` -> заполнить настройки;
- KP-ссылка + `все сезоны` для series -> `series_master`;
- KP-ссылка на movie + `все сезоны` -> игнорировать series intent и показать warning в тексте: `Кинопоиск определил фильм, поэтому ищу одну раздачу`.

## Ошибки и конфликты

| Ситуация | Поведение |
|---|---|
| `все сезоны 3 сезон` | options screen, текст: `Похоже, есть два намерения: сезон 3 или сериал целиком.` |
| `4к 720` | взять последнее явно указанное качество или показать options; безопаснее options |
| `без субтитров с субтитрами` | options |
| `lostfilm`, но в обычной выдаче нет LostFilm | показать альтернативы с баннером `LostFilm не нашёл, показываю другие озвучки` |
| `lostfilm`, но в bulk-эталоне нет LostFilm | не применять hard-фильтр; оставить обычный выбор озвучки и собрать план по другим найденным вариантам |
| GPT не ответил | deterministic fallback без пользовательской ошибки |
| GPT вернул невалидный JSON | deterministic fallback + debug/admin label |
| Запрос стал пустым после удаления служебных слов | показать обычное `Введите название` |
| Распознан tracker, но Jackett не настроен | игнорировать tracker hint |
| Распознан Rutracker-only, но Rutracker не настроен | не применять, оставить Jackett |

## Partial-season UX

Текущий экран `search_download_pick` остаётся, но порядок и labels меняются.

### Для partial result

Новый порядок:

1. `⬇️ Скачать доступные + следить`
2. `⬇️ Скачать только доступные`
3. `📦 Скачать, когда сезон завершится`
4. `📚 Скачать недостающие сезоны` если это series result
5. `⬅️ К результатам`
6. `❌ Отмена`

Callback можно оставить существующий:

- `now_and_follow` = текущий `srch:sub_preset:N:each`;
- `now` = текущий `srch:dl:N`;
- `wait_full` = текущий `srch:sub_preset:N:after`.

То есть меняется только порядок/текст кнопок и тесты.

### Для full series result

Оставить как сейчас:

1. `⬇️ Скачать сейчас`
2. `📚 Скачать недостающие сезоны`

Не добавлять подписочные partial-действия, если `partial=False`.

### Для notification-only row

`🔔 N` остаётся вторичным входом. Его можно не менять в этой задаче.

## Ранние Plex-подсказки в выдаче

Цель: пользователь должен увидеть возможный дубль до клика `⬇️ N`, а не только после выбора раздачи.

Важно: это не замена текущему Plex pre-check в `_download_and_add`. Ранняя подсказка может быть неточной из-за cache stale, названий релизов, сезонов, качества и неполной metadata. Финальная проверка перед созданием задачи остаётся источником истины.

### Где показывать

В `_build_results_text` можно добавить короткую строку под конкретным результатом:

```text
   ✅ Уже есть в Plex: 1080p
   🔼 В Plex есть хуже: 720p, можно заменить лучше
   ⚠️ В Plex есть лучше: 4K
```

Для сериалов:

```text
   ✅ Сезон уже есть в Plex: S3, 1080p
   ⚠️ В Plex есть этот сезон лучше: S3, 4K
   📺 В Plex уже есть другие сезоны: S1, S2, S4
```

Для partial-сезона:

- если этот сезон уже есть в Plex полностью, показывать `✅ Сезон уже есть в Plex`;
- если Plex знает только часть сезона, не говорить `уже есть`, а писать осторожно: `📺 В Plex уже есть часть сезона`;
- если есть другие сезоны, это контекст, а не блокировка.

### Как это видно при сгруппированных кнопках

Сейчас кнопки действий сгруппированы отдельно: верхняя строка `⬇️ 1 2 3 4 5`, а сами результаты перечислены текстом выше. Поэтому Plex-hint должен жить не в кнопке, а в текстовом блоке конкретного результата под тем же номером.

Пример:

```text
⭐ 1. [RT] Дюна: Часть вторая / Dune: Part Two (2024) BDRip 1080p
   📦 12.4 GB | 🌱 140
   ✅ Уже есть в Plex: 1080p

🔎 2. [RT] Дюна: Часть вторая / Dune: Part Two (2024) UHD BDRemux 2160p
   📦 68.0 GB | 🌱 52
   🔼 В Plex есть хуже: 1080p

Кнопки:
⬇️ 1  ⬇️ 2  ⬇️ 3
```

Так пользователь видит предупреждение до нажатия `⬇️ 1`, а клавиатура остаётся компактной и не раздувается отдельными кнопками на каждый Plex-сценарий.

### Как это видно на экране выбора сезона/кластера

Скрин с `Острые козырьки` - это ещё не список конкретных раздач, а промежуточный cluster picker: пользователь выбирает сезон, весь season pack или отдельный фильм с похожим названием. Здесь Plex-проверка должна быть грубее, но раньше:

1. До показа `_cluster_picker_keyboard` обогатить кластеры компактным `plex_hint`.
2. Для сериалов матчить Plex по нормализованному названию шоу, а не по году строки: `2022` у `S6` - это год сезона, он не обязан совпадать с годом старта сериала.
3. Для строк `S1`, `S2`, `S6` проверять наличие именно этого сезона в Plex cache.
4. Для строки фильма `Острые козырьки Бессмертный человек (2026)` делать отдельный movie lookup по названию и году; совпадение с сериалом `Острые козырьки` не должно помечать фильм как уже существующий.
5. Если cache пустой, старый или уверенность низкая, на cluster picker ничего не показывать; финальный pre-check при скачивании всё равно остаётся.

Компактный вариант для такого экрана:

```text
Plex: по «Острые козырьки» уже есть S1-S5; S6 не найден.

📺 S6 · Острые козырьки (2022) · 4 разд.
✅ S5 · Острые козырьки (2019) · 5 разд.
✅ S4 · Острые козырьки (2017) · 1 разд.
✅ S3 · Острые козырьки (2016) · 1 разд.
✅ S2 · Острые козырьки (2014) · 1 разд.
✅ S1 · Острые козырьки (2013) · 1 разд.
🎬 Острые козырьки Бессмертный человек (2026) · 1 разд.
```

Значки:

- `✅` - этот сезон/фильм уже есть в Plex в сопоставимом качестве;
- `🔼` - в Plex есть хуже, текущая выдача может быть апгрейдом;
- `⚠️` - в Plex есть лучше, скачивание этой версии, скорее всего, downgrade;
- без значка или `📺`/`🎬` - совпадения в Plex не найдено или уверенность недостаточная.

Если у сезона в Plex есть 720p, а поиск идёт в 1080p, строка может стать `🔼 S5 · ...`. Если в Plex уже 2160p, а пользователь смотрит 1080p, строка станет `⚠️ S5 · ...`. Подробность по качеству лучше держать в верхнем текстовом блоке, а кнопки оставить короткими, чтобы Telegram не раздувал клавиатуру.

После клика по `S6` пользователь попадает в обычный список конкретных раздач, и там Plex hint повторяется точнее уже под каждой раздачей: качество, partial/full, размер и конкретный релиз.

### Где не показывать

- если Plex отключён;
- если результат не похож на фильм/сезон с достаточной уверенностью;
- если запрос из `/new`, но карточка уже помечена `in_plex`, не дублировать слишком много текста;
- если lookup требует сетевого запроса на каждый результат.

### Источник данных

Использовать только дешёвые данные:

1. Текущий in-memory Plex cache.
2. Уже существующие `_plex_pre_check` / `_plex_pre_check_series` или тонкую обёртку вокруг них.
3. Pre-warm для первого partial-result уже есть; его можно не расширять на весь список в первом шаге.

Не делать полный Plex refresh внутри каждого поиска. Если cache пустой или старый, лучше не показывать раннюю подсказку, чем тормозить поиск.

### Уровни уверенности

Ранний Plex hint должен быть отдельным полем результата, например:

```text
plex_hint:
  kind: movie | series | none
  action: same | better_exists | upgrade_available | partial_exists | other_seasons | unknown
  title: str
  quality: str
  season: int | None
  confidence: high | medium | low
```

Показывать пользователю только `high` и часть `medium`:

- `high`: same title/year или exact show+season;
- `medium`: title совпал, year/season не полностью подтверждены, формулировка осторожная;
- `low`: не показывать.

### Влияние на кнопки

В первой реализации кнопки не меняем. Только текстовая подсказка.

Возможное следующее улучшение после проверки:

- для `upgrade_available` можно рядом с `⬇️ N` не добавлять отдельную кнопку, потому что существующий confirm-dialog всё равно предложит `Заменить версией получше`;
- для `better_exists` можно оставить обычное скачивание, но пользователь уже предупреждён до клика.

### Ошибки

Если Plex lookup упал:

- не показывать ошибку пользователю;
- не ломать выдачу;
- логировать debug/info без raw token/url;
- финальный `_download_and_add` всё равно сделает свою проверку.

### Ложные совпадения

Особенно опасные случаи:

- remake с тем же названием, но другим годом;
- сериал и фильм с одинаковым названием;
- season pack `S1-S6`, который может совпасть с одним сезоном;
- аниме/specials/нулевой сезон;
- русское и оригинальное название расходятся.

Правило: при сомнении не показывать early hint.

## Тестовая матрица

### Deterministic parser

- `дюна 4к` -> base `дюна`, quality `4K`.
- `дюна 2160p original subs` -> quality `4K`, original, subs.
- `дюна 1080р` с русской `р` -> quality `1080p`.
- `дюна без субтитров` -> subs false.
- `дюна в оригинале с русскими субтитрами` -> original true, subs true.
- `клиника 3 сезон 1080` -> season 3, one_release, quality 1080p.
- `клиника все сезоны 1080` -> series_master, quality 1080p.
- `сериал клиника` -> one_release/unknown, not series_master.
- `скачать клинику целиком lostfilm` -> series_master, voice hint LostFilm.
- `клиника в озвучке lostfilm` -> voice hint LostFilm, `voice_required=true`.
- `клиника без lostfilm` -> LostFilm не попадает в wanted voices.
- `офис 4к 720` -> low/conflict.
- `дюна без субтитров с субтитрами` -> low/conflict.
- `найди` -> empty base -> invalid.

### GPT validation

- valid JSON maps to draft.
- unknown quality ignored.
- unknown intent ignored.
- season 0/1000 ignored.
- extra fields ignored.
- malformed JSON falls back.
- timeout falls back.
- quota/auth falls back but diagnostics can record.

### Text integration

- simple title still reaches existing search with default quality.
- explicit settings prefill `srch_settings`.
- `все сезоны` sets `srch_intent`.
- explicit voice requirement filters/prioritizes matching voice when found.
- explicit voice requirement falls back to alternatives with a clear banner when not found.
- conflict shows options instead of direct search.
- existing magnet and KP URL routing not broken.

### Personal defaults

- personal default overrides global default.
- explicit query settings override personal default.
- defaults change only through `/settings`, not from search results.
- reset defaults removes only current user's entry.
- saving failure shows friendly text and search still works.
- defaults are keyed by `chat_id`; another user's defaults are not affected.
- `/settings` opens defaults screen from Telegram menu.
- preferred voices are saved as normalized known labels, not raw free text.
- preferred voices boost matching results but do not hide alternatives.
- explicit voice in query overrides preferred voices.
- preferred voice gives bulk scoring a soft bonus only when found in the reference release.

### Voice integration

- transcription with `4к оригинал` fills settings.
- failed transcription unchanged.
- GPT failure does not break search.

### KP integration

- KP movie + quality fills settings.
- KP series + `все сезоны` sets `series_master`.
- KP movie + `все сезоны` does not set `series_master`.

### Partial picker

- partial result first button is `Скачать доступные + следить`.
- first button callback remains `sub_preset:each`.
- `Скачать только доступные` callback remains direct download.
- full series result does not show partial subscription rows.

### Early Plex hints

- movie result with same quality in Plex renders `Уже есть в Plex`.
- movie result lower than Plex renders `В Plex есть лучше`.
- movie result higher than Plex renders upgrade hint.
- series season already in Plex renders season hint.
- partial season with Plex partial data uses cautious wording.
- cluster picker marks seasons already present in Plex.
- cluster picker treats season year as season year, not show start year.
- cluster picker does not mark a movie/spinoff from a series match with the same prefix.
- season pack cluster can summarize `часть сезонов уже есть` or `все сезоны уже есть`.
- empty/stale Plex cache renders no cluster hint and no error.
- low-confidence match renders no hint.
- Plex disabled renders no hint.
- Plex lookup exception does not change search output.
- final `_download_and_add` pre-check still runs even when early hint was shown.

- только отдельная команда `/settings`;
- никаких save/reset-кнопок в результатах поиска;
- `quality/audio/subs/preferred_voices`;
- preferred voices мягкие, не строгий фильтр;
- без админского UI;
- без скрытого обучения по истории.

### Шаг 5. Подключить parser к text/voice/KP как prefill

Цель: пользователь видит `SEARCH_OPTIONS`, но настройки уже заполнены.

Граница:

- без auto-start поиска;
- меньше риска, легко проверить вручную;
- обновить `docs/bot-structure.md`.

### Шаг 6. Auto-start для high-confidence one-release

Цель: убрать лишний клик `Искать` для очевидных запросов.

Граница:

- только one-release;
- не для `series_master`;
- не для конфликтов;
- не для unknown tracker/voice hints.

### Шаг 7. GPT fallback

Цель: разговорные и voice-запросы разбираются лучше.

Граница:

- GPT only after deterministic;
- strict JSON validation;
- silent fallback on error;
- diagnostics only for admins.

## Открытые вопросы для согласования

1. Автозапуск поиска делать сразу в первой реализации или сначала только prefill + options?
2. Для `клиника все сезоны` сразу открывать `SEARCH_OPTIONS` в режиме `сериал целиком` или сразу запускать поиск эталона?
3. Разрешаем ли voice hints вроде `LostFilm` влиять на bulk profile сразу, если в эталоне такая озвучка найдена? Предлагаемое правило: да, если пользователь явно попросил эту озвучку.
4. Нужно ли распознавать `скачай` как более сильный сигнал, или всё равно требовать клик по конкретной раздаче? Предлагаемое правило: клик всё равно обязателен.
5. Ранние Plex hints делаем до подключения parser к Telegram flow или после? Предлагаемое правило: до, потому что это независимый безопасный UX-win.
6. Показывать ли early hint для `medium` confidence или только `high`? Предлагаемое правило: только `high` в первой реализации.
7. Персональные defaults включать в ближайший релиз? Предлагаемое правило: да, только через `/settings`, с `quality/audio/subs/preferred_voices`.
8. Нужен ли default tracker/indexer на пользователя? Предлагаемое правило: не в первом шаге, чтобы не смешивать UX-настройки с доступностью Jackett.
9. Делать ли preferred voices строгим фильтром по умолчанию? Правило: нет, только мягкий приоритет; строгим становится только ручной выбор озвучки в bulk-профиле.
