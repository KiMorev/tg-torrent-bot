# PlexLoader - массовая загрузка сезонов сериала

Проектный документ для крупной фичи v1.5: пользователь хочет скачать сериал целиком
или все недостающие сезоны, а бот должен сделать это с минимумом ручной работы и
без опасного автоподбора неподходящих раздач.

Главная идея: пользователь выбирает одну подходящую раздачу как отправную точку,
а PlexLoader строит план по сезонам, показывает спорные места и скачивает только
то, что можно выбрать уверенно.

## Цель

Добавить сценарий `Скачать все недостающие сезоны` для сериалов.

Фича считается успешной, когда:

- пользователь может стартовать сценарий из найденной раздачи сериала;
- настройки первоначального поиска переносятся в план;
- PlexLoader понимает, какие сезоны уже есть в Plex или уже качаются;
- бот строит понятный план по сезонам до начала массового скачивания;
- автоматом скачиваются только уверенные кандидаты;
- спорные сезоны не скачиваются молча, а требуют решения пользователя;
- неполный текущий сезон не маскируется под полный;
- длинные сериалы не создают десятки задач без отдельного предупреждения.

## Пользовательский сценарий

1. Пользователь ищет сериал обычным поиском.
2. В поиске он может заранее включить качество, оригинальную дорожку, субтитры и
   выбрать Jackett-трекеры.
3. Пользователь выбирает конкретную раздачу сериала.
4. Вместо ручного перехода по сезонам он нажимает `📚 Скачать недостающие сезоны`.
5. Бот показывает профиль подбора и спрашивает, насколько важна озвучка.
6. Бот строит план по сезонам.
7. Пользователь подтверждает скачивание уверенных кандидатов.
8. Спорные сезоны разбираются отдельно или пропускаются.

## Входная точка в UI

Кнопку массовой загрузки стоит показывать только для результатов, которые бот
уверенно распознал как сериал:

- в названии есть сезон;
- `_extract_series_base_query` вернул базовое название;
- есть номер сезона или понятный season-pack marker.

Пример экрана после нажатия `⬇️ N` у сериального результата:

```text
📺 Клиника / Сезон 3 / 1080p

Что скачать?
```

Кнопки:

```text
⬇️ Скачать эту раздачу
📚 Скачать недостающие сезоны
⬅️ К результатам
❌ Отмена
```

Для неполных сезонов текущие варианты подписки остаются отдельным блоком. Bulk
не должен смешиваться с подпиской молча.

## Профиль подбора

Профиль собирается из двух источников.

### Явные настройки поиска

Это прямое намерение пользователя. Эти параметры переносятся в bulk-план как
жёсткие фильтры по умолчанию:

- качество: `1080p`, `2160p`, `720p` или `any`;
- оригинальная дорожка;
- субтитры;
- выбранные Jackett-трекеры;
- базовый поисковый запрос.

Если пользователь включил `Original` в первоначальном поиске, bulk-план обязан
искать сезоны с оригинальной дорожкой. Если включил субтитры, субтитры тоже
обязательны. Переспрашивать это не нужно, но нужно показать в профиле.

### Выбранная раздача

Это эталон, но не абсолютная истина. Из неё можно извлечь мягкие предпочтения:

- тип релиза: `WEB-DL`, `WEBRip`, `BDRip`, `BDRemux`, `HDTV`;
- возможные озвучки;
- языки дорожек;
- релиз-группа;
- примерный размер сезона;
- tracker/source.

Эти параметры используются для скоринга и объяснения, но не должны становиться
жёсткими фильтрами без подтверждения пользователя.

## Озвучка и дорожки

Сложный кейс: у выбранной раздачи может быть 5-6 дорожек, а у длинного сериала
набор дорожек по сезонам может меняться. Поэтому нельзя требовать точного
совпадения всего набора.

После выбора эталонной раздачи бот показывает:

```text
📚 Профиль подбора

Из поиска:
✅ 1080p
✅ Оригинальная дорожка
✅ Субтитры

Из выбранной раздачи:
WEB-DL
Озвучки: LostFilm, NewStudio, Кравец

Что важно по озвучке?
```

Кнопки:

```text
🔘 Любая из найденных в выбранной раздаче
⚪ Любая русская озвучка
⚪ Выбрать конкретную
⚪ Только Original
🔍 Собрать план
⬅️ Назад
❌ Отмена
```

Рабочая политика озвучки:

| Политика | Что значит | Автоскачивание |
|---|---|---|
| `any_from_reference` | Подходит любая озвучка из найденных в эталонной раздаче. | Да, если есть хотя бы одно совпадение. |
| `any_russian` | Конкретная озвучка не важна, нужна русская дорожка. | Да, если раздача выглядит русскоязычной. |
| `require_selected` | Пользователь выбрал одну или несколько конкретных озвучек. | Да только при совпадении выбранной озвучки. |
| `original_only` | Нужна только оригинальная дорожка. | Да только если original явно найден. |

По умолчанию лучше использовать `any_from_reference`: это сохраняет связь с
выбранной пользователем раздачей, но не ломает длинные сериалы из-за изменения
числа дорожек.

Если бот не смог распознать озвучки в выбранной раздаче, показываем только:

```text
🔘 Любая русская озвучка
⚪ Только Original
🔍 Собрать план
```

## Ожидательный UI сборки плана

Сборка плана - долгий процесс. Для длинного сериала бот может делать широкий
поиск, несколько точечных поисков, проверку Plex и проверку активных задач
Download Station. Это нормально может занять несколько минут.

Перед стартом нужно честно предупредить пользователя:

```text
📚 Собираю план по сезонам

Это может занять несколько минут: проверю список сезонов, Plex, текущие загрузки
и раздачи на трекерах.
```

Кнопки:

```text
❌ Отмена
```

Во время работы бот должен обновлять одно и то же сообщение, а не присылать
цепочку новых сообщений. Промежуточные состояния не должны оставаться без
кнопки отмены.

Ожидательный экран должен использовать тот же анимированный файл, что и обычный
поиск: `SEARCH_ANIMATION_PATH` / `assets/searching_cats.mp4`. В Telegram это
отправляется как animation-сообщение рядом с текстовым статусом. После сборки
плана или отмены animation нужно удалить, как это уже делает `ProgressiveStatus`
для обычного поиска, чтобы в чате остался только итоговый экран плана.

Пример динамического прогресса:

```text
📚 Собираю план: Клиника

✅ Определяю список сезонов
✅ Проверяю Plex
✅ Проверяю текущие загрузки
🔎 Ищу раздачи на трекерах
⏳ Уточняю сезоны без кандидатов

Это может занять несколько минут.
```

Этапы прогресса:

1. `Определяю список сезонов` - Кинопоиск / fallback.
2. `Проверяю Plex` - какие сезоны уже есть.
3. `Проверяю текущие загрузки` - что уже качается в Download Station.
4. `Ищу раздачи на трекерах` - широкий запрос.
5. `Уточняю сезоны без кандидатов` - точечные запросы.
6. `Оцениваю кандидатов` - отсев, score, спорные статусы.
7. `Готовлю план` - финальный render.

Если процесс занимает дольше ожидаемого, сообщение обновляется мягким статусом:

```text
⏳ Всё ещё собираю план.
Некоторые сезоны ищутся отдельно, поэтому это может занять ещё пару минут.
```

Отмена должна:

- остановить дальнейшие точечные поиски, если это возможно;
- не создавать загрузки;
- удалить/закрыть ожидательное сообщение или показать `Отменено`;
- не оставлять частичный план как активную задачу.

Если часть источников упала, но план можно показать, бот не должен завершаться
ошибкой. Нужно показать деградированный план:

```text
⚠️ План собран не полностью: Jackett ответил не по всем трекерам.
Показываю то, что удалось проверить.
```

Если план собрать невозможно вообще, экран ошибки должен иметь:

```text
🔄 Попробовать снова
⬅️ Назад
❌ Отмена
```

## Источник списка сезонов

Основной источник количества сезонов - Кинопоиск, потому что поиск по трекерам
для длинных сериалов может не вернуть весь диапазон: fetch limit, разные
названия, разные индексеры, старые сезоны без сидов.

Порядок определения диапазона:

1. Кинопоиск: `1..N`, если клиент настроен и вернул количество сезонов.
2. Альтернативный metadata source: отдельная будущая интеграция, которую нужно
   выбрать и проверить перед реализацией. Кандидаты нужно оценивать отдельно по
   доступности API, лицензии, лимитам и качеству данных по русским/иностранным
   сериалам.
3. Plex: можно понять, что уже есть, но обычно нельзя надёжно узнать полный
   диапазон сезонов, если сериал неполный.
4. Tracker inventory fallback: собрать сезоны из результатов поиска, но пометить
   диапазон как непроверенный.

Если диапазон сезонов не подтверждён metadata source, план должен явно писать:

```text
Диапазон сезонов не подтверждён.
Показываю сезоны, которые удалось найти на трекерах.
```

## Как искать сезоны

Нужен гибридный подход.

### Шаг 1. Широкий запрос

Сначала делаем широкий поиск по базовому названию сериала с настройками исходного
поиска и выбранными трекерами:

```text
Клиника 1080p Original Sub
```

Технически в текущей модели запрос к трекерам должен оставаться чистым, а
качество/original/subs применяются клиентским фильтром, как в обычном поиске.

Цель широкого запроса:

- быстро собрать кандидатов по нескольким сезонам;
- увидеть season-pack раздачи;
- понять доступные озвучки/релиз-группы;
- не делать десятки сетевых запросов сразу.

### Шаг 2. Группировка

Результаты группируются по номеру сезона:

- `Сезон: N`;
- `N-й сезон`;
- `SNN`;
- `SNNE...`;
- отдельный тип `season_pack`, если раздача покрывает несколько сезонов.

Результаты без понятного сезона не участвуют в автоплане, но могут быть
показаны как спорные.

### Шаг 3. Точечные запросы

После широкого запроса бот делает точечный поиск только для сезонов, по которым:

- нет кандидата;
- есть только слабые кандидаты;
- широкий поиск явно обрезался лимитом;
- сезон входит в подтверждённый диапазон, но отсутствует в результатах.

Формат точечного запроса:

```text
Клиника Сезон: 5 1080p Original Sub
```

Так мы не полагаемся только на широкий поиск, но и не начинаем с N запросов для
каждого длинного сериала.

## Выбор кандидата

Для каждого сезона планировщик выбирает лучший результат и присваивает ему
уровень уверенности.

Важно разделить два шага:

1. **Отсев неподходящих раздач.** Всё, что нарушает обязательные условия,
   не может стать автокандидатом.
2. **Оценка достойных кандидатов.** Среди оставшихся бот выбирает лучший и
   объясняет, почему он достаточно хорош или почему нужен выбор пользователя.

### Обязательный отсев

- совпадает базовое название сериала;
- совпадает номер сезона;
- сезон не найден в Plex, если Plex включён;
- сезон не качается уже в Download Station;
- качество совпадает, если пользователь выбрал конкретное качество;
- original/subs совпадают, если они были включены в первоначальном поиске;
- политика озвучки выполнена.

Если раздача провалила обязательный отсев, она не скачивается автоматически.
Она может попасть только в список спорных вариантов, если выглядит близкой и
может быть осознанно выбрана пользователем.

### Score достойного кандидата

После отсева кандидат получает score. Точные веса можно уточнить при реализации,
но модель должна быть стабильной и объяснимой:

| Признак | Влияние |
|---|---|
| Точное совпадение сезона | Обязательное условие. |
| Совпадение качества из поиска | Обязательное условие, если качество не `any`. |
| Original/Sub из поиска | Обязательное условие, если включено пользователем. |
| Политика озвучки выполнена | Обязательное условие. |
| Тип релиза совпадает с эталоном | Большой плюс. |
| Релиз-группа совпадает с эталоном | Средний плюс. |
| Tracker/source совпадает с эталоном | Средний плюс. |
| Есть несколько озвучек из эталона | Плюс, но не требование. |
| Размер сезона похож на соседние сезоны | Плюс. |
| Много сидов | Плюс с потолком, чтобы сиды не перебивали качество. |
| Название содержит признаки чужого сериала | Сильный минус или отсев. |
| Раздача выглядит как pack нескольких сезонов | Отдельный тип, не автокандидат MVP. |
| Неполный сезон `5 из 8` | Статус `partial`, не обычный кандидат. |

Практическое правило: достойный кандидат - это раздача, которая прошла
обязательный отсев, не имеет красных флагов и набрала достаточно score, чтобы
её можно было объяснить пользователю одной строкой.

Пример объяснений:

```text
✅ Сезон 4 - WEB-DL 1080p, LostFilm, Original
Причина: совпали качество, original, озвучка из эталона и тип релиза.

⚠️ Сезон 5 - WEB-DL 1080p, NewStudio
Причина: качество совпало, но original не найден в названии.
```

### Красные флаги

Красные флаги переводят кандидата в `needs_decision` или отсекают его:

- сезон не совпадает или не распознан;
- название похоже на другой сериал;
- качество ниже выбранного пользователем;
- нет original/subs, хотя пользователь включил их в поиске;
- не выполнена выбранная политика озвучки;
- размер сильно выбивается относительно соседних сезонов;
- слишком мало сидов при наличии нормальной альтернативы;
- результат выглядит как pack, а план строится по отдельным сезонам;
- результат неполный.

### Пороги уверенности

Рабочая модель порогов:

- `exact` - обязательные условия выполнены, нет красных флагов, совпали ключевые
  мягкие признаки: release type, озвучка/source или релиз-группа.
- `good` - обязательные условия выполнены, красных флагов нет, но часть мягких
  признаков отличается. Такой кандидат можно скачать после общего подтверждения,
  но в плане нужно показать отличие.
- `needs_decision` - есть близкие кандидаты, но есть важное отличие или несколько
  вариантов слишком близки по score.
- `missing` - после широкого и точечного поиска нет кандидатов.

Уровни уверенности:

| Статус | Значение | Что делает бот |
|---|---|---|
| `exact` | Совпали жёсткие условия и сильные мягкие признаки. | Можно скачивать после общего подтверждения. |
| `good` | Жёсткие условия совпали, мягкие признаки частично отличаются. | Можно скачивать после общего подтверждения, но показать отличие. |
| `needs_decision` | Есть кандидаты, но важный признак спорный. | Не скачивать молча. |
| `missing` | Кандидатов нет. | Показать как не найдено. |
| `already_in_plex` | Сезон уже есть в Plex. | Пропустить. |
| `already_downloading` | Сезон уже есть в Download Station. | Пропустить. |
| `partial` | Сезон найден неполным. | Отдельное решение: скачать сейчас или подписаться. |

## Спорные случаи

### Несколько хороших кандидатов

Если у сезона несколько кандидатов с близким score, бот выбирает лучший, но
помечает сезон как спорный, если отличие касается важного признака:

- другая озвучка при `require_selected`;
- нет original при включённом `Original`;
- нет субтитров при включённых субтитрах;
- слишком разный размер;
- другой тип релиза при равных остальных параметрах.

В плане показывается `⚠️ Нужно выбрать раздачу`.

### Неполный последний сезон

Если найдено `5 из 8`, это статус `partial`.

Варианты:

- скачать доступные серии сейчас;
- скачать доступные серии сейчас и докачивать новые по мере выхода;
- ждать завершения сезона и скачать потом;
- только уведомлять о новых сериях;
- пропустить.

Это тот же смысл, что текущий блок подписок. Bulk не должен автоматически
включать подписку, но должен предложить её после построения плана.

### Пак всех сезонов

Пак `Сезоны 1-8` не выбирается автоматически в MVP.

Его можно показать отдельно:

```text
📦 Найден пак всех сезонов: 1-8, 1080p, WEB-DL
```

Кнопки:

```text
📦 Скачать одним паком
📚 Скачать по сезонам
```

Автовыбор пака опасен: он может включать лишнее, другое качество, другой перевод
или огромный объём.

### Уже есть в Plex

Если Plex включён, сезоны из Plex по умолчанию пропускаются. Это сценарий
`скачать недостающие сезоны`, а не `перекачать всё`.

Позже можно добавить режим замены качества, но не в MVP.

### Уже качается

Перед подтверждением плана нужно проверить Download Station:

- если сезон уже качается, статус `already_downloading`;
- если название совпадает с другим активным task, не добавлять дубль;
- если совпадение неуверенное, показать как спорное.

### Нет Кинопоиска или metadata source

Если не удалось определить полный диапазон сезонов, не обещать "все сезоны".
Текст должен быть честным:

```text
Нашёл сезоны на трекерах: 1, 2, 4, 5.
Полный список сезонов не подтверждён.
```

### Слишком много задач

Если план добавляет больше 20 задач, нужен отдельный экран подтверждения:

```text
Будет создано 27 задач в Download Station.
Это может занять много места и времени.
```

Кнопки:

```text
✅ Да, добавить 27 задач
⬅️ Вернуться к плану
❌ Отмена
```

## Экран плана

Пример:

```text
📚 План загрузки: Клиника

Профиль:
1080p · Original · Sub
Озвучка: любая из LostFilm / NewStudio / Кравец

В Plex уже есть: 1, 2
Уже качается: 3

✅ Сезон 4 - WEB-DL 1080p, Кравец, Original
✅ Сезон 5 - WEB-DL 1080p, LostFilm, Original
⚠️ Сезон 6 - найдено 2 кандидата, нужен выбор
⏳ Сезон 7 - доступно 5/8 серий
❌ Сезон 8 - не найдено

Скачаю сейчас: 2 сезона
Пропущу: 3 сезона
Нужно решение: 2 сезона
```

Кнопки:

```text
⬇️ Скачать уверенные
⚙️ Разобрать спорные
🔄 Пересобрать план
⬅️ Назад
❌ Отмена
```

Если уверенных кандидатов нет, кнопка `⬇️ Скачать уверенные` не показывается.

## Разбор спорных сезонов

Кнопка `⚙️ Разобрать спорные` открывает первый сезон со статусом
`needs_decision` или `partial`. Это не общий список всех найденных раздач, а
узкий экран по одному сезону, где понятно, почему бот не выбрал автоматически.

Пример:

```text
⚠️ Сезон 6 - нужен выбор

Профиль:
1080p · Original · Sub
Озвучка: любая из LostFilm / NewStudio / Кравец

Нашёл 2 кандидата:

1. WEB-DL 1080p · LostFilm · Original
   Причина: подходит по профилю, но другой release group.

2. WEB-DL 1080p · NewStudio
   Причина: нет явного Original, поэтому не скачиваю автоматически.
```

Кнопки:

```text
⬇️ Скачать 1
⬇️ Скачать 2
⏭ Пропустить сезон
🔄 Искать мягче
⬅️ К плану
❌ Отмена
```

Для `partial` сезона текст должен быть другим:

```text
⏳ Сезон 7 - доступно 5/8 серий

Этот сезон ещё не завершён. Автоматически как полный сезон не скачиваю.
```

Кнопки:

```text
⬇️ Скачать доступные серии
⬇️ Скачать доступные + новые по мере выхода
📦 Скачать, когда сезон завершится
🔔 Только уведомлять
⏭ Пропустить сезон
⬅️ К плану
❌ Отмена
```

Вариант `⬇️ Скачать доступные + новые по мере выхода` должен создавать обычную
подписку с политикой "новые серии по мере выхода": текущая неполная раздача
добавляется в Download Station, а следующие обновления сезона скачиваются
автоматически. Это bulk-эквивалент текущей кнопки
`⬇️ Скачать сейчас + новые серии по мере выхода`.

После любого выбора пользователь возвращается к обновлённому плану. Если спорных
сезонов больше нет, план показывает только оставшиеся действия:
`⬇️ Скачать уверенные`, `🔄 Пересобрать план`, `❌ Отмена`.

## Состояние

Есть отдельная сущность плана/джобы, чтобы фиксировать собранный план и
результаты действий по каждому сезону. Текущий Telegram-flow уже создаёт job и
обновляет её при скачивании/ручном выборе/подписке, но отдельного resume-экрана
после рестарта пока нет.

Рабочий файл: `series_bulk_jobs.json`.

Пример формы:

## MVP

В первую реализацию входит:

- вход из выбранной сериальной раздачи;
- перенос качества, original/subs и выбранных трекеров из первоначального поиска;
- экран выбора политики озвучки;
- динамический ожидательный UI сборки плана с предупреждением, что процесс может
  занять несколько минут;
- построение плана по сезонам;
- Кинопоиск как основной источник диапазона сезонов;
- tracker inventory fallback с честным предупреждением;
- гибридный поиск: широкий запрос + точечные запросы для проблемных сезонов;
- пропуск сезонов, которые уже есть в Plex;
- пропуск сезонов, которые уже качаются в Download Station;
- статусы `exact`, `good`, `needs_decision`, `missing`, `partial`,
  `already_in_plex`, `already_downloading`;
- подтверждение перед добавлением задач;
- лимит-предупреждение для большого числа задач;
- тесты планировщика, UI callbacks, state store и основных спорных кейсов.

## Не входит в MVP

- автоматическая замена сезонов, уже имеющихся в Plex;
- автоскачивание season-pack без выбора пользователя;
- идеальное распознавание всех студий озвучки;
- сравнение точного количества аудиодорожек;
- парсинг содержимого torrent-файла до скачивания;
- веб-интерфейс;
- новая база данных вместо JSON;
- полная интеграция с будущим request center.

## Тесты

Минимальный набор тестов:

- планировщик выбирает `exact` при совпадении сезона, качества, original/subs и
  политики озвучки;
- изменение числа дорожек не ломает `any_from_reference`, если есть хотя бы одна
  совпавшая озвучка;
- `require_selected` не пропускает сезон без выбранной озвучки;
- original/subs из первоначального поиска становятся жёсткими фильтрами;
- сезоны из Plex получают `already_in_plex`;
- активные Download Station задачи получают `already_downloading`;
- неполный сезон получает `partial`;
- season-pack не выбирается автоматически;
- без Кинопоиска план помечается как непроверенный;
- точечный поиск запускается для сезонов, которых не хватило после широкого
  запроса;
- partial-действие "скачать доступные + новые по мере выхода" создаёт загрузку и
  подписку с правильной политикой;
- ожидательный UI показывает этапы, сохраняет кнопку отмены и переходит в план
  или понятную ошибку;
- UI callbacks не превышают лимиты Telegram callback-data;
- финальные и ошибочные экраны имеют `⬅️ Назад` / `❌ Отмена` / `✖️ Закрыть`
  согласно правилам кнопок.
