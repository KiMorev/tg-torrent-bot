# План доработок вокруг истории загрузок

Документ фиксирует идеи, которые опираются на `download_history.jsonl`. Это не обязательства на ближайший релиз, а backlog, чтобы позже вернуться к теме без повторного разбора.

Базовые правила для всех пунктов:

- история всегда привязана к конкретному `chat_id`;
- пользовательские действия не выполняются молча: история может предлагать, но скачивание подтверждает пользователь;
- в payload не добавляем magnet-ссылки, Jackett `/dl` URL, API keys, пароли, torrent-содержимое и raw-ответы сервисов;
- для рекомендаций используем в первую очередь успешные события: `download_added` + `download_completed` / `download_soft_completed` / `plex_found`;
- если уверенность низкая, бот показывает подсказку как вариант, а не делает вид, что точно знает правильный выбор.

## P0. Самая заметная польза

### 1. "Скачать как в прошлый раз"

Пользовательская польза: меньше ручного выбора качества, трекера, озвучки и настроек, особенно для сериалов.

Как может выглядеть:

- пользователь ищет фильм или сериал;
- бот видит похожий прошлый успешный выбор этого пользователя;
- рядом с обычными действиями показывает короткую подсказку:
  - `Как в прошлый раз: 1080p, RuTracker, LostFilm + Original`;
  - кнопка `⚡ Как в прошлый раз`;
  - рядом остаются обычные варианты, чтобы не запирать пользователя в прошлый выбор.

Технический план:

- добавить helper поиска релевантной истории по `chat_id`, `kind`, `canonical_title` / `series_query`;
- брать последнюю успешную запись, где есть `quality`, `tracker`/`indexer`, `topic_id` или release profile;
- при поиске релизов ранжировать кандидатов выше, если они похожи на прошлый профиль;
- перед скачиванием показать подтверждение с объяснением, почему выбран именно этот вариант.

Риски:

- прошлый выбор мог быть случайным;
- для длинных сериалов набор дорожек меняется от сезона к сезону;
- нельзя слепо требовать точное совпадение всех аудио/субтитров, иначе бот часто ничего не найдёт.

Первый безопасный MVP: только показывать кнопку-подсказку, без автоматического выбора в фоне.

### 2. Автозаполнение профиля для bulk-сезонов

Пользовательская польза: при скачивании всех сезонов бот сразу предлагает настройки, похожие на уже выбранную пользователем раздачу.

Как может выглядеть:

- пользователь нажимает `📚 Скачать недостающие сезоны`;
- профиль подбора уже заполнен:
  - качество из текущей раздачи или истории;
  - Original/subtitles из текущего поиска;
  - озвучка: `как в выбранной раздаче` или конкретные студии, если пользователь явно выбрал их раньше.

Технический план:

- расширить bulk profile builder чтением `download_history.jsonl`;
- приоритет источников:
  1. текущая выбранная раздача;
  2. последняя успешная история этого же сериала;
  3. общие предпочтения пользователя из истории;
- в UI явно помечать, откуда взята настройка: `из выбранной раздачи` / `по прошлому выбору`.

Риски:

- история не должна перебивать текущую раздачу, которую пользователь только что выбрал;
- нужно аккуратно разделять "пользователь всегда любит 1080p" и "у конкретного сериала был только 1080p".

### 3. Предупреждение "вы уже качали эту раздачу"

Пользовательская польза: меньше дублей, даже если Plex ещё не сматчил файл или контент уже удалён из библиотеки.

Как может выглядеть:

- пользователь выбирает раздачу;
- бот видит в истории тот же `topic_id` или очень близкий title;
- показывает предупреждение:
  - `Вы уже добавляли эту раздачу 12.05.2026`;
  - кнопки `Скачать всё равно` и `⬅️ Назад`.

Технический план:

- проверять совпадение по `topic_id` как самый надёжный сигнал;
- fallback: `canonical_title` + `year` + `quality` + нормализованный title;
- не блокировать скачивание, только предупреждать.

Риски:

- повторное скачивание иногда нужно сознательно: файл удалили, нужен другой диск, Plex не увидел прошлый файл;
- предупреждение не должно появляться на каждый похожий сезон как ложная тревога.

### 4. Подбор следующего сезона "в том же стиле"

Пользовательская польза: пользователь скачал сезон сериала, потом хочет следующий, а бот сам подтягивает качество/озвучку/трекер по прошлому выбору.

Как может выглядеть:

- после завершения сезона или из `/status` появляется действие `🔎 Следующий сезон`;
- бот ищет следующий сезон и ставит выше кандидатов с похожим профилем;
- если совпадение сильное, показывает `Похоже на прошлый сезон`.

Технический план:

- использовать `series_query`, `season`, `quality`, `release` из истории;
- искать `season + 1`;
- не требовать точное совпадение всех аудиодорожек, а считать score по профилю.

Риски:

- не всегда понятно, какой сезон следующий, если в истории есть паки, specials или неполные сезоны;
- для сериалов без Kinopoisk metadata нужна осторожная fallback-логика.

## P1. Сильные улучшения, но не первый шаг

### 5. Личные предпочтения пользователя

Пользовательская польза: бот со временем лучше сортирует выдачу под конкретного человека.

Возможные сигналы:

- чаще выбираемое качество;
- любимые трекеры/indexers;
- частое требование Original;
- частое наличие субтитров;
- предпочитаемые озвучки/языки.

Технический план:

- не хранить отдельный "профиль" вручную на первом этапе;
- вычислять предпочтения на лету из последних N успешных записей;
- позже, если понадобится скорость, добавить materialized cache.

Риски:

- история одного-двух скачиваний не должна слишком сильно влиять на выдачу;
- для разных типов контента предпочтения могут отличаться: фильмы, сериалы, мультфильмы.

### 6. Экран "последние загрузки" для пользователя

Пользовательская польза: можно быстро вернуться к тому, что недавно добавлял, и понять, чем закончилась загрузка.

Возможный UI:

- команда или кнопка `🕘 Последние загрузки`;
- список последних 5-10 действий:
  - название;
  - дата;
  - статус: добавлено, скачано, найдено в Plex, Plex не подтвердил;
  - кнопка `Показать задачу`, если task id ещё актуален.

Технический план:

- читать только записи текущего `chat_id`;
- группировать события по `task_id`;
- показывать не raw history, а итоговую карточку на задачу.

Риски:

- это уже пользовательский экран, значит нужны кнопки, пагинация и правила закрытия;
- task id может устареть, если задача удалена из Download Station.

### 7. Умнее обрабатывать Plex not found

Пользовательская польза: если Plex не нашёл скачанное, бот может подсказать, что делать дальше, не начиная расследование с нуля.

Как может выглядеть:

- после `plex_not_found` бот знает, какой был tracker/topic/title/task;
- в админском или пользовательском разборе можно показать:
  - какая раздача скачивалась;
  - был ли soft-complete DS;
  - какой canonical title использовался для Plex lookup.

Технический план:

- сгруппировать историю по `task_id`;
- добавить helper "итог по задаче";
- использовать его в будущем экране диагностики конкретной загрузки.

Риски:

- не превращать обычный пользовательский экран в технический лог;
- админские подробности и пользовательские подсказки должны отличаться.

### 8. История как страховка после рестарта

Пользовательская польза: меньше потерянных контекстов, если бот перезапустился между добавлением задачи и финальным Plex polling.

Технический план:

- при старте сверять активные Download Station tasks с `download_history.jsonl`;
- для задач с владельцем, но без `task_meta`, пытаться восстановить минимум meta из истории;
- не заменять `task_meta.json`, если он уже есть.

Риски:

- это fallback, не основной источник истины;
- восстановленная meta может быть хуже исходной.

## P2. Аналитика и эксплуатация

### 9. Надёжность трекеров и источников

Пользовательская польза: бот может реже предлагать источники, которые часто падают именно у этого пользователя/инсталляции.

Что считать:

- успешные добавления по tracker/indexer;
- ошибки добавления;
- `download_soft_completed`;
- `plex_not_found` после конкретного источника.

Технический план:

- построить агрегатор по истории;
- сначала использовать только в админской диагностике;
- позже осторожно учитывать в scoring.

Риски:

- нельзя наказывать трекер за временную проблему навсегда;
- маленькая выборка легко вводит в заблуждение.

### 10. Privacy/retention настройки

Пользовательская польза: владелец инсталляции контролирует, сколько истории хранится.

Варианты:

- хранить последние N месяцев;
- хранить последние N событий;
- ручная очистка истории;
- экспорт для диагностики без секретов.

Технический план:

- добавить pruning append-only файла;
- не делать до появления реальной потребности, чтобы не усложнять state store заранее.

Риски:

- очистка истории ломает будущие "как в прошлый раз";
- экспорт должен повторно проходить через security filter.

### 11. Миграция старых task-состояний в историю

Пользовательская польза: история станет полезной сразу после обновления, а не только после новых скачиваний.

Источники:

- `task_meta.json`;
- `task_owners.json`;
- `notified_tasks.json`;
- `series_bulk_jobs.json`.

Технический план:

- сделать одноразовый idempotent migration marker;
- создавать только безопасные синтетические события;
- не пытаться восстановить то, чего нет в старых state-файлах.

Риски:

- старые данные неполные;
- можно создать ложную уверенность, что задача точно была успешной.

## P3. Нишевые идеи

### 12. Семейные рекомендации по истории

Пользовательская польза: если несколько пользователей в одной домашней инсталляции часто выбирают похожее, бот может лучше сортировать общую выдачу.

Почему нишево:

- приватность важнее удобства;
- по умолчанию история должна оставаться персональной;
- нужна явная настройка opt-in.

### 13. "Перекачать лучшее качество"

Пользовательская польза: бот может заметить, что раньше скачивали 720p, а теперь появился хороший 1080p/4K вариант.

Почему нишево:

- частично пересекается с Plex quality checks;
- может провоцировать лишние скачивания;
- нужно точно понимать, что старый файл действительно хуже.

### 14. История как источник объяснений GPT

Пользовательская польза: GPT-пояснения могут учитывать прошлые предпочтения пользователя.

Пример:

- `Этот вариант похож на ваши прошлые выборы: 1080p, WEB-DL, Original + русская дорожка`.

Почему нишево:

- можно сделать без GPT обычным текстом;
- нельзя отправлять в GPT лишние пользовательские данные без явной причины.

## Рекомендуемый порядок реализации

1. `P0.1` - кнопка `⚡ Как в прошлый раз` как явная подсказка без авто-скачивания.
2. `P0.2` - автозаполнение bulk-профиля из текущей раздачи и истории.
3. `P0.3` - предупреждение о повторной загрузке той же раздачи.
4. `P0.4` - подбор следующего сезона в похожем профиле.
5. `P1.5` - лёгкие пользовательские предпочтения для сортировки выдачи.
6. `P1.6` - пользовательский экран последних загрузок, только если станет понятно, что он реально нужен.

Первый рабочий коммит лучше держать маленьким: только helper поиска прошлой успешной записи + UI-подсказка в одном флоу. Не стоит сразу строить большой preference engine.
