# PlexLoader v1.5 - Центр запросов

Проектный документ для крупной фичи версии 1.5. Цель - заранее зафиксировать
границы, пользовательскую модель и порядок реализации, чтобы фича не расползлась
в переписывание половины бота.

## Цель версии

Сделать в PlexLoader единое место, где пользователь видит всё, что он попросил у
бота: текущие загрузки, отложенные попытки, подписки на серии, ожидание финала
сезона, проверку появления в Plex, ошибки и действия, которые требуют решения.

Рабочее пользовательское название: **Центр запросов** или **Мои запросы**.

Главная идея: пользователь думает не категориями Download Station task,
Jackett/Rutracker subscription, pending download или Plex polling, а простым
вопросом:

> Я попросил PlexLoader достать контент. Где он сейчас и что будет дальше?

## Пользовательская боль

Сейчас состояние запроса размазано по разным экранам и внутренним механизмам:

- `/status` показывает Download Station задачи, но не объясняет подписки и
  ожидание будущих серий.
- `pending_downloads` живёт как очередь повторных попыток, но пользователь видит
  только отдельные уведомления.
- Подписки на сериалы живут отдельно от факта, что пользователь уже что-то
  скачал или ждёт финал сезона.
- Plex polling после завершения загрузки выглядит как отдельная магия.
- В ошибочных состояниях сложно ответить на простой вопрос: "бот ещё что-то
  делает или уже ждёт меня?"

v1.5 должна дать один понятный слой поверх этих механизмов.

## MVP

В v1.5 входит:

- Новая сущность `request` как слой состояния над существующими механизмами.
- Новый экран списка запросов: команда `/requests` или раздел внутри `/status`.
- Детальный экран одного запроса.
- Понятные пользовательские статусы без внутренних терминов.
- Связь запроса с текущими сущностями:
  - Download Station `task_id`;
  - `task_meta`;
  - `pending_downloads`;
  - `topic_subscriptions`;
  - Plex polling / Plex result.
- Минимальная история событий по запросу: создан, добавлен в загрузку, ошибка,
  повторная попытка, найдено в Plex.
- Возможность закрыть/скрыть завершённый запрос из активного списка.
- Тесты на модель состояния, рендер списка, callback-data и основные переходы.

## Что не входит в v1.5

Чтобы версия осталась подъёмной, в v1.5 не делаем:

- новую базу данных вместо JSON;
- полную миграцию всех старых state-файлов в новый формат;
- сложный multi-user request marketplace;
- идеальный движок сериалов с планированием нескольких сезонов;
- автоматический подбор лучшего релиза без участия текущего поиска;
- рекомендации и персональную ленту поверх `/new`;
- веб-интерфейс;
- глобальную перепись `bot.py`.

## Пользовательские сценарии

### Скачать фильм

1. Пользователь находит фильм через поиск или `/new`.
2. Нажимает скачать.
3. Создаётся `request` со статусом `Скачивается`.
4. В деталях видно Download Station задачу и следующий шаг: дождаться окончания.
5. После завершения статус меняется на `Проверяем Plex`.
6. Если Plex нашёл файл - `Добавлено в Plex`.

### Скачать серию и следить за обновлениями

1. Пользователь находит неполный сезон.
2. Выбирает "Скачать сейчас + обновлять".
3. Создаётся `request`, связанный и с текущей Download Station задачей, и с
   подпиской.
4. После завершения текущей загрузки запрос остаётся активным в статусе
   `Ждём новые серии`.
5. При новой серии история запроса пополняется событием, а статус временно
   возвращается в `Скачивается`.

### Ждать полный сезон

1. Пользователь выбирает "Скачать полный сезон, когда выйдет".
2. Создаётся `request` без текущей Download Station задачи.
3. Статус: `Ждём финал сезона`.
4. Когда подписка видит полный сезон, запрос переходит в `Скачивается`.

### Отложенная загрузка

1. Download не удался, но есть смысл повторить позже.
2. Создаётся или обновляется `request` со статусом `Ждём повторной попытки`.
3. В деталях видно последнюю ошибку, количество попыток и следующий retry.
4. При успехе запрос связывается с новым `task_id`.

### Нужна помощь пользователя

1. Бот не может продолжить автоматически: источник недоступен, раздача пропала,
   Plex не находит файл слишком долго, или нужна ручная развилка.
2. Статус: `Нужно действие`.
3. Детальный экран показывает короткую причину и доступные кнопки.

## Экраны и кнопки Telegram

### Список запросов

Команда: рабочий вариант `/requests`.

Содержимое:

- активные запросы пользователя;
- короткий статус каждой строки;
- дата последнего события;
- пагинация, если запросов много.

Базовые кнопки:

- открыть детали запроса;
- фильтр `Активные`;
- фильтр `Требуют действия`;
- фильтр `Завершённые`;
- `✖️ Закрыть`.

### Детали запроса

Показывает:

- название;
- тип: фильм, сериал, сезон, раздача, ручной magnet/torrent;
- текущий статус;
- следующий шаг;
- связанные сущности: DS task, подписка, pending retry, Plex;
- последние события.

Возможные кнопки зависят от статуса:

- `🔄 Обновить`;
- `⬇️ Скачать сейчас`;
- `🔔 Настроить подписку`;
- `🗑️ Убрать из активных`;
- `✖️ Закрыть`.

Точные названия кнопок утверждаются при реализации UI. Обязательные правила
кнопок остаются из `AGENTS.md`: финальные экраны закрываются через
`task:close:`/эквивалентный глобальный close, промежуточные состояния не должны
оставлять пользователя без клавиатуры.

## Статусы запроса

| Статус для пользователя | Внутренний код | Смысл |
|---|---|---|
| `Создан` | `created` | Запрос зафиксирован, но действие ещё не начато. |
| `Готов к скачиванию` | `ready` | Есть выбранный релиз, пользователь может подтвердить скачивание. |
| `Скачивается` | `downloading` | Есть активный Download Station task. |
| `Ждём повторной попытки` | `queued_retry` | Запрос связан с `pending_downloads`. |
| `Ждём новые серии` | `watching_updates` | Подписка активна, текущий известный прогресс не финальный. |
| `Ждём финал сезона` | `waiting_finale` | Подписка активна, скачивание нужно только после полного сезона. |
| `Проверяем Plex` | `checking_plex` | Download Station завершил задачу, Plex polling ещё идёт. |
| `Добавлено в Plex` | `in_plex` | Plex нашёл фильм/сериал/сезон. |
| `Нужно действие` | `needs_action` | Бот не может безопасно продолжить без выбора пользователя. |
| `Ошибка` | `failed` | Автоматический путь исчерпан или ошибка финальная. |
| `Завершён` | `done` | Запрос завершён без дополнительных действий. |
| `Скрыт` | `archived` | Пользователь убрал запрос из активного списка. |

Статусы должны быть производными от текущего состояния, а не вторым источником
истины. Если связанная Download Station задача или подписка изменилась, request
должен пересчитаться или обновиться при ближайшем фоновом цикле.

## Модель `request`

Рабочий файл состояния: `requests.json`.

Пример формы:

```json
{
  "req_20260524_abc123": {
    "id": "req_20260524_abc123",
    "chat_id": 100,
    "kind": "series_season",
    "title": "Название",
    "year": 2024,
    "season": 1,
    "status": "watching_updates",
    "source": "search",
    "created_at": "2026-05-24T20:30:00Z",
    "updated_at": "2026-05-24T21:10:00Z",
    "links": {
      "ds_task_id": "dbid_123",
      "pending_id": "",
      "subscription_key": "rt:123456",
      "plex_rating_key": ""
    },
    "next_action": "Ждём новые серии",
    "last_error": "",
    "events": [
      {
        "at": "2026-05-24T20:30:00Z",
        "type": "created",
        "text": "Запрос создан из поиска"
      }
    ]
  }
}
```

Минимальные поля MVP:

- `id`;
- `chat_id`;
- `kind`;
- `title`;
- `status`;
- `source`;
- `created_at`;
- `updated_at`;
- `links`;
- `next_action`;
- `last_error`;
- `events`.

Историю событий нужно ограничивать, например последними 20 событиями на запрос,
чтобы JSON не рос бесконечно.

## Связь с текущими механизмами

### `task_meta`

`task_meta` остаётся источником метаданных Download Station задачи. `request`
ссылается на `task_id` и может подтягивать title/year/quality/season из
`task_meta`, но не обязан дублировать все поля.

### `pending_downloads`

Если скачивание ушло в pending retry, `request.links.pending_id` связывает
запрос с очередью. При успешной попытке pending entry удаляется, а request
получает новый `ds_task_id`.

### `topic_subscriptions`

Подписка остаётся существующим механизмом проверки новых серий. `request`
ссылается на ключ подписки и показывает её человекочитаемое состояние:
`Ждём новые серии` или `Ждём финал сезона`.

### Plex polling

Plex polling остаётся частью фоновой логики завершения задачи. `request`
отражает пользовательский статус: `Проверяем Plex`, `Добавлено в Plex` или
`Нужно действие`, если Plex не нашёл файл после лимита попыток.

### `/status`

`/status` не нужно ломать в первом релизе. Он может остаться экраном Download
Station задач, а Центр запросов стать отдельной командой. После стабилизации
можно решить, объединять ли эти экраны.

## Миграция и совместимость

Подход v1.5: постепенная совместимость, а не big bang migration.

- Новые действия создают `request` сразу.
- Старые Download Station задачи без request продолжают отображаться в
  `/status`.
- При открытии `/requests` можно показать только новые запросы MVP, не пытаясь
  восстановить всю историю задним числом.
- Для активных старых задач допустима lazy-связка: если есть `task_meta` и
  владелец задачи, можно создать request при первом касании. Это не обязательно
  для первого MVP.
- Старые `topic_subscriptions` не обязательно мигрировать сразу. Их можно
  показывать в `/subs`, пока новые подписки уже создают request.

## План реализации

1. Описать state API для `requests.json` в `state_store.py`.
2. Добавить чистую модель/политики request-статусов в отдельный модуль, без
   Telegram UI.
3. Подключить создание request при успешном добавлении загрузки из поиска.
4. Подключить создание request для pending download.
5. Сделать `/requests`: список активных запросов + детали одного запроса.
6. Подключить обновление request из `_run_task_notifications_once` и Plex
   polling.
7. Подключить создание request для новых подписок на сериалы.
8. Добавить действия "убрать из активных" и "обновить".
9. Обновить README: новая команда, пользовательский сценарий, ограничения.
10. Прогнать полный `python -m pytest tests/ -v`.

## Критерии готовности v1.5

Фича готова, когда:

- новый запрос создаётся при обычном скачивании из поиска;
- новый запрос создаётся при отложенной загрузке;
- новый запрос создаётся при новой подписке на сериал;
- `/requests` показывает активные запросы пользователя;
- у каждого запроса есть детальный экран с понятным статусом и следующим шагом;
- завершение Download Station задачи обновляет request;
- Plex polling отражается в request;
- пользователь может убрать завершённый запрос из активного списка;
- существующие `/status`, `/subs`, `/new`, pending downloads и уведомления не
  ломаются;
- новые state-переходы покрыты тестами;
- README и `docs/bot-structure.md` обновлены по фактической реализации.

## Открытые вопросы

- Делать `/requests` отдельной командой сразу или сначала встроить в `/status`?
- Нужно ли показывать завершённые запросы по умолчанию или только по фильтру?
- Какой срок хранения архивных запросов: бесконечно, N дней или до ручной
  очистки?
- Нужна ли кнопка "повторить поиск" из failed/needs_action состояния в MVP?
- Как назвать пользовательский экран окончательно: "Мои запросы", "Центр
  запросов" или "Ожидания и загрузки"?
