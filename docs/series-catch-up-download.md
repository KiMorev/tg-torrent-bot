# Докачивание неполного сезона

Фича для сценария: пользователь уже скачал неполный сезон, посмотрел доступные серии в Plex и позже хочет получить остальные серии без повторного ручного подбора раздачи.

Цель: дать простой вход `📺 Докачать сезон`, где PlexLoader сам находит неполные сезоны, понимает прошлый профиль скачивания, честно объясняет уровень уверенности и предлагает понятное действие.

## Принятые решения

- Медиатека семейная: любой разрешённый пользователь может докачать любой неполный сезон из Plex.
- В списке нужен переключатель `🙋 Моё` / `🌐 Всё`:
  - `Моё` показывает сезоны, связанные с историей текущего `chat_id`;
  - `Всё` показывает общие Plex-кандидаты и сезоны из истории других пользователей.
- Нужна новая команда `/continue`; кнопка в меню может называться `📺 Докачать сезон` или `📺 Продолжить сериал`.
- Если та же тема обновилась и сезон ещё не полный, основное действие по умолчанию: `⬇️ Докачать и следить`.
- Основной кандидат для докачки должен совпадать по качеству с текущим/прошлым выбором. Если найден вариант лучше, бот может показать его отдельной строкой `Есть версия лучше`, но не подменяет им действие "докачать".
- На первом экране показываем до 10 кандидатов, дальше пагинация кнопками `⬅️` и `➡️`.
- Первый старт после внедрения фичи работает через Plex-only fallback: история старых загрузок может быть пустой, но бот всё равно должен показать неполные сезоны из Plex.
- Plex Series agent не даёт KP id как базовый идентификатор. На живом примере Plex отдаёт `plex://show/...` и внешние `Guid`: `imdb://...`, `tmdb://...`, `tvdb://...`. Поэтому KP - только fallback, а не основа identity.
- `viewedLeafCount` и watch-state не используем в MVP как критерий кандидата. Пользователь может захотеть докачать сезон до просмотра или в момент, когда просто дошёл до последнего доступного эпизода.
- Первый экран не должен запускать долгую полную сборку архива. Используем быстрый Plex scan + ленивое обогащение выбранного сезона.

## Главный принцип формулировок

Слово "докачать" используем только когда бот действительно опирается на тот же `topic_id` / ту же тему раздачи.

Причина: если тема та же и раздача обновилась с `1-8` до `1-12` или `1-18`, пользовательский смысл близок к "докачать недостающие серии". Download Station обычно сможет переиспользовать уже скачанные файлы при совпадающей структуре, а пользователь не воспринимает это как новый поиск.

Если найден другой `topic_id`, pack, repack или похожая раздача, формулировка должна быть осторожнее:

- не `докачаем`;
- а `скачаем обновлённую раздачу`;
- или `подберём продолжение`;
- или `нашёл похожий вариант, проверьте перед скачиванием`.

## Пользовательский результат

Пользователь должен сделать минимум действий:

1. Нажать `📺 Докачать сезон`.
2. Выбрать сериал/сезон из списка неполных.
3. Увидеть, что уже есть в Plex и что найдено на трекере.
4. Подтвердить действие:
   - `⬇️ Докачать и следить`, если `topic_id` тот же и сезон ещё не полный;
   - `⬇️ Докачать из той же раздачи`, если `topic_id` тот же и сезон уже стал полным;
   - `⬇️ Скачать обновлённую раздачу`, если это другой кандидат;
   - `🔔 Следить за новыми сериями`, если продолжение ещё не найдено;
   - `📦 Скачать полный сезон, когда выйдет`, если сезон ещё выходит.

## Источники данных

### 1. История загрузок

Главный источник для точного сценария.

Что берём из `download_history.jsonl`:

- `chat_id`;
- `kind=series`;
- `series_query`;
- `season`;
- `topic_id`;
- `topic_url`;
- `tracker` / `indexer`;
- `quality`;
- release profile: audio, langs, source, HDR, subtitles/Original, если они есть;
- события `download_added`, `download_completed`, `download_soft_completed`, `plex_found`.

Плюсы:

- понятно, что именно выбирал этот пользователь;
- можно искать обновление той же темы;
- можно предлагать "как прошлый раз" без долгих вопросов.

Минусы:

- история появится только после обновления бота;
- старые загрузки без истории придётся восстанавливать через Plex-only путь.

### 2. Plex

Главный источник факта: что реально лежит в библиотеке сейчас.

Что нужно получить:

- список сериалов;
- сезоны;
- количество эпизодов, которые уже есть в Plex (`leafCount` сезона);
- `viewedLeafCount` только как справочная информация, не как критерий MVP;
- file paths по эпизодам/сезону, если доступны;
- качество, если есть в текущей модели Plex;
- title, originalTitle и year;
- `plex_rating_key`;
- `plex_guid`;
- внешние `Guid` из metadata details: `imdb://...`, `tmdb://...`, `tvdb://...`.

Плюсы:

- работает даже если история пустая;
- показывает реальную картину после ручных удалений, переездов файлов и старых скачиваний.
- Plex Series agent на живом примере отдаёт внешние идентификаторы `imdb/tmdb/tvdb`, которых достаточно для нормального identity resolver.

Минусы:

- `leafCount` у сезона означает сколько серий есть в Plex, а не сколько должно быть всего в сезоне;
- Plex не гарантирует total episodes сезона, если недостающие серии вообще не добавлены;
- по Plex-only сложно понять нужную озвучку, трекер и прошлую раздачу.
- watch-state (`viewedLeafCount`, `viewCount`, `lastViewedAt`) не должен решать, показывать сезон или нет: пользователь может захотеть докачать до просмотра.

### 3. Plex identity resolver

Нужен до detector'а кандидатов, чтобы не завязать фичу на KP id.

Поля identity:

- `plex_rating_key` - стабильная ссылка на объект внутри текущего Plex;
- `plex_guid` - обычно `plex://show/...` для Plex Series agent;
- `imdb_id` из `Guid id="imdb://..."`;
- `tmdb_id` из `Guid id="tmdb://..."`;
- `tvdb_id` из `Guid id="tvdb://..."`;
- `title`;
- `original_title`;
- `year`.

Приоритет matching:

1. `tmdb_id` / `tvdb_id` / `imdb_id`;
2. `plex_guid` + `plex_rating_key` внутри текущей библиотеки;
3. `original_title` / `title` + `year`;
4. Kinopoisk lookup как дополнительный fallback.

Почему так:

- в Plex TV Series agent основной `guid` не KP;
- внешние GUID реально доступны через `/library/metadata/<ratingKey>?includeGuids=1`;
- KP полезен для русскоязычного поиска и metadata, но не должен быть обязательным условием `/continue`.

### 4. Внешний metadata source

Нужен для понимания, сколько серий должно быть в сезоне.

Что желательно получить:

- количество сезонов;
- количество эпизодов в сезоне;
- статус сезона, если доступен;
- год/название для более точного поиска.

Приоритет:

1. TMDB/TVDB/IMDb по внешним GUID из Plex, если будет реализован соответствующий клиент;
2. Kinopoisk по title/year как fallback;
3. трекеры и найденные episode ranges как практический fallback, если metadata не дали total.

Плюсы:

- позволяет определить "в Plex 8 из 18";
- помогает не показывать завершённые сезоны как неполные.

Минусы:

- не всегда есть точные данные;
- для некоторых сериалов эпизодность может отличаться по specials, split seasons, альтернативным порядкам.

### 5. Трекеры

Нужны для проверки, появилась ли обновлённая раздача.

Приоритет:

1. Та же Rutracker topic page по `topic_id`.
2. Jackett/Rutracker поиск по `series_query + season + quality`.
3. Более мягкий поиск без части требований.

## Основные сценарии

### A. История есть, тот же topic_id обновился

Это идеальный сценарий.

Пример:

- история: `The Rookie / Season 8 / Episodes 1-8 of 18`, `topic_id=123`;
- Plex: сезон 8, 8 серий;
- Rutracker topic `123` сейчас: `Episodes 1-12 of 18`;
- бот показывает:

```text
📺 The Rookie — сезон 8

В Plex: 8 из 18 серий
В той же раздаче теперь: 1-12 из 18
Профиль: 1080p, LostFilm + Original

Можно докачать новые серии из той же темы.
```

Кнопки:

- `⬇️ Докачать и следить`, если сезон всё ещё неполный;
- `⬇️ Докачать из той же раздачи`, если обновление уже содержит полный сезон;
- `🔔 Следить за новыми сериями`;
- `📦 Скачать полный сезон, когда выйдет`;
- `⬅️ Назад`;
- `✖️ Закрыть`.

Что делает бот:

- скачивает обновлённый `.torrent` из той же темы;
- добавляет в Download Station;
- если выбран основной вариант `Докачать и следить`, создаёт подписку на ту же тему для дальнейших серий;
- сохраняет новое событие history;
- если уже есть активная задача по этому сезону, предупреждает и не создаёт дубль без подтверждения.

Формулировка "докачать" допустима.

Ограничение: если Download Station создаст новую папку из-за изменившейся структуры файлов, это уже особенность DS/раздачи. В UI можно коротко добавить: `Если структура файлов изменилась, Download Station может создать новую задачу рядом с прежней.`

### B. История есть, тот же topic_id не обновился

Пример:

- Plex: 8 серий;
- старая тема всё ещё `1-8 из 18`;
- новых серий в той же теме нет.

UI:

```text
📺 The Rookie — сезон 8

В Plex: 8 из 18 серий
В прошлой раздаче новых серий пока нет.

Можно включить слежение, чтобы PlexLoader сообщил или скачал продолжение позже.
```

Кнопки:

- `🔔 Сообщать о новых сериях`;
- `⬇️ Скачивать новые серии автоматически`;
- `📦 Скачать полный сезон, когда выйдет`;
- `🔎 Искать похожие раздачи`;
- `⬅️ Назад`;
- `✖️ Закрыть`.

Формулировка "докачать" не используется, потому что обновления нет.

### C. История есть, найден другой topic_id с продолжением

Пример:

- старая тема была `1-8`;
- та же тема не обновилась;
- поиск нашёл другую тему `1-12`.

UI:

```text
📺 The Rookie — сезон 8

В Plex: 8 из 18 серий
В прошлой раздаче новых серий нет.
Нашёл похожий вариант: 1-12 из 18, 1080p, LostFilm + Original.

Это другая раздача, поэтому лучше проверить перед скачиванием.
```

Кнопки:

- `👀 Посмотреть вариант`;
- `⬇️ Скачать обновлённую раздачу`;
- `🔔 Следить за старой темой`;
- `🔎 Искать ещё`;
- `⬅️ Назад`;
- `✖️ Закрыть`.

Формулировка "докачать" не используется.

### D. Истории нет, но Plex видит неполный сезон

Это Plex-only fallback.

Пример:

- в истории нет записи;
- Plex показывает `Show X`, сезон 2, 6 эпизодов;
- внешний metadata source или найденные релизы показывают, что в сезоне 10 эпизодов.

UI:

```text
📺 Show X — сезон 2

В Plex: 6 из 10 серий
Прошлую раздачу я не знаю, поэтому подберу варианты по Plex.

Выберите, как искать:
```

Кнопки:

- `🔎 Найти продолжение`;
- `⚙️ Уточнить качество и озвучку`;
- `🔔 Следить за новыми сериями`;
- `⬅️ Назад`;
- `✖️ Закрыть`.

Что делает бот:

- строит запрос из Plex title + season;
- предлагает выбрать качество/Original/subtitles, если профиль неизвестен;
- после выбора сохраняет результат в history, чтобы следующий раз был точнее.

Формулировка: `подобрать продолжение`, не `докачать`.

### E. Plex видит сезон, но total episodes неизвестен

Пример:

- Plex: 6 эпизодов;
- внешние metadata/Kinopoisk не дали total;
- трекеры находят варианты `1-6`, `1-8`, `1-10`.

UI:

```text
📺 Show X — сезон 2

В Plex: 6 серий
Сколько серий должно быть в сезоне, я точно не знаю.
Нашёл варианты, где серий больше.
```

Кнопки:

- `👀 Показать варианты`;
- `🔔 Следить за обновлениями`;
- `⬅️ Назад`;
- `✖️ Закрыть`.

Здесь бот не утверждает "сезон неполный", а говорит "есть варианты, где серий больше".

### F. Пользователь хочет не только докачать, но и подписаться

Если сезон ещё выходит, после любого сценария нужна мягкая развилка:

- `⬇️ Докачать и следить` - основной вариант для same-topic update, если сезон ещё неполный;
- `⬇️ Докачать сейчас` - дополнительный вариант, если пользователь не хочет подписку;
- `📦 Скачать полный сезон, когда выйдет`;
- `🔔 Только уведомлять`.

Если `topic_id` тот же, первая кнопка должна быть `Докачать и следить`.

Если `topic_id` другой, первая кнопка должна быть `Скачать обновлённую раздачу`.

## Входы в фичу

### 1. Команда или кнопка `/continue`

Основной вход:

- команда `/continue`;
- кнопка в стартовом меню: `📺 Докачать сезон` или `📺 Продолжить сериал`;
- возможно кнопка в `/status`, если у пользователя есть сериальные задачи.

Плюс: пользователь сам говорит, что хочет продолжить, бот не пытается угадать момент просмотра.

### 2. После Plex timeout / Plex found

Когда Plex нашёл неполный сезон, можно не спамить, но добавить кнопку:

- `📺 Следить за продолжением`;
- или `📺 Докачать позже`.

Для MVP лучше не добавлять везде сразу, чтобы не перегрузить финальные экраны.

### 3. Из `/subs`

Если пользователь уже подписан или раньше отказался от подписки:

- рядом с неполной подпиской можно показать `📺 Докачать сейчас`;
- но это вторичный вход.

### 4. Из bulk-плана

Если bulk-план видит partial season:

- использовать те же компоненты выбора;
- не дублировать отдельную логику.

## Экран списка неполных сезонов

Пример:

```text
📺 Продолжить сериал

Нашёл сезоны, где в Plex есть не все серии или появились варианты с большим количеством серий.

Режим: 🙋 Моё

1. The Rookie — сезон 8
   Plex: 8 из 18
   Прошлый выбор: 1080p, LostFilm + Original

2. Drive — сезон 1
   Plex: 3 серии
   Есть варианты, где серий больше

3. Show X — сезон 2
   Plex: 6 из 10
   Прошлая раздача неизвестна
```

Кнопки:

- `📺 1`;
- `📺 2`;
- `📺 3`;
- `🙋 Моё` / `🌐 Всё`;
- `⬅️`;
- `➡️`;
- `🔄 Обновить`;
- `✖️ Закрыть`.

Правила:

- `Моё` показывает кандидаты из истории текущего пользователя;
- `Всё` показывает семейную медиатеку: Plex-only кандидаты и сезоны из истории других пользователей;
- обычный пользователь может докачать кандидата из `Всё`, потому что медиатека общая;
- если кандидат из чужой истории, не делать из этого приватный акцент; достаточно показать профиль прошлой загрузки, если он полезен;
- максимум 10 кандидатов на экран;
- если кандидатов больше 10, показывать навигацию `⬅️` / `➡️`;
- если в режиме `Моё` пусто, но есть Plex-only кандидаты, показать текст: `В ваших прошлых загрузках ничего не нашёл, но в Plex есть неполные сезоны. Переключить на "Всё"?`.

## Как определять кандидатов

### Кандидат из истории

Условия:

- для режима `Моё`: `chat_id` совпадает;
- для режима `Всё`: подходит история любого разрешённого пользователя;
- `kind=series`;
- есть `series_query` и `season`;
- была успешная загрузка или Plex found/soft-complete;
- Plex сейчас видит этот сериал/сезон.

Дальше:

- считаем episodes_in_plex;
- пытаемся определить total episodes;
- проверяем текущий topic по `topic_id`, если он есть;
- если в topic стало больше эпизодов, кандидат получает высокий score.

### Кандидат из Plex-only

Условия:

- Plex видит сезон;
- total episodes известен и больше episodes_in_plex;
- или total неизвестен, но трекеры находят релизы с большим диапазоном серий.

Ограничение:

- Plex-only кандидаты показываются в режиме `Всё`;
- в режиме `Моё` они не смешиваются с личной историей, чтобы пользователь понимал разницу;
- для семейной медиатеки любой разрешённый пользователь может выбрать Plex-only кандидата и докачать/подобрать продолжение.

### Кандидат из активных задач

Если Download Station уже качает этот сезон:

- не предлагать докачку;
- показать `Уже есть активная загрузка`;
- дать кнопку `📋 Показать задачу`.

## Completeness resolver

Это центральная часть фичи. Он отвечает не на вопрос "досмотрел ли пользователь", а на вопрос "есть ли разумный признак, что сезон в Plex неполный или его можно продолжить".

Входные данные:

- Plex show identity: `plex_rating_key`, `plex_guid`, `imdb_id`, `tmdb_id`, `tvdb_id`, title/originalTitle/year;
- Plex season summary: season number, `leafCount`, episode indexes, quality, file paths;
- history summary: прошлый `topic_id`, episode range, quality, tracker/indexer, release profile;
- active Download Station tasks;
- external metadata total episodes, если уже известен;
- tracker/topic enrichment, если кандидат открыт пользователем.

Выходные данные:

- `present_count` - сколько эпизодов есть в Plex;
- `present_episode_numbers` - какие номера эпизодов есть в Plex;
- `known_total` - сколько эпизодов должно быть в сезоне, если известно;
- `missing_numbers` - конкретные пропущенные номера, если можно определить;
- `quality` - качество сезона в Plex;
- `confidence` - тип уверенности;
- `reason_for_user` - короткая причина, почему сезон показан.

Уровни уверенности:

| `confidence` | Когда ставим | Пример текста |
|---|---|---|
| `exact_total` | `known_total > present_count` | `В Plex 8 из 18 серий` |
| `gap` | внутри Plex есть пропущенные номера эпизодов | `В Plex нет серии 3` |
| `topic_update` | та же тема `topic_id` содержит больше серий, чем Plex | `В той же раздаче появились новые серии` |
| `history_partial` | история знает, что пользователь качал partial-раздачу | `Раньше был скачан неполный сезон` |
| `tracker_more` | похожая раздача содержит больший диапазон серий, чем Plex | `Нашёл вариант, где серий больше` |
| `unknown` | признаков неполноты нет | не показывать в основном списке |

Критерий показа в `/continue`:

- показываем сезон, если `confidence != unknown`;
- не показываем сезон, если Plex выглядит полным и нет history/topic/tracker-сигнала;
- просмотренность (`viewedLeafCount`, `viewCount`, `lastViewedAt`) не участвует в `confidence`.

Правило для первого списка:

- быстрый список может показывать только `exact_total`, `gap`, `history_partial` и уже известные `topic_update`;
- `tracker_more` появляется после ленивого enrichment конкретного сезона или после ручного `🔄 Обновить`;
- `unknown` можно показывать только в отдельном debug/admin режиме, но не в пользовательском MVP.

Почему это важно:

- пользователь видит актуальный список неполных сезонов, а не список досмотренного;
- сезон, который ещё не смотрели, всё равно попадёт в список, если он реально неполный;
- полный сезон вроде `Белгравия S01` не должен попадать в основной список только потому, что он есть в Plex.

## Scoring кандидатов

Высший приоритет:

1. Та же тема `topic_id`, больше эпизодов, тот же tracker.
2. Та же тема `topic_id`, сезон стал полным.
3. Другой topic, но совпадает качество + озвучка/Original/subtitles.
4. Другой topic, совпадает качество, но профиль озвучки неполный/неясный.
5. Отдельная подсказка `Есть версия лучше`, если найдено качество выше текущего.
6. Plex-only мягкий кандидат, где просто серий больше.

Сигналы для score:

- `topic_id` exact match;
- `series_query` exact/normalized match;
- `season` exact;
- episode range больше текущего Plex count;
- full season лучше partial, если пользователь выбрал "ждать полный";
- quality совпадает;
- release audio/langs совпадают;
- tracker/indexer совпадает;
- seeders выше нуля.

Правило качества:

- основной кандидат для "докачать" должен быть в том же качестве, что уже есть в Plex или было выбрано раньше;
- лучшее качество не подменяет основной вариант, а показывается отдельно;
- если найдено только лучшее качество и нет same-quality продолжения, бот озвучивает это явно: `В том же качестве не нашёл, есть вариант лучше: 2160p`.

```text
Добавил обновлённую раздачу.

Сезон всё ещё неполный: 1-12 из 18.

```

Кнопки:

- `⬇️ Автоскачивать новые серии`;
- `📦 Скачать полный сезон, когда выйдет`;
- `Не нужно`;

Если сезон стал полным, подписку не предлагаем.

## State и данные

Новый отдельный state на MVP может не понадобиться.

Используем:

- `download_history.jsonl` - прошлые выборы и события;
- `task_meta.json` - meta активных/известных задач;
- `task_owners.json` - владелец task id;
- `topic_subscriptions.json` - если пользователь решит следить дальше;
- `series_bulk_jobs.json` - только если действие пришло из bulk-флоу.

Если понадобится сохранять промежуточный список кандидатов после кнопки `/continue`, можно добавить:

- in-memory `CONTINUE_SERIES_CONTEXT`;
- позже `series_continue_jobs.json`, если нужно восстановление после рестарта.

Для MVP лучше начать с in-memory context: это обычный интерактивный подбор, потеря контекста после рестарта не критична.

## Первый старт после внедрения

Проблема: на момент появления фичи `download_history.jsonl` не знает старые скачивания. Если опираться только на историю, пользователь увидит пустой экран, хотя в Plex уже есть неполные сезоны.

Решение для первого запуска:

1. `/continue` сначала делает быстрый Plex scan или берёт свежий Plex-кэш:
   - список шоу;
   - сезоны;
   - `leafCount`;
   - `ratingKey`;
   - title/originalTitle/year;
   - внешние `Guid` только для кандидатов, которые реально попадут в список или будут открыты пользователем.
2. Первый экран не ждёт полной проверки всех трекеров и всех metadata providers.
3. Бот строит быстрый Plex-only список кандидатов:
   - сезоны из Plex, которые могут требовать продолжения;
   - сезоны с историей текущего или другого пользователя;
   - сезоны, для которых уже есть явный сигнал из topic/history.
4. Точное обогащение делается лениво:
   - пользователь открыл конкретный сезон;
   - тогда бот проверяет external GUID, total episodes, ту же тему `topic_id`, похожие раздачи и качество.
5. История используется как усилитель:
   - если для сезона есть запись текущего пользователя, кандидат попадает в `Моё`;
   - если история есть у другого пользователя, кандидат виден в `Всё`;
   - если истории нет вообще, кандидат всё равно виден в `Всё` как Plex-only.
6. Для Plex-only кандидата бот не утверждает, что знает прошлую раздачу:
   - пишет `Прошлую раздачу я не знаю`;
   - предлагает `Найти продолжение`;
   - при необходимости спрашивает качество/озвучку.
7. После первой успешной докачки бот сохраняет событие в `download_history.jsonl`; следующие заходы уже становятся точнее.

Что не делаем на первом старте:

- не строим полный архив всех сериалов со всеми внешними metadata;
- не ходим по всем темам трекеров заранее;
- не блокируем UI ожиданием долгого enrichment;
- не используем `viewedLeafCount` как фильтр "пользователь досмотрел".

Стартовый пустой экран:

```text
📺 Докачать сезон

В ваших прошлых загрузках неполных сезонов не нашёл.
Но в Plex есть сезоны, которые могут быть неполными.
Показать всю семейную медиатеку?
```

Кнопки:

- `🌐 Показать всё`;
- `🔄 Обновить Plex`;
- `✖️ Закрыть`.

## Callback namespace

Предложение:

- `cont:list` - список кандидатов;
- `cont:open:<idx>` - открыть кандидат;

## Реализация по коммитам

### Коммит 1. Plex identity resolver и модель кандидата

Задача:

- создать чистый модуль, например `series_continue.py`;
- расширить Plex-модель/клиент для чтения внешних `Guid` у шоу через `includeGuids=1`;
- собрать `PlexSeriesIdentity`: `plex_rating_key`, `plex_guid`, `imdb_id`, `tmdb_id`, `tvdb_id`, `title`, `original_title`, `year`;
- описать dataclass кандидата;
- без UI и скачивания.

Проверки:

- Plex Series agent с `plex://show/...` и `Guid imdb/tmdb/tvdb`;
- Plex show без внешних `Guid` fallback'ается на title/year;
- KP id не обязателен для identity;
- `leafCount` трактуется как "эпизоды есть в Plex", а не total сезона.

### Коммит 2. Быстрый Plex/history detector без watch-state

Задача:

- собрать кандидатов из history + Plex;
- использовать `leafCount` как количество доступных в Plex серий;
- не использовать `viewedLeafCount` / `viewCount` как фильтр кандидатов;
- не запускать полную проверку всех external metadata/tracker topics на первом списке.

Проверки:

- история есть, Plex видит меньше серий, topic id есть;
- история есть, Plex сезон полный - кандидат не показывается;
- истории нет, Plex-only total известен - fallback-кандидат в режиме `Всё`;
- режим `Моё` показывает только историю текущего `chat_id`;
- режим `Всё` показывает семейные Plex-only кандидаты и кандидаты из истории других пользователей.
- полностью непросмотренный сезон всё равно может быть кандидатом;
- полностью просмотренный сезон не получает особый приоритет в MVP.

### Коммит 3. Completeness resolver

Задача:

- создать pure helper, который определяет `confidence` и `reason_for_user`;
- использовать Plex episode indexes для `gap`;
- использовать external metadata total для `exact_total`;
- использовать history partial как отдельный сигнал;
- не учитывать watch-state.

Проверки:

- `present=[1,2,4]` -> `gap`, missing `[3]`;
- `known_total=10`, `present_count=6` -> `exact_total`;
- `known_total=6`, `present_count=6`, нет history/topic -> `unknown`;
- history partial без total -> `history_partial`;
- `viewedLeafCount` не меняет результат.

### Коммит 4. Проверка той же темы и lazy enrichment

Задача:

- по `topic_id` обновить данные темы;
- распарсить episode range;
- определить, стало ли серий больше;
- выставить action type: `same_topic_update`, `no_update`, `unknown`.
- при открытии конкретного кандидата лениво догрузить external GUID/metadata/трекеры;
- не делать полную фоновую сборку всего Plex при первом входе.

Проверки:

- `1-8 из 18` -> `1-12 из 18`;
- `1-8 из 18` -> `1-8 из 18`;
- topic unavailable;
- парсинг без total.

### Коммит 5. UI `/continue`

Задача:

- команда `/continue`;
- список кандидатов с режимами `Моё` / `Всё`;
- пагинация по 10 кандидатов кнопками `⬅️` / `➡️`;
- экран конкретного кандидата;
- кнопки `Докачать и следить`, `Скачать обновлённую`, `Следить`, `Закрыть`.

Проверки:

### Коммит 6. Действие "докачать из той же темы"

## MVP-границы

В MVP входит:

- ручной вход `/continue` / `📺 Докачать сезон` / `📺 Продолжить сериал`;
- режимы `Моё` / `Всё` для семейной медиатеки;
- кандидаты из history + Plex-only fallback для первого старта;
- same-topic update как главный сценарий;
- same-quality как основной выбор;
- отдельная подсказка, если найдено качество лучше;
- создание подписки, если новых серий пока нет или сезон ещё неполный.

В MVP не входит:

- автоматическое определение, что пользователь досмотрел сезон;
- пуш "вам пора докачать";
- анализ Plex watch history;
- фильтрация кандидатов по `viewedLeafCount`, `viewCount` или `lastViewedAt`;
- полная фоновая сборка архива всех Plex-сериалов со всеми внешними metadata на первом старте;
- сложный preference engine;
- автоматическое удаление старых partial-файлов;
- гарантия, что Download Station не создаст дубль папки при изменившейся структуре раздачи.

## Решённые вопросы

1. Plex-only кандидаты показываем обычному пользователю в режиме `Всё`, потому что медиатека семейная.
2. Делаем отдельную команду `/continue`.
3. Для same-topic update с неполным сезоном основной вариант - `Докачать и следить`.
4. Основной автокандидат должен совпадать по качеству; лучшее качество показываем отдельной подсказкой.
5. Лимит первого экрана - 10 кандидатов, дальше `⬅️` / `➡️`.
6. Первый старт строится через Plex-only fallback, потому что истории старых загрузок ещё может не быть.
7. Plex Series agent отдаёт внешние `Guid` (`imdb/tmdb/tvdb`), поэтому identity строится на них, а не на KP id.
8. Watch-state не участвует в MVP-критериях: список должен быть актуальным по неполноте/возможности продолжения, а не по факту просмотра.
9. Enrichment ленивый: первый список быстрый, точные total/topic/tracker проверки делаются при открытии кандидата.

## Открытые вопросы перед кодом

1. Точное название кнопки в стартовом меню: `📺 Докачать сезон` или `📺 Продолжить сериал`.
2. В режиме `Всё` показывать подпись владельца прошлого выбора (`прошлый выбор: user`) или не акцентировать владельца и показывать только профиль.
3. Какой возраст Plex-кэша считать свежим для `/continue`: использовать текущий кэш или всегда запускать refresh перед первым списком.
