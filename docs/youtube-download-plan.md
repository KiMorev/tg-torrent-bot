# План YouTube-download для PlexLoader

Документ фиксирует договоренности перед реализацией фичи: пользователь отправляет
YouTube-ссылку в Telegram-чат, бот ставит скачивание в очередь, сохраняет ролик
на NAS и в перспективе доводит его до Plex.

## Цель

Сделать опциональный модуль "YouTube inbox для NAS/Plex":

1. Пользователь отправляет одиночную YouTube-ссылку.
2. Бот показывает preview: название, канал, длительность, доступные качества.
3. Пользователь выбирает качество, максимум 1080p.
4. Бот ставит задачу в очередь.
5. Worker скачивает ролик в папку `youtube` на NAS.
6. Бот уведомляет об успехе или понятной ошибке.
7. Целевое состояние: после скачивания ролик появляется в отдельной
   YouTube-секции Plex; бот сообщает пользователю, что файл скачан, а при
   включенном Plex polling - что Plex уже увидел ролик.

Фича должна быть выключена по умолчанию и доступна только разрешенным чатам.
Главный рубильник: `YOUTUBE_DOWNLOADS_ENABLED=false` полностью выключает
YouTube-блок без удаления кода.

## Зафиксированные решения

- Если уже идет скачивание, новая задача ставится в очередь.
- Стартовая защита NAS от перегруза: 1 активная YouTube-загрузка. Жесткие
  лимиты очереди по умолчанию не включаем, но предусматриваем env-настройки для
  общего лимита и лимита на чат.
- Максимальное качество: 1080p.
- Transcode не используем вообще: бот скачивает только доступные готовые
  качества/форматы, которые можно собрать без перекодирования.
- Папка для файлов: `youtube`.
- Безопасный NAS-путь для MVP: отдельная shared folder `/volume1/youtube`.
- Практический путь в контейнере: отдельный bind mount, например
  `/youtube_storage`.
- Точный mount при включении фичи:
  `${YOUTUBE_NAS_PATH:-/volume1/youtube}:/youtube_storage:rw`.
- Подпапка `/volume1/video/youtube` допустима только если существующие Plex
  библиотеки не сканируют корень `/volume1/video` рекурсивно.
- `DS_DESTINATION=video` остается настройкой Download Station для торрентов.
  YouTube-download в MVP пишет файлы напрямую в Docker bind mount, а не добавляет
  URL в Download Station.
- Лимит длительности: 5 часов (`YOUTUBE_MAX_DURATION_MINUTES=300`).
- Жесткий лимит размера файла в MVP не вводим; перед стартом проверяем
  свободное место, чтобы не заполнить диск в ноль.
- Plex-интеграцию закладываем сразу как целевую архитектуру, а не как случайное
  последующее расширение.
- Основной расчет для Plex: отдельная библиотека `YouTube` плюс принудительный
  refresh этой секции после скачивания. Автоскан Plex можно оставить включенным,
  но на bind mount он не всегда срабатывает сразу.
- Библиотеку Plex называем `YouTube`.
- Политика совместимости: сначала предпочитаем MP4/H.264/AAC без transcode; если
  подходящего формата в выбранном качестве нет, предлагаем более низкое качество
  или понятный отказ.
- Telegram upload не является целью MVP. Основной результат - файл на NAS/Plex.
- Cookies, обход авторизации, private/members-only/age-restricted и DRM-сценарии
  не поддерживаются.
- Пользователь должен видеть статус длинной загрузки: место в очереди,
  старт скачивания, прогресс/скорость/ETA, финальный результат.
- Progress берем через Python API `yt-dlp` и `progress_hooks`, а не парсим
  обычный stdout.

## Главный рубильник

Главный эксплуатационный выключатель:

```env
YOUTUBE_DOWNLOADS_ENABLED=false
```

Поведение при выключенном рубильнике:

- бот не вызывает `yt-dlp`;
- бот не запускает YouTube worker;
- бот не создает и не продолжает YouTube-очередь;
- бот не делает Plex refresh/polling для YouTube;
- бот не требует наличия `/youtube_storage` для обычной работы;
- обычный torrent-поиск, magnet, Kinopoisk, `/new`, Download Station и Plex для
  фильмов/сериалов работают как раньше.

Если пользователь прислал YouTube-ссылку при выключенной фиче, бот должен мягко
ответить, что YouTube-download сейчас отключен, и дать кнопки "Открыть YouTube"
и "Закрыть". Не нужно превращать такую ссылку в обычный torrent-поиск.

Дополнительные частичные выключатели могут появиться позже, но главный рубильник
должен иметь приоритет над ними:

```env
YOUTUBE_PLEX_REFRESH_AFTER_DOWNLOAD=false
YOUTUBE_PLEX_POLL_AFTER_DOWNLOAD=false
```

Это позволяет оставить код в проекте и выключить весь блок на NAS до лучших
времен, если он начнет мешать стабильности.

## Почему нужен лимит по длительности

Лимит длительности нужен не из-за самого `yt-dlp`, а из-за эксплуатации на DS223:

- длинный ролик может занимать много места;
- при скачивании video+audio и последующей сборке временно может потребоваться
  примерно до 1.5-2 размеров итогового файла;
- `ffmpeg` в режиме remux/copy обычно легкий, но все равно создает дисковую и
  CPU-нагрузку;
- без лимита пользователь может случайно отправить стрим или архив на десятки
  часов;
- очередь из длинных роликов будет блокировать все последующие задачи.

Зафиксированный стартовый лимит: `YOUTUBE_MAX_DURATION_MINUTES=300`.

5 часов достаточно для длинных интервью, лекций и подкастов, но все еще защищает
от случайного добавления многодневных трансляций и больших архивов.

Отдельный hard cap по размеру файла не вводим на старте. Вместо этого бот должен
проверять свободное место перед скачиванием и учитывать временный запас на
сборку файла.

## Что значит "скачиваем только то, что можно"

Это не означает: "если у пользователя есть YouTube Premium, значит бот может
скачать все то же самое".

Для MVP правило такое:

- бот работает только с публично доступными роликами по обычной ссылке;
- бот не использует cookies пользователя;
- бот не входит в YouTube-аккаунт;
- бот не обходит Premium, age gate, private, members-only, региональные или DRM
  ограничения;
- пользователь сам отвечает за то, что у него есть право сохранять конкретный
  ролик.

YouTube Premium offline downloads обычно разрешены внутри официальных приложений
YouTube, а не как получение обычного MP4-файла внешним ботом. Поэтому Premium не
считаем основанием добавлять cookies или account-based download.

Технически `yt-dlp` умеет работать с cookies. Это могло бы открыть доступ к
части роликов, доступных конкретному аккаунту. Для PlexLoader это сознательно
не MVP:

- cookies являются чувствительным секретом аккаунта;
- их надо безопасно хранить, обновлять и отзывать;
- это повышает риск блокировок/челленджей аккаунта;
- это смешивает личную авторизацию пользователя с серверным ботом;
- это хуже с точки зрения ToS и поддержки.

Если когда-нибудь возвращаться к этому вопросу, это должен быть отдельный
admin-only режим с явным предупреждением, отдельным secret-файлом, отключением
по умолчанию и без обещаний стабильности.

## Технический подход

Базовая реализация не добавляет отдельные контейнеры для `yt-dlp` и `ffmpeg`.
Они становятся зависимостями основного Docker image бота:

- `yt-dlp` - Python/CLI-зависимость;
- `ffmpeg`/`ffprobe` - системные бинарники в Docker image.

Плюсы embedded-варианта:

- меньше инфраструктуры;
- Telegram-бот сам управляет preview, очередью, статусами и уведомлениями;
- проще связать результат с Plex и текущими правами доступа.

Минусы:

- Docker image станет тяжелее;
- для обновления `yt-dlp` нужен rebuild image;
- скачивание и remux идут внутри контейнера бота, поэтому нужны строгие лимиты.

## Переиспользование текущей логики PlexLoader

Главный UX-принцип: YouTube-блок не должен выглядеть как отдельный второй бот.
Все, что можно безопасно взять из текущего флоу фильмов/сериалов, нужно
переиспользовать.

Что берем напрямую или через маленький adapter:

- Access control: `_is_allowed()`, `_reply_access_pending()`.
- Роутинг plain text: добавляем YouTube URL в `text_message_entry` до обычного
  torrent-поиска, как сейчас обрабатываются magnet/Kinopoisk.
- Безопасное редактирование сообщений: `_safe_edit_message()` и
  `_safe_edit_callback()`.
- Форматирование размера/прогресса/ETA: `_format_size()`, `_format_progress()`,
  `_format_eta()`, `_progress_bar()`, `_progress_meter()`, `_progress_percent()`
  из `formatters.py`.
- Карточка статуса: сделать adapter `YouTubeJob -> task-like dict` и по
  возможности использовать `task_views.format_task_card()` для одинаковых строк
  `Статус`, `Прогресс`, `Скачано`, `Скорость`, `Осталось`.
- Кнопка закрытия: использовать общий `task:close:` и существующее поведение
  удаления сообщения + авто-уведомление `Закрыто`.
- Ошибки: использовать стиль существующих `_search_error_keyboard()` /
  `_download_error_keyboard()` - понятный текст, retry/queue где уместно,
  `Закрыть` всегда есть.
- Disk guard: переиспользовать текущую политику `_check_disk_space_for_download()`
  и пороги 5%/15%, дополнив ее проверкой write-access к `YOUTUBE_DOWNLOAD_DIR`.
- История: писать события в `download_history.jsonl` через существующий стиль
  `_record_download_history()`, но с event/source для YouTube.
- Классификация Telegram delivery errors: использовать `_classify_send_error()`,
  чтобы permanent/transient ошибки доставки работали одинаково.
- Plex final-push pattern: переиспользовать подход `_plex_poll_after_finish()` -
  новая push-нотификация при появлении в Plex, отдельная hint-message
  `Ищем файл в библиотеке Plex`, удаление hint после found/timeout,
  `▶️ Смотреть в Plex` + `Закрыть`.

Что не копируем буквально:

- Download Station task id, `task_owners.json`, `TASK_CARD_MESSAGES`,
  `_run_task_notifications_once()` и `_start_task_card_refresh()` завязаны на
  задачи Download Station. Для YouTube нужен собственный `youtube_job_id` и
  свой worker/status loop.
- Pending-downloads очередь для торрентов хранит tracker/torrent/magnet
  payload. Для YouTube нужна отдельная очередь, но с похожими статусами и
  retry/cleanup-паттерном.
- Plex movie/series lookup из `_plex_poll_lookup_target()` ищет фильмы/сезоны.
  Для YouTube нужен lookup в YouTube-секции по file path/video_id.

Целевой пользовательский флоу должен повторять текущий:

1. Пользователь подтверждает скачивание.
2. Бот показывает карточку статуса с тем же форматом прогресса.
3. После завершения бот отправляет новое push-сообщение `✅ Видео скачано` и
   убирает/закрывает устаревшую live-карточку, как обычные task cards после
   финальной нотификации.
4. Если включен Plex polling, бот отправляет отдельное сообщение
   `🔄 Ищем файл в библиотеке Plex — сообщим, как только появится.`
5. Когда Plex нашел ролик, бот отправляет новое push-сообщение
   `✅ <title> добавлен в Plex` с кнопкой `▶️ Смотреть в Plex`.
6. Если Plex не нашел ролик за лимит, скачивание считается успешным, а
   пользователь получает предупреждение в стиле текущего Plex timeout.

### История загрузок

Да, YouTube-события должны попадать в существующий `download_history.jsonl`.
Сейчас это внутренняя append-only память, а не отдельный пользовательский экран,
но именно туда уже пишутся добавление, завершение, ошибки и Plex-result для
фильмов/сериалов.

YouTube не должен маскироваться под Download Station task:

- используем `youtube_job_id`, а не `task_id`;
- сохраняем `source="youtube"`;
- сохраняем безопасные поля: `title`, `channel`, `duration_seconds`,
  `quality`, `format_id`, `youtube_video_id`, canonical watch URL, итоговый
  `file_path`, `file_size`;
- не сохраняем cookies, headers, signed media URLs, raw API/debug output и
  временные direct download URLs.

События:

- `youtube_download_queued` - пользователь подтвердил скачивание, задача в
  очереди;
- `youtube_download_started` - worker начал скачивание;
- `youtube_download_completed` - файл успешно сохранен на NAS;
- `youtube_download_failed` - metadata/download/remux завершились ошибкой;
- `youtube_plex_found` - YouTube-ролик появился в Plex-библиотеке;
- `youtube_plex_not_found` - файл скачан, но Plex не подтвердил появление за
  окно ожидания.

Если в будущем появится пользовательский экран истории, YouTube-записи должны
отображаться там рядом с фильмами/сериалами как отдельный source/type, а не в
отдельной изолированной истории.

## Обновление `yt-dlp` и `ffmpeg`

`yt-dlp` и `ffmpeg` в этом плане не отдельные контейнеры.

### Как узнаем о новой версии

Рекомендуемый механизм для MVP - GitHub-side уведомление, а не проверка из
работающего NAS-контейнера.

Вариант A, предпочтительный:

- добавить Dependabot config для `pip`;
- держать `yt-dlp` pinned в `requirements.txt`, например `yt-dlp==YYYY.MM.DD`;
- Dependabot сам открывает PR, когда выходит новая версия;
- GitHub присылает уведомление о PR;
- CI прогоняет тесты;
- после merge обычный build-and-push публикует новый image.

В MVP это реализовано через `.github/dependabot.yml`: проверяется только
зависимость `yt-dlp`, чтобы не смешивать YouTube emergency bumps с обычными
обновлениями Python-пакетов.

Вариант B, если Dependabot окажется неудобен:

- добавить scheduled GitHub Actions workflow, например раз в день;
- workflow сравнивает pinned-версию в `requirements.txt` с последней версией на
  PyPI/GitHub releases;
- если версия новее - создает GitHub issue или PR;
- опционально можно добавить Telegram-уведомление из Actions, но это потребует
  отдельные GitHub Secrets для bot token/chat id.

Не рекомендуемый MVP-путь:

- проверять новую версию самим ботом на NAS и слать админам push при каждом
  старте. Это добавляет внешний сетевой dependency в runtime, риск шумных
  уведомлений и не решает сам update без rebuild image.

Рекомендуемый путь обновления:

1. Держать `yt-dlp` как pinned-зависимость в `requirements.txt`.
2. Настроить автоматическое отслеживание новых версий через Dependabot/Renovate
   или отдельный GitHub Actions workflow, который открывает PR с bump версии.
3. В PR прогонять тесты и, если возможно, smoke-check metadata без скачивания.
4. После merge GitHub Actions собирает новый Docker image в GHCR.
5. Watchtower на NAS подтягивает новый digest и пересоздает контейнер.

Так сохраняется текущая модель деплоя проекта: commit/push -> GitHub Actions ->
GHCR -> Watchtower.

Для срочных поломок YouTube допускается ручной emergency bump `yt-dlp` отдельным
коммитом. Сначала пробуем последнюю stable-версию; если YouTube уже сломался, а
stable еще не содержит фикса, можно отдельным осознанным коммитом временно
перейти на pre-release/nightly. Он все равно должен идти через rebuild image, а
не через обновление внутри живого контейнера.

Если пользователь сам узнал о новой версии, ручной порядок действий такой:

1. Открыть release/changelog `yt-dlp` и убедиться, что версия нужна.
2. Обновить строку `yt-dlp==...` в `requirements.txt`.
3. Запустить тесты.
4. Сделать commit/push в `main`.
5. Дождаться GitHub Actions: tests -> build-and-push.
6. Watchtower на NAS сам подтянет новый digest и пересоздаст контейнер.

Ручной `docker compose up -d --build` нужен только для локальной проверки из
checkout. Для продового NAS после обычного push он не нужен.

Не рекомендуемый путь для MVP:

- запускать `yt-dlp -U` внутри работающего контейнера;
- обновлять зависимости при каждом старте контейнера;
- ставить unpinned/latest-зависимости без проверки тестами.

Причина: это ухудшает воспроизводимость. Лучше пусть новая версия приходит через
обычный CI/deploy-путь.

`ffmpeg` обновляется вместе с base image/Alpine packages при rebuild Docker image.
Если понадобится чаще обновлять именно `ffmpeg`, это отдельное инфраструктурное
решение.

## Политика форматов без transcode

Не путать три операции:

- download - скачать video/audio потоки;
- remux - без перекодирования переложить потоки в другой контейнер, например
  MP4; это относительно легкая операция;
- transcode - перекодировать видео/аудио в другой codec; это тяжелая CPU-задача.

Для DS223 в MVP разрешены только download + remux/copy. Transcode не запускаем
ни для 1080p, ни для 720p, ни как fallback.

Итоговое правило MVP:

1. Пользователь выбрал 1080p.
2. Бот ищет 1080p-формат, который можно скачать и собрать без transcode.
3. Если такой 1080p есть - скачиваем его.
4. Если такого 1080p нет - бот не транскодирует, а предлагает 720p/480p или
   показывает отказ для этого качества.

То есть 1080p - это верхний допустимый предел качества, а не обещание любой
ценой получить 1080p через перекодирование.

Работаем только с тем качеством, которое реально доступно у YouTube/`yt-dlp` в
подходящем формате. Стартовая политика совместимости:

1. Сначала предпочитать MP4/H.264/AAC без transcode.
2. Если выбранное качество недоступно в таком варианте, предложить более низкое
   качество.
3. Если совместимого варианта нет, показать понятный отказ.

Это осознанный выбор в пользу предсказуемого Plex-воспроизведения на первом
этапе, а не максимального качества любой ценой.

Альтернатива на будущее: вынести YouTube-download в отдельный сервис/контейнер
вроде MeTube/Pinchflat и дать Watchtower обновлять его image. Но это уже другой
продуктовый контур: больше возможностей, больше памяти, больше интеграционной
сложности.

## Модель папок NAS/Plex

Текущее состояние проекта:

- Download Station получает destination `video` через `DS_DESTINATION=video`;
- контейнер бота монтирует `/volume1/video` как `/storage`;
- Plex уже видит фильмы и сериалы из своих существующих библиотек.

Риск:

Если текущие Plex-библиотеки "Фильмы" или "Телепередачи" добавлены на корень
`/volume1/video`, Plex сканирует вложенные папки. В этом случае подпапка
`/volume1/video/youtube` может попасть в существующую библиотеку фильмов или
сериалов. Это нежелательно.

Рекомендуемый безопасный MVP-вариант:

```text
NAS:       /volume1/youtube
Container: /youtube_storage
Plex:      отдельная библиотека "YouTube" типа "Другие видео" -> /volume1/youtube
```

Что нужно сделать:

- создать shared folder `youtube` в DSM;
- дать контейнеру бота write-доступ к этой папке;
- дать Plex read-доступ к этой папке;
- добавить отдельный bind mount в `compose.yaml`;
- выставить `YOUTUBE_DOWNLOAD_DIR=/youtube_storage`;
- в Plex добавить новую библиотеку типа "Другие видео", назвать ее `YouTube` и
  указать `/volume1/youtube`.

Точный фрагмент `compose.yaml` при включении фичи:

```yaml
volumes:
  - tg_torrent_drop_state:/data
  - /volume1/video:/storage:rw
  - ${YOUTUBE_NAS_PATH:-/volume1/youtube}:/youtube_storage:rw
```

Новые env:

```env
YOUTUBE_NAS_PATH=/volume1/youtube
YOUTUBE_DOWNLOAD_DIR=/youtube_storage
```

Если `YOUTUBE_DOWNLOADS_ENABLED=false`, код YouTube-блока не использует этот
путь. Но сам bind mount в Docker требует, чтобы `/volume1/youtube` уже был
создан и доступен на NAS. Поэтому mount добавляем только когда shared folder
создана и фичу реально включаем.

Что такое `YOUTUBE_PLEX_SECTION`:

- в интерфейсе Plex пользователь видит название библиотеки, например `YouTube`;
- внутри Plex API эта библиотека имеет section key/id;
- для refresh/polling боту нужен именно section key/id;
- если `YOUTUBE_PLEX_SECTION` пустой, бот должен попытаться найти библиотеку по
  названию `YOUTUBE_PLEX_LIBRARY_NAME=YouTube`;
- если auto-detect не сработал, section id можно будет указать вручную в env.

Рекомендованные настройки библиотеки Plex `YouTube`:

- Тип библиотеки: `Другие видео`.
- Сканер: `Plex Video Files`.
- Агент: `Plex Personal Media`.
- Видимость: можно оставить `Добавить на домашний экран и в глобальный поиск`,
  если YouTube-ролики должны быть видны в общем поиске Plex.
- `Включить трейлеры фильмов`: выключить.
- `Использовать оригинальные названия`: оставить выключенным на старте.
- `Предпочитать иллюстрации на основе языка библиотеки`: не критично, можно
  оставить включенным.
- `Использовать локальные данные`: включить.
- `Allow matching to explicit content`: выключить.
- `Включить создание миниатюр для видеофайлов`: выключить для DS223 на старте.
- `Коллекции`: оставить как есть или выбрать любой удобный режим отображения.
- `Включить обнаружение титров`: выключить для DS223 на старте.
- `Обнаружение рекламы`: выключить/не использовать для YouTube-библиотеки, если
  доступен такой вариант.
- `Enable voice activity detection`: выключить для DS223 на старте.

Логика: эта библиотека должна быть простым личным каталогом файлов, а не
медиа-агентом, который пытается матчить фильмы, искать трейлеры, генерировать
превью, титры и дополнительные анализы.

Альтернатива без новой share:

- использовать `/volume1/video/youtube`;
- убедиться, что существующие библиотеки Plex смотрят не на `/volume1/video`, а
  на более узкие папки, например `/volume1/video/Movies` и
  `/volume1/video/TV`;
- либо добавить `.plexignore` в корень `/volume1/video`, чтобы существующие
  библиотеки игнорировали `youtube/*`.

Для MVP выбираем отдельную shared folder `/volume1/youtube`, если текущие Plex
библиотеки сканируют корень `/volume1/video`.

YOUTUBE_DOWNLOADS_ENABLED=false

## Этап 3. Очередь и устойчивость

Изменения:

- добавить state-файл `youtube_downloads.json`;
- статусы MVP: `queued`, `fetching_metadata`, `downloading`, `processing`,
  `completed`, `failed`;
- один worker;
- активная загрузка одна: `YOUTUBE_MAX_PARALLEL=1`;
- лимиты очереди предусмотрены, но по умолчанию выключены:
  `YOUTUBE_MAX_QUEUE_SIZE=0`, `YOUTUBE_MAX_QUEUE_PER_CHAT=0`, где `0` означает
  "без жесткого лимита";
- очередь переживает рестарт контейнера;
- running-задача после рестарта возвращается в `queued` с пометкой в `error`,
  чтобы worker продолжил её штатно;
- queued-cancel, защита от дублей одной ссылки и очистка старых записей не входят
  в MVP и остаются кандидатами на отдельную итерацию;
- лог-маркеры идут через общий logger `tg_torrent_drop` с префиксами `YouTube worker`
  и `YouTube Plex ...`.

Статус для пользователя:

- после подтверждения бот показывает позицию в очереди;
- при старте скачивания обновляет сообщение: качество, стадия, прогресс,
  скорость и ETA, если `yt-dlp` их отдает;
- технически progress идет через `progress_hooks`: hook получает status,
  downloaded/total bytes, speed, eta и filename, если эти поля доступны;
- скачивание выполняется в отдельном worker-thread, а hook обновляет
  thread-safe state задачи;
- progress сохраняется примерно раз в секунду, а карточка в Telegram
  обновляется примерно раз в 2 секунды, чтобы быстрые загрузки тоже были видны;
- после успеха удаляет карточку статуса и отдельно отправляет сообщение о
  завершении, чтобы Telegram показал уведомление;
- после ошибки удаляет карточку статуса и отдельно отправляет понятную причину
  с кнопкой "Открыть YouTube".

Критерий успеха:

- несколько ссылок подряд обрабатываются последовательно;
- рестарт контейнера не оставляет непонятное состояние;
- админ может понять по логам, что произошло.

## Этап 4. Plex: простое появление в отдельной секции

Подготовка вне бота:

- создать в Plex отдельную библиотеку для YouTube-видео;
- выбрать тип библиотеки "Другие видео";
- назвать библиотеку `YouTube`;
- указать ей папку `/volume1/youtube` или соответствующий путь на NAS;
- включить в Plex автоматическое сканирование библиотеки при изменениях как
  дополнительную страховку; основной ускоритель в боте - refresh секции через API.
- в дополнительных настройках включить local assets / local media assets, если
  эта опция доступна для выбранного типа библиотеки;
- для DS223 не включать генерацию video preview thumbnails "when media is
  added"; если нужны превью, оставить только scheduled task.

Изменения в боте:

- сохранять файлы в структуре, удобной для Plex;
- после скачивания сообщать путь;
- не смешивать YouTube-ролики с фильмами/сериалами.

Начальное правило naming:

```text
/youtube_storage/<channel>/<YYYY-MM-DD> - <title> [yt-<video_id>]/
  <YYYY-MM-DD> - <title> [yt-<video_id>].mp4
  poster.jpg
  fanart.jpg
  info.json
```

Почему так:

- отдельная папка на ролик дает место для poster/background/metadata рядом с
  видео;
- дата помогает сортировать выпуски одного канала;
- `video_id` защищает от дублей и одинаковых названий;
- `info.json` сохраняет исходные metadata `yt-dlp` для бота, даже если Plex не
  использует его напрямую.

Ожидаемое поведение Plex:

- отдельная библиотека "Другие видео" не должна матчить ролики как фильмы или
  сериалы;
- локальные `poster.jpg` и `fanart.jpg` кладем рядом с видео, потому что Plex
  поддерживает такие имена local artwork в папке элемента;
- описание/`info.json` Plex может не использовать напрямую, поэтому они нужны
  боту и для будущих улучшений, а не как гарантированный источник описания в UI.

Smoke-test для закрытия Plex-вопроса в MVP:

1. Положить один короткий тестовый ролик в `/volume1/youtube/...`.
2. Положить рядом `poster.jpg`, `fanart.jpg`, `info.json`.
3. Просканировать библиотеку `YouTube`.
4. Проверить, что ролик появился именно в `YouTube`, а не в "Фильмы" или
   "Телепередачи".
5. Проверить, подтянулся ли `poster.jpg`.
6. Если `poster.jpg` не подтянулся автоматически, не блокировать MVP: файл все
   равно скачан и виден в Plex; улучшение обложек вынести в следующую итерацию.

Критерий успеха:

- скачанный ролик появляется в отдельной Plex-библиотеке после автоматического
  сканирования или ручного scan в Plex.
- Smoke-test 2026-06-16 пройден на DS223: тестовый ролик из `/volume1/youtube`
  появился в библиотеке `Youtube`, подтянул `poster.jpg` и воспроизводится.

## Этап 5. Plex refresh/polling через API

Новые env-переменные:

```env
YOUTUBE_PLEX_SECTION=
YOUTUBE_PLEX_LIBRARY_NAME=YouTube
YOUTUBE_PLEX_REFRESH_AFTER_DOWNLOAD=true
YOUTUBE_PLEX_POLL_AFTER_DOWNLOAD=true
```

Изменения:

- по умолчанию после успешного скачивания вызывать refresh конкретной Plex-секции,
  потому что автоскан через NAS/Docker bind mount может сработать не сразу;
- если `YOUTUBE_PLEX_POLL_AFTER_DOWNLOAD=true`, дождаться появления файла в Plex
  ограниченное число попыток;
- для выбора секции использовать `YOUTUBE_PLEX_SECTION`, а если он пустой -
  искать библиотеку по `YOUTUBE_PLEX_LIBRARY_NAME`;
- уведомить: "скачано" или "скачано и появилось в Plex";
- если Plex не нашел ролик за лимит времени, не считать скачивание ошибкой.

Критерий успеха:

- после скачивания Plex сам видит ролик или бот ускоряет это через API;
- пользователь получает финальное состояние, а не только путь к файлу.

## Закрытые технические решения

- Docker/NAS: используем отдельную shared folder `/volume1/youtube` и mount
  `${YOUTUBE_NAS_PATH:-/volume1/youtube}:/youtube_storage:rw`; mount добавляем
  только после создания shared folder.
- Plex: создаем отдельную библиотеку `YouTube` типа "Другие видео"; ролики не
  кладем внутрь `/volume1/video`, чтобы не смешивать их с фильмами/сериалами.
- Plex artwork: сохраняем `poster.jpg` и `fanart.jpg` рядом с видео; ожидаем
  local artwork behavior, но подтверждаем smoke-test'ом на реальном Plex.
- Progress: используем Python API `yt-dlp` и `progress_hooks`; Telegram-статус
  обновляем с throttle, без парсинга обычного stdout.

## Риски

- YouTube может ломать скачивание изменениями на своей стороне.
- `yt-dlp` нужно регулярно обновлять.
- DS223 не рассчитан на transcode и массовые загрузки в рамках этой фичи.
- 1080p может быть тяжелым для длинных роликов по диску и времени.
- Без cookies часть роликов будет недоступна.
- Правовые и ToS-риски остаются; бот не должен позиционироваться как обход
  ограничений YouTube.

## Что не делаем в первой версии

- playlists;
- каналы и подписки;
- cookies;
- private/members-only/age-restricted;
- live streams;
- audio-only;
- 4K;
- transcode;
- отдельный self-hosted YouTube-архиватор;
- автоматическую отправку больших файлов в Telegram.
