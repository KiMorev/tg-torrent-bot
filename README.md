# PlexLoader

> **Версия 1.0** · 🧠 умный AI-поиск

[![tests](https://github.com/KiMorev/tg-torrent-bot/actions/workflows/test.yml/badge.svg)](https://github.com/KiMorev/tg-torrent-bot/actions/workflows/test.yml)
[![ghcr](https://img.shields.io/badge/ghcr.io-kimorev%2Ftg--torrent--bot-blue?logo=docker)](https://github.com/KiMorev/tg-torrent-bot/pkgs/container/tg-torrent-bot)

Telegram-помощник для домашнего киносервера на базе Synology + Plex с **умным AI-поиском**.

Ищет фильмы и сериалы по нескольким трекерам (через Jackett или прямо в Rutracker), ведёт ленту новинок с фильтрацией по рейтингу Кинопоиска, проверяет наличие в Plex до скачивания, и присылает push-уведомление с прямой ссылкой на просмотр когда файл появился в библиотеке. Сами раздачи — `.torrent` и magnet — скачиваются в Synology Download Station в фоне, без участия пользователя.

**🧠 AI-фичи** (когда настроен `OPENAI_API_KEY`):
- Автоисправление опечаток в поиске («Дюра» → «Дюна» одним тапом)
- Проверка GPT'ом привязки фильма к Кинопоиску — меньше неверных матчей в `/new`
- Короткие пояснения «почему этот фильм» в карточках `/new`
- Голосовой поиск через Whisper — записать voice вместо ввода названия

## Что умеет бот

### 🔍 Поиск и подбор контента

- **Умный поиск через Jackett** — единая точка для всех настроенных индексеров (Rutracker, NNMClub, BigFanGroup, RuTor и др.) с выбором подмножества трекеров, фильтром качества, и отображением источника `[RT] [NNM] [BFG]…` в карточках результатов.
- **Прямой поиск по Rutracker.org** — доступен отдельной кнопкой когда настроены оба источника; при отсутствии Jackett — основной движок.
- **Поиск сериалов** с поддержкой обоих форматов в названиях: `Сезон: N / Серии: K-M из L` (Rutracker) и `SxxExx of N` (foreign tracker через Jackett). Кнопка **«🔎 Другой сезон»** после добавления открывает пикер сезонов (если Кинопоиск знает общее число — кнопки 1…N с разбивкой по 5; иначе «✏️ Свой номер»).
- **Поиск по ссылке Кинопоиска** — вставил ссылку → бот извлёк название/год/режиссёра и сделал запрос.
- **Подсказки на пустой выдаче** — на экране «ничего не найдено» бот предлагает повторить с ослабленными фильтрами: «🔍 Без фильтра качества», «🌐 На всех трекерах», комбинация — кнопки появляются только когда они реально что-то меняют.

### 🎬 Новинки и рейтинг

- Команда **`/new`** — фоновый кэш топ-фильмов и мультфильмов с фильтром по году, качеству (1080p по умолчанию, опциональный 2160p), отсечением сериалов / CAM/TS / adult-раздач, обогащением через Кинопоиск и сортировкой по нормализованной [0,1] оценке (рейтинг КП 35% / свежесть 20% / популярность 20% / тех. качество 25%).
- **Двуязычные карточки**: «Проект «Конец света» / Project Hail Mary».
- **Значок ✅ 1080** напротив фильмов, которые уже есть в Plex-библиотеке — не качаем по второму разу.
- **Per-user «🆕»-плашка**: для каждого пользователя ведётся отдельный набор «уже виденных» фильмов. Push в чат не гасит плашку — она остаётся пока пользователь не откроет `/new` и не увидит фильм в списке (так удобнее найти его глазами).
- **Push-уведомления о новых топ-фильмах** в подписанных чатах — раздельные сигналы «уведомлён» и «показан в /new» для каждого пользователя, без спама.
- **Выбор трекеров для рейтинга** через UI без перезапуска бота; авто-мониторинг состава индексеров Jackett — при удалении трекера его данные вычищаются из кэша автоматически.

### 📺 Интеграция с Plex (опционально)

- **Pre-download check**: предупреждение если фильм/сериал уже в Plex и качество сравнимо — экономит трафик и место. Кнопка «⬇️ Скачать всё равно» для случая «хочу 4K вместо 1080».
- **Polling после завершения**: бот мониторит Plex до 10 минут после загрузки. Когда фильм проиндексировался — приходит сообщение `✅ … добавлен в Plex` с кнопкой **«▶️ Смотреть в Plex»** ([Plex Universal Link](https://support.plex.tv/articles/218168898-universal-links/) — iOS/Android Plex app перехватывает, на десктопе открывается Plex Web).
- **Радар «без матча» в Plex**: список файлов в библиотеке без metadata + опциональные push-уведомления когда появляется новый несматченный файл.

### 🔔 Подписки и автоматизация

- **Подписка на сезон сериала** — бот следит когда выйдут новые эпизоды и сам докачает; для Jackett-результатов сохраняется trackerName + topic_url + season + last_episode_end как якорь.
- **Уведомления о завершении** скачивания: текст со статусом + кнопки «▶️ Открыть Plex», «🧹 Удалить из списка», «📋 К списку загрузок».
- **Уведомления о дублирующих загрузках** — если торрент уже есть в Download Station, бот автоматически удаляет дубль и отправляет контекстное сообщение: «✅ уже скачан», «📌 скачивается (X%)» с кнопкой **«🔔 Уведомить когда готово»** для подписки на чужую активную загрузку, «⚠️ на паузе» с кнопкой запуска, и т.п.
- **Отложенная очередь** для скачиваний, которые сейчас не получаются: трекер недоступен → кнопка «⏳ Поставить в очередь» → задача в `state/pending_downloads.json`, ретрай раз в 5 минут, TTL 24ч, push об успехе/отказе.

### 📋 Управление и админ

- Команда **`/status`** — список активных и недавно завершённых загрузок с постраничной навигацией. Авто-обновление каждые 30 с пока есть активные задачи.
- **Карточка задачи** с прогрессом, скоростью, оставшимся временем, кнопками паузы/удаления.
- Команда **`/admin`** — главная сводка состояния (загрузки, подписки, зависшие уведомления), диагностика всех сервисов одной кнопкой, управление пользователями, выбор трекеров для рейтинга `/new`, кнопки сброса KP-кэша и счётчиков доставки.
- **Управление доступом**: запросы новых пользователей, одобрение/отказ, просмотр списка одобренных, удаление с очисткой всех связанных подписок и задач.

### ⚙️ Технический фундамент

- Принимает `.torrent`-файлы и magnet-ссылки из чата → задача в Synology Download Station через официальный API; retry при временных сбоях сети.
- **Auto-fallback цепочка** для скачивания через Jackett: ошибка прокси → прямой `rutracker_client` (для Rutracker-результатов) → magnet (если есть). Закрывает типичный случай «Jackett-сессия Rutracker протухла».
- **Auto-добавление публичных трекеров** к BT-задачам для ускорения раздачи.
- **Авто-очистка** завершённых задач из Download Station по таймеру.
- **Устойчивая доставка push-уведомлений**: различаются временные ошибки Telegram (rate-limit/timeout/network — без штрафа, ретрай) и постоянные (бот заблокирован / chat not found — после 3 попыток skip). Защита от багов в формате нашего сообщения — не блокирует здоровый чат за наш баг.
- **Self-healing**: если task_owner потерян, но карточка задачи активна — получатель push восстанавливается; sticky-owner на регистрации карточки. Подробные INFO-логи причин пропуска (legacy_done / no recipients / failures cap) для диагностики из `docker logs`.
- **Backoff/cooldown** для Rutracker при CAPTCHA, HTTP 403/503, таймаутах и сетевых ошибках.

## Установка

1. Скачай два файла из репозитория в папку `/volume1/docker/tg_torrent_drop` на NAS:
   - [`compose.yaml`](compose.yaml) — описание контейнеров
   - [`.env.example`](.env.example) → переименуй в `.env` и заполни секреты
2. В **Synology Container Manager** → **Проекты** → создай проект из этой папки.
3. Нажми **Собрать** → **Запустить**.

Container Manager скачает готовый образ из GHCR (`ghcr.io/kimorev/tg-torrent-bot:latest`)
и поднимет рядом контейнер Watchtower, который автоматически обновляет бота при
каждом push в `main` (см. ниже «Автоматическое обновление»). Никаких `.py`-файлов
на NAS держать не нужно — они живут внутри образа.

**Опциональный bind-mount для блока «📀 Хранилище» в `/admin`**: в `compose.yaml`
по умолчанию есть строка `- /volume1/video:/storage:ro` — read-only пробрасывает
видеотом в контейнер, бот считает свободное место через `shutil.disk_usage()`.
Если у тебя видеотека в другом пути — поправь **левую** часть mount-а
(`/volume1/share/movies:/storage:ro`), правая часть `/storage` остаётся неизменной.
Хочешь отключить feature — удали строку целиком.

## Автоматическое обновление

Пайплайн end-to-end:

1. Push в `main` на GitHub → workflow `.github/workflows/test.yml` запускает `pytest`.
2. Если тесты зелёные — job `build-and-push` собирает multi-arch образ
   (`linux/amd64` + `linux/arm64`) и пушит в GHCR с двумя тегами:
   - `:latest` — всегда указывает на свежий main
   - `:sha-<short>` — immutable, для отката
3. Watchtower на NAS опрашивает GHCR раз в 5 минут. Когда видит новый digest
   у `:latest` — делает `docker pull`, останавливает старый контейнер, поднимает
   новый, удаляет устаревший layer.
4. После успешного обновления — Watchtower шлёт push в Telegram **только
   администраторам** (используются `BOT_TOKEN` и `ADMIN_CHAT_IDS` из `.env`;
   обычные approved-пользователи уведомление НЕ получают).

Если CI падает — образ не публикуется, бот остаётся на старой версии.

### Уведомления об автообновлении

Watchtower использует уже настроенный `BOT_TOKEN` и список `ADMIN_CHAT_IDS`
из `.env` — отдельных секретов не нужно. При каждом успешном обновлении
контейнера приходит сообщение примерно такого вида:

```
1 Scanned, 1 Updated, 0 Failed
- /tg_torrent_drop (ghcr.io/kimorev/tg-torrent-bot:latest):
    Updated to sha256:abc123...
```

Если `ADMIN_CHAT_IDS` в `.env` пуст — Watchtower молча работает без
уведомлений (само обновление не страдает). Чтобы временно отключить
уведомления не убирая админов — закомментируй строку
`WATCHTOWER_NOTIFICATION_URL` в `compose.yaml` и пересобери проект.

### Откат на предыдущую версию

В `compose.yaml` поменяй `:latest` на конкретный SHA:

```yaml
    image: ghcr.io/kimorev/tg-torrent-bot:sha-9f7aafc
```

Затем в Container Manager → **Action → Build**. Список доступных тегов:
https://github.com/KiMorev/tg-torrent-bot/pkgs/container/tg-torrent-bot

После того как починили проблему в `main` — верни `:latest`.

### Принудительное обновление

Watchtower проверяет GHCR раз в 5 минут. Если не хочешь ждать — рестартни
контейнер `watchtower` через Container Manager (стартовый цикл сразу проверит
новые образы).

## Настройка

### Обязательные параметры

```env
BOT_TOKEN=123456:ваш_токен
ALLOWED_CHAT_IDS=123456789
DS_URL=https://192.168.1.103:5001
DS_ACCOUNT=tg_bot_ds
DS_PASSWORD=ваш_пароль
DS_DESTINATION=video
```

`ALLOWED_CHAT_IDS` — один или несколько chat_id через запятую:
```env
ALLOWED_CHAT_IDS=123456789,987654321
```

Узнать свой chat_id: написать `/id` боту.

`DS_DESTINATION` — имя общей папки без `/volume1/`. Для `/volume1/video` пишем `video`.

---

### Пользователь Download Station

Создай отдельного пользователя Synology (например, `tg_bot_ds`) и выдай ему:
- доступ к DSM;
- доступ к Download Station;
- права чтения/записи на папку назначения;
- папку назначения по умолчанию в Download Station.

`DS_VERIFY_SSL=false` — практичный режим для домашней Synology с самоподписанным сертификатом: контейнер не будет проверять TLS-сертификат DSM и соединение не сломается из-за локального self-signed сертификата.

Если DSM доступен через публичный домен, reverse proxy или другой недоверенный сегмент сети, лучше выпустить доверенный сертификат и поставить:

```env
DS_VERIFY_SSL=true
```

---

### Администраторы и доступ

```env
ADMIN_CHAT_IDS=123456789
ACCESS_APPROVALS_ENABLED=true
```

`ADMIN_CHAT_IDS` — если не задан, администраторами считаются все из `ALLOWED_CHAT_IDS`.
Если ботом пользуются несколько человек, задайте `ADMIN_CHAT_IDS` явно: администратор видит все загрузки, управляет пользователями и может удалять подписки.

При `ACCESS_APPROVALS_ENABLED=true` новый пользователь пишет `/start`. Бот отправляет администратору запрос с кнопками **✅ Разрешить** / **🚫 Отклонить**. После одобрения chat_id сохраняется в постоянное хранилище Docker-тома — редактировать `.env` не нужно.

Команда `/users` (только для администратора) показывает всех пользователей с доступом: администраторов, постоянных (из конфига), одобренных — с именем и датой выдачи доступа. Одобренных можно удалить кнопкой прямо из списка.

### Часовой пояс и лимиты

```env
TZ=Europe/Moscow
BOT_TIMEZONE=Europe/Moscow
LOG_LEVEL=INFO
MAX_TORRENT_FILE_MB=20
STATE_DIR=/data
```

`TZ` — часовой пояс контейнера.

`BOT_TIMEZONE` — часовой пояс для времени в сообщениях бота. Если не задан, используется `TZ`, затем `Europe/Moscow`.

`LOG_LEVEL` — уровень логирования (`INFO`, `DEBUG`, `WARNING` и т.д.).

`MAX_TORRENT_FILE_MB` — максимальный размер `.torrent`-файла, принимаемого от пользователя.

`STATE_DIR=/data` — директория для хранения JSON-файлов состояния. Должна быть смонтирована как volume (по умолчанию `tg_torrent_drop_state`).

### Публичные трекеры

```env
TRACKERS_MODE=auto
TRACKERS_URL=https://raw.githubusercontent.com/ngosang/trackerslist/master/trackers_best.txt
TRACKERS_MAX=20
TRACKERS_CACHE_TTL_HOURS=24
TRACKERS_BACKGROUND_ENABLED=true
TRACKERS_BACKGROUND_INTERVAL_SECONDS=180
```

`TRACKERS_MODE=auto` — автоматически добавлять публичные трекеры к новым BT-задачам. Существующие трекеры сохраняются, дубли пропускаются, приватные `.torrent`-файлы не затрагиваются.

При `TRACKERS_BACKGROUND_ENABLED=true` фоновый монитор сканирует активные BT-задачи каждые `TRACKERS_BACKGROUND_INTERVAL_SECONDS` секунд. Кнопка **➕ Добавить трекеры** показывается только для незавершённых BT-задач, которые ещё не были обработаны.

### Уведомления

```env
TASK_NOTIFICATIONS_ENABLED=true
TASK_NOTIFICATION_STATUSES=finished,seeding,error
TASK_NOTIFY_EXTERNAL_TASKS=false
NOTIFY_CHAT_IDS=
```

Бот присылает уведомление, когда задача достигает одного из статусов в `TASK_NOTIFICATION_STATUSES`. Статус `seeding` включён по умолчанию, так как для BT-задач означает завершение скачивания. Уведомление отправляется тому, кто создал задачу. Задачи, созданные не через бота, игнорируются, если не задан `NOTIFY_CHAT_IDS` или `TASK_NOTIFY_EXTERNAL_TASKS=true`.

После успешной отправки финального уведомления бот удаляет временную карточку задачи, которая появилась при добавлении или просмотре статуса. Итоговое уведомление остаётся в чате как история результата.

**Устойчивость доставки.** Бот различает временные и постоянные ошибки Telegram:
- **Временные** (`RetryAfter`/429, `TimedOut`, `NetworkError`) — не учитываются как «провальная попытка». При `RetryAfter` бот ждёт `retry_after` секунд (до 30с), потом продолжает; для timeout/network — просто повторит на следующем цикле через 10 секунд.
- **Постоянные** (`Forbidden` — пользователь заблокировал бота, `BadRequest "chat not found"`) — после 3 попыток адресат исключается из списка получателей этой задачи.
- **Ошибки формата сообщения** (`BadRequest "inline keyboard button url is invalid"`, `"can't parse entities"` и т. п.) — это **наш баг**, не вина чата. Счётчик `failures` для chat_id **не** растёт, в логе ERROR-строка, ретраи продолжаются на каждом цикле (пока код не починен). Так бот защищается от ситуации когда плохой URL/HTML в сообщении блокирует здоровый чат.

Состояние доставки сохраняется в `state/notified_tasks.json` после каждой задачи (а не один раз в конце цикла), так что перезапуск бота во время доставки не приводит к повторной отправке уже доставленных уведомлений.

**Зависшие уведомления.** В админ-панели `/admin` появляется строка состояния:
- `• Уведомления о завершении: ✅ всё доставляется` — все push-уведомления долетают.
- `• Уведомления о завершении: ⚠️ зависших N задач — тапни «🔄 Сбросить счётчики (N)»` — для N задач счётчик `failures[chat_id]` достиг лимита (3). Кнопка-сброс появляется в клавиатуре и обнуляет счётчики разом; на следующем тике (раз в 180с) попытки доставки начнутся заново с 0.

Кнопка сброса нужна редко: только если push массово застряли из-за устаревшей версии кода (например, баг с `plex://` URL, май 2026) или после длительного периода когда Telegram возвращал ошибки. Авто-очистка зависших записей при удалении задачи из DS работает штатно через `prune_stale_task_state`.

**Кнопка «🔔 Уведомить когда готово»** доступна в уведомлении о дублирующей загрузке, если оригинал ещё скачивается. После нажатия пользователь получит финальное уведомление наравне с тем, кто добавил задачу.

**Дублирующие загрузки** — если торрент с таким же содержимым уже существует в Download Station, бот автоматически удаляет дубль из очереди и отправляет уведомление с контекстом:
- *✅ Этот файл уже загружен* — оригинал завершён или раздаётся; показывает размер, статус, кто и когда добавил
- *📌 Этот файл уже скачивается* — показывает прогресс и кнопку подписки на уведомление
- *⚠️ Такой файл уже добавлен, но остановлен* — оригинал на паузе; предлагает кнопку запуска
- *📌 Такой торрент уже был добавлен ранее* — оригинал не найден (возможно, уже удалён из DS)

### Отложенная загрузка (pending queue)

```env
PENDING_DOWNLOADS_ENABLED=true
PENDING_DOWNLOADS_INTERVAL_SECONDS=300
PENDING_DOWNLOADS_TTL_HOURS=24
```

Когда скачивание `.torrent` не удалось (Rutracker лёг, Jackett 404, network blip), бот показывает компактное сообщение об ошибке с кнопками:

- **🔄 Повторить** — сейчас же попробовать ту же раздачу заново.
- **⏳ Поставить в очередь** — сохранить запрос в `state/pending_downloads.json` и пытаться скачать в фоне раз в `PENDING_DOWNLOADS_INTERVAL_SECONDS` (default 5 минут). Каждая попытка идёт по той же цепочке fallback: Jackett-proxy → rutracker_client напрямую → magnet (если есть).
- **✖️ Закрыть** — закрыть сообщение.

При успехе бот присылает уведомление «✅ Отложенная загрузка стартовала», и задача попадает в обычный мониторинг прогресса. Если за `PENDING_DOWNLOADS_TTL_HOURS` (default 24ч) не получилось — пользователь получает «⌛ Не удалось скачать за …ч» с числом попыток и последней ошибкой.

Отключается через `PENDING_DOWNLOADS_ENABLED=false` — кнопка «⏳ Поставить в очередь» тогда не показывается, фоновой петли нет.

### Поиск на Rutracker.org

```env
RUTRACKER_USERNAME=ваш_логин
RUTRACKER_PASSWORD=ваш_пароль
RUTRACKER_MAX_RESULTS=50
```

Если не настроены ни Rutracker, ни Jackett — поиск отключён. Если настроен только Rutracker — поиск идёт напрямую через Rutracker. Если настроен Jackett — поиск по умолчанию идёт через Jackett; при наличии обоих источников дополнительно доступна кнопка **🔍 Rutracker (прямой)** для прямого поиска в обход Jackett.

Если Rutracker возвращает CAPTCHA, HTTP 403/503, таймаут или сетевую ошибку, бот ставит обращения к Rutracker на короткую паузу. Повторные ошибки увеличивают задержку до 15 минут, успешный запрос сбрасывает паузу.

**Поиск без команды.** Когда Rutracker или Jackett настроены, можно просто написать название прямо в чат — бот сам предложит варианты качества и запустит поиск через доступные источники.

**🎙 Голосовой поиск** (опционально). Если задан `OPENAI_API_KEY`, бот принимает голосовые сообщения: запишите голосом название фильма/сериала → бот пропускает аудио через OpenAI Whisper API → получает текст → запускает обычный поиск как будто бы вы напечатали этот запрос вручную.

```env
OPENAI_API_KEY=sk-...
VOICE_SEARCH_ENABLED=true
VOICE_MAX_SECONDS=30
```

- **Цена**: ~$0.006/мин у OpenAI, pay-per-use — без подписки. Типичное голосовое 5-15 сек = $0.001-0.003. 0 запросов = $0. Рекомендую сразу выставить **hard limit** в [OpenAI Settings → Limits](https://platform.openai.com/settings/limits) на $5/мес — даже при поломке отрубится автоматически.
- **Многоязычность**: Whisper auto-detect, отлично работает с русскими названиями, английскими, миксом.
- **Graceful degrade**: пустой `OPENAI_API_KEY` → голосовые игнорируются (фича просто выключена). Если Whisper упал/таймаут → пользователь видит «Не получилось распознать, попробуйте ещё раз или напишите текстом».
- **Лимит длительности**: `VOICE_MAX_SECONDS` (default 30 сек) — защита от случайного многоминутного аудио. Слишком длинное → дружелюбное сообщение с просьбой повторить короче.
- **Мониторинг в `/admin`**: блок «🎙 Голосовой поиск» доступен в разделе **🧭 Диагностика**. Показывает: валидность ключа (live-ping `/v1/models`), счётчик запросов и расход за текущий месяц (`request_count`, `total_seconds`, `~$X`), последний запрос (timestamp + распознанный текст + outcome ✅/❌), последнюю ошибку (`auth`, `quota_exceeded`, `timeout`, `network`, …). Если ключ невалиден или баланс исчерпан — в **главной /admin** появляется warning-строка «⚠️ 🎙 Голосовой поиск: …», чтобы заметить сразу, не открывая диагностику. Счётчики автоматически сбрасываются на 1-е число каждого месяца.

### 🧠 GPT для точности поиска (опционально)

Если задан `OPENAI_API_KEY` (тот же что для голосового поиска), включаются GPT-улучшения точности подбора метаданных:

```env
GPT_ENABLED=true
GPT_MODEL=gpt-4o-mini
```

**Текущие применения** (PR1):

- **🎯 KP confidence check** — после того как Кинопоиск возвращает совпадение для фильма в `/new`, GPT-проверка: «действительно ли это тот фильм?». Если confidence < 0.7 — карточка остаётся без КП-метаданных (вместо привязки к неверному фильму). Сильно снижает ложные матчи.
- **🔎 Did-you-mean** — на странице с 0 результатами поиска бот через GPT генерирует до 3 вариантов запроса (опечатки, оригинальные названия, добавление года) и показывает их кнопками. Промпт с few-shot примерами заточен на **исправление опечаток в названиях**: «Дюра» → «Дюна», «Аркаин» → «Аркейн». В GPT уходит **базовый запрос без quality-суффикса** (бот сам убирает «1080p» из текста — иначе модель тратит ходы на варианты суффикса вместо лечения опечатки).

### 🎞 Quality-aware поиск (Strategy 2)

Бот не отправляет «1080p» в текст поиска к Jackett/Rutracker. Вместо этого:

1. Из запроса извлекается **base** (название) и **preferred quality** (если есть).
2. Источникам уходит только base — получаем **полную выдачу** независимо от качества.
3. Результаты классифицируются client-side через `movie_discovery.detect_quality` (распознаёт 1080p / 2160p / 720p / 480p в названии торрента надёжнее чем Jackett-фильтр по строке).
4. Фильтр оставляет только preferred bucket. Banner показывает статистику:
   ```
   🎬 Найдено 47 раздач, показаны 28 в 1080p. Также есть: 2160p × 7, 720p × 12.
   ```
5. **Если в preferred quality пусто, но есть в других** — показываются все с банером:
   ```
   ⚠️ В 1080p ничего не найдено. Показаны все качества: 2160p × 2, 720p × 3.
   ```
   Это закрывает баг «фильм существует, но не в 1080p → 0 результатов → дед-енд».
6. Если total = 0 — обычный no-results screen с GPT-подсказками.

**Зачем это нужно**: точнее (наш парсер vs Jackett text-filter), быстрее (1 запрос вместо потенциальных 2), и открывает дорогу к будущим quality-aware фичам (per-user preferences, upgrade detection, smart recommendations).

Цена: gpt-4o-mini по $0.150/1M input + $0.600/1M output. Типичный KP-вызов ≈ $0.00006, did-you-mean ≈ $0.00005. **Общий бюджет с Whisper** — лимит ставится в OpenAI Settings → Limits. При проблемах OpenAI (`auth`/`quota_exceeded`/timeout) GPT-улучшения молча отключаются — основной поиск/новинки продолжают работать как обычно (graceful fallback).

**Поток поиска (Jackett настроен):**

```
Написать название
        → [🔍 Искать (1080p)]  [⚙️ Расширенный поиск]  [🔍 Rutracker (прямой)]
```

Если Jackett не настроен, кнопки **Jackett** отсутствуют и быстрый поиск идёт напрямую через Rutracker.

**Быстрый поиск** — сразу ищет с добавлением `1080p` к запросу через Jackett (или Rutracker, если Jackett не настроен).

**Расширенный поиск** — настройка перед запуском:

| Параметр | Варианты |
|---|---|
| Качество | 🎬 4K / 📺 1080p / 📺 720p / 🔍 Любое |
| 🎵 Оригинальная дорожка | вкл/выкл (добавляет `Original` к запросу) |
| 💬 Субтитры | вкл/выкл (добавляет `Sub` к запросу) |
| Трекеры | выбор конкретных индексеров Jackett или все сразу |

После выбора бот оценивает все полученные результаты своим рейтингом и показывает список (до 5 на странице) с размером и числом сидеров:

- ⭐ — рекомендованный результат (выбирается автоматически по качеству раздачи: тип релиза, разрешение, озвучка/субтитры, число сидеров)
- 🔎 — остальные результаты; в скобках указан источник: `[RT]`, `[NNM]` и т.д.
- **◀ / ▶** — листать страницы при большом количестве результатов

Нажми на нужный → открывается карточка с обложкой (если доступна на странице раздачи), размером и числом сидеров → **✅ Скачать** — торрент добавляется в Download Station.

**Сериал: добавить другие сезоны.** Если в названии добавленной раздачи есть маркер сезона — русский (`Сезон: N`) или английский (`SNN` / `SNNeNN`, типичный формат foreign-трекеров через Jackett) — бот после успешного добавления показывает кнопку **«🔎 Другой сезон»**. По нажатию открывается пикер сезонов (если Кинопоиск знает общее число сезонов — кнопки 1…N с разбивкой по 5; иначе только «✏️ Свой номер»). Фильтр по сезону тоже понимает обе формы — запрос «Аркейн сезон 1» корректно отсекает результаты `S02E03`. Особенности флоу:

- **Качество переносится автоматически.** Если ты вручную выбрал релиз в 1080p, текст пикера показывает «Будет искать в качестве 1080p (по выбранному торренту)» — следующий поиск автоматически добавит фильтр качества, не нужно набирать вручную.
- **Кнопка «⬅️ Назад»** в пикере возвращает к карточке с «🔎 Другой сезон», если передумал — можно остаться без новой задачи и потом снова открыть пикер.
- **Если нужного сезона нет** в раздачах, бот вместо тихого «ничего не найдено» подскажет: *«На трекерах найдены сезоны: 1, 2, 3, 4»* и предложит «⬅️ К выбору сезона».

**Ничего не найдено — кнопки fallback.** Если по запросу 0 результатов, бот предлагает повторить поиск с ослабленными фильтрами (показываются только те кнопки, которые реально что-то меняют):

- **🔍 Без фильтра качества** — если в запросе был фильтр качества (например, `1080p` из настроек), повторяет поиск с базовым запросом.
- **🌐 На всех трекерах** — если используется Jackett и сейчас выбран не весь набор индексеров (по умолчанию только Rutracker), расширяет до всех.
- **🔍🌐 Без качества + все трекеры** — обе релаксации сразу, когда применимо.
- **❌ Отмена** — закрыть.

В админ-панели `/admin` кнопка **🧭 Диагностика** запускает единую проверку: Download Station, Rutracker, Jackett, Кинопоиск, кэш public-трекеров и состояние Plex-кнопки.

### Новинки фильмов и мультфильмов

```env
MOVIE_DISCOVERY_ENABLED=true
MOVIE_DISCOVERY_INTERVAL_HOURS=12
MOVIE_DISCOVERY_CACHE_FILE=/data/movie_discovery.json
MOVIE_DISCOVERY_SETTINGS_FILE=/data/movie_discovery_settings.json
MOVIE_DISCOVERY_DEBUG_FILE=/data/movie_discovery_debug.json
MOVIE_DISCOVERY_RUTRACKER_TM=32
MOVIE_DISCOVERY_JACKETT_REQUIRE_DATE=true
MOVIE_DISCOVERY_JACKETT_MAX_AGE_DAYS=32
MOVIE_DISCOVERY_LIMIT=30
MOVIE_DISCOVERY_MIN_KP_RATING=6.0
MOVIE_DISCOVERY_QUALITIES=1080p
```

Команда `/new` показывает готовый кэш свежих фильмов и мультфильмов. Бот обновляет его в фоне каждые `MOVIE_DISCOVERY_INTERVAL_HOURS` часов, а если кэш ещё пустой, первый запрос `/new` запускает обновление сразу. По умолчанию новинки ищутся только в `1080p`; если нужен 4K, укажи, например, `MOVIE_DISCOVERY_QUALITIES=1080p,2160p`.

Первичный поиск выполняется запросами по году и настроенному качеству: например, `2026 1080p`, `2025 1080p`; если включить 4K, добавятся запросы вроде `2026 2160p`. Для Rutracker дополнительно используется параметр `MOVIE_DISCOVERY_RUTRACKER_TM`, который соответствует полю **Торренты за** на веб-форме поиска: `1` — сегодня, `3` — последние 3 дня, `7` — неделя, `14` — две недели, `32` — месяц, `-1` — всё время. По умолчанию стоит `32`, поэтому Rutracker-часть блока действительно ограничена раздачами за последний месяц.

Для Jackett по умолчанию включён строгий режим: `MOVIE_DISCOVERY_JACKETT_REQUIRE_DATE=true`. В `/new` участвуют только результаты, где Jackett отдал дату публикации (`PublishDate`, `FirstSeen` или `pubDate`), и эта дата не старше `MOVIE_DISCOVERY_JACKETT_MAX_AGE_DAYS`. Это уменьшает широту выдачи, зато блок лучше соответствует идее именно новых раздач.

Логика специально грубая и осторожная: берутся только текущий и прошлый год, позапрошлые фильмы не попадают в выдачу. Сериалы, CAM/TS/HDTS, трейлеры, сборники и adult-раздачи отсекаются по заголовку и категории. Размер проверяется с учётом качества: для 2160p минимальный порог выше, чем для 1080p.

Если заголовок раздачи двуязычный (`Русское / Foreign`), бот сохраняет оба названия и показывает их в карточке и кнопке: «Проект «Конец света» / Project Hail Mary». Для фильмов только с русским или только с иностранным названием отображается одно.

Для корректной работы двуязычных названий из Rutracker необходимо снять галочку **Strip Russian Letters** в настройках Rutracker-индексера Jackett (см. [раздел Jackett](#jackett-расширенный-поиск)).

Если задан `KINOPOISK_API_KEY`, бот дополнительно ищет карточку фильма в Кинопоиске, добавляет ссылку, жанры и рейтинг, а результаты ниже `MOVIE_DISCOVERY_MIN_KP_RATING` скрывает. Если ключ не задан, новинки всё равно работают, но без рейтинга и ссылки на Кинопоиск.

**Кэш результатов Кинопоиска** хранится внутри того же файла `MOVIE_DISCOVERY_CACHE_FILE`. Найденные фильмы не перепроверяются 14 дней, «не найдено» — 3 дня (новые фильмы успевают появиться в KP-индексе). Устаревшие записи вычищаются автоматически перед каждым обновлением; размер ограничен 2 000 позициями. Если запись привязалась к неверному фильму (например, из-за омонима в названии), её можно сбросить кнопкой **🗑 KP кеш** в админ-панели.

Кэш хранится в `MOVIE_DISCOVERY_CACHE_FILE`. В Docker по умолчанию это `/data/movie_discovery.json`, поэтому обычная пересборка контейнера кэш не сбрасывает. Он пропадёт только при удалении Docker volume, смене `STATE_DIR`/`MOVIE_DISCOVERY_CACHE_FILE` или ручном удалении файла.

Диагностический файл `MOVIE_DISCOVERY_DEBUG_FILE` показывает, какие запросы выполнялись, сколько результатов пришло из Rutracker/Jackett, почему конкретные кандидаты были приняты или отброшены, какие карточки попали в финальный список и были ли данные о дате публикации раздачи.

**Управление трекерами рейтинга.** Кнопка **🎬 Трекеры новинок** в админ-панели открывает экран выбора трекеров Jackett, которые участвуют в построении рейтинга `/new`. По умолчанию участвуют все; можно отключить отдельные трекеры — изменение вступает в силу немедленно (без сетевых запросов, из кэша). Настройка сохраняется в `MOVIE_DISCOVERY_SETTINGS_FILE`.

**Автомониторинг трекеров.** При каждом плановом обновлении бот сравнивает текущий список трекеров Jackett с предыдущим. Если трекер был удалён в Jackett, его релизы и fingerprint-ы автоматически вычищаются из кэша, а сам трекер убирается из списка участвующих в рейтинге — всё это происходит без участия администратора в течение одного интервала обновления (до `MOVIE_DISCOVERY_INTERVAL_HOURS` часов).

### Подписки на обновления серий

```env
SUBSCRIPTION_CHECK_INTERVAL_HOURS=6
```

Когда результат поиска на Rutracker — неполная раздача (например, `Серии: 1-8 из 10`), бот показывает **две кнопки подписки** — выбираешь режим уведомлений сразу:

- **⬇️📺 Серии** — `notify_mode=per_episode`. Уведомление приходит на каждую новую серию или группу серий (текущее «классическое» поведение).
- **⬇️🎯 Сезон** — `notify_mode=season_complete`. Скачивание идёт каждый раз (Plex получает файлы по мере выхода серий), но push'и пользователю **подавляются** до момента когда сезон полностью завершён. Один консолидированный push «сезон вышел» вместо череды промежуточных. Удобно тем кто марафонит.

Режим хранится в подписке (`notify_mode`-поле) и его можно поменять в любой момент через `/admin → 🔔 Подписки` — рядом с каждой подписью есть toggle-кнопка (📺→🎯 или 🎯→📺). Текущий режим виден строкой «Режим: 📺 каждая серия» / «🎯 сезон целиком».

При нажатии торрент добавляется в Download Station и бот начинает отслеживать тему. Каждые `SUBSCRIPTION_CHECK_INTERVAL_HOURS` часов он проверяет заголовок страницы на Rutracker и при изменении счётчика серий автоматически скачивает обновлённый торрент. Уведомление зависит от режима: per_episode шлёт push сразу, season_complete тихо обновляет `last_episode_end` и ждёт пока `new_end >= total_episodes`.

Для Jackett-подписок бот сохраняет выбранный результат как якорь: трекер, ссылку на тему/карточку, название, сезон и текущий счётчик серий. При последующих проверках:

- Если подписка содержит ссылку на Rutracker-тему (`topic_url`) и Rutracker настроен — проверка идёт напрямую через Rutracker (быстрее и надёжнее): бот проверяет заголовок темы, и при обнаружении новых серий сразу скачивает обновлённый торрент.
- Иначе — бот повторно ищет через Jackett и выбирает подходящего кандидата с тем же трекером/сезоном и увеличившимся номером серии.

В обоих случаях при обнаружении обновления бот **автоматически скачивает** торрент и добавляет задачу в Download Station; уведомление содержит кнопку **🔄 Статус задачи**. Если скачивание не удалось — уведомление всё равно отправляется с кнопкой ручной загрузки. Когда сезон завершён и торрент успешно скачан — подписка снимается автоматически; если скачивание финального обновления не удалось, подписка сохраняется для повторной попытки.

**Уведомления:**

```
🔔 Клиника / Сезон 10: новая серия!
Эпизодов: 8 → 9 из 10
Торрент обновлён в Download Station.
[📋 К загрузкам]  [🔕 Отписаться]

🔔 Клиника / Сезон 10: сезон завершён!
Эпизодов: 9 → 10 из 10 ✅
Торрент обновлён в Download Station.
Подписка снята автоматически.
```

Команда `/subs` показывает список активных подписок с кнопками **🔕 Отписаться**.

### Подписка на уведомления о новинках `/new`

Любой разрешённый пользователь может подписаться на уведомления о новых фильмах в списке `/new`. При каждом фоновом обновлении кэша бот проверяет — какие фильмы в **топ 10** этот пользователь ещё **не видел**, и рассылает подписчикам короткое сообщение в чат с такими фильмами.

**Как подписаться:** открыть `/new` → нажать кнопку **🔔 Подписаться на /new**. После подписки кнопка меняется на **🔕 Отписаться от /new**.

**Формат уведомления:**
```
🎬 Новые фильмы в /new:

• Дюна: Часть вторая / Dune: Part Two (2024) · КП 8.1
• Антидот (2025) · КП 7.3

[🎬 Открыть /new]  [🔕 Отписаться]
```

Нажатие **🎬 Открыть /new** раскрывает полный список прямо в уведомлении. **🔕 Отписаться** немедленно снимает подписку.

**Per-user семантика — два независимых сигнала.** Бот ведёт для каждого пользователя свой набор отметок на каждый фильм в `state/movie_discovery_settings.json` (поле `movie_seen_by_user`). Каждая запись содержит **два сигнала**:

- `notified_at` — бот отправил пользователю уведомление про этот фильм
- `shown_at` — пользователь открыл `/new` и увидел этот фильм в показанном топ-10

Что это означает на практике:
- **Плашка `🆕`** гаснет **только** после `shown_at` — то есть когда пользователь реально открыл `/new`. Push сам по себе плашку **не** гасит — поэтому, кликнув «🎬 Открыть /new» из уведомления, ты увидишь фильм **с плашкой** и легко найдёшь его глазами в списке.
- **Повторное уведомление** не приходит, если у фильма выставлен либо `notified_at` (уже пушнули), либо `shown_at` (юзер уже открывал `/new` и видел фильм без помощи push).
- **Плашка индивидуальна для каждого пользователя**: один открыл `/new` → его плашки исчезли. Другой открывая `/new`, всё ещё увидит свои плашки на тех же фильмах.
- **Рестарт бота** не вызывает ложных уведомлений (`movie_seen_by_user` персистится).
- **Фильм поднялся с 13-й позиции в 8-ю** (впервые в топ-10 для этого пользователя) — уведомление прилетит.

> Legacy-данные из старой версии (одна timestamp-строка на фильм вместо пары `{notified_at, shown_at}`) трактуются как «фильм уже был и в push, и в `/new`» — то есть после обновления бот **не** засыпет пользователя сводкой про прошлые фильмы.

**Тихие часы.** Уведомления отправляются только с 09:00 до 22:00 по локальному времени бота (`DISPLAY_TIMEZONE`). Вне окна бот просто не пушит — поскольку seen-set не обновляется, при следующем in-window refresh diff включит те же фильмы, что и должно было прилететь.

**Защита от ложных уведомлений (три слоя).** Источники (особенно Jackett) после рестарта контейнера дают «холодную» выдачу — индексеры ещё не прогрелись, top-10 формируется из неполных данных. Без защиты пользователь получает push о фильме, который через минуту исчезнет из выдачи. Бот защищается тремя слоями:

- **A. Skip push на первом refresh после старта.** Кэш обновляется (через `/new` всё видно), но рассылка не идёт — ждём следующего регулярного цикла. В логах: `notify skipped — first refresh after startup`.
- **B. Защита от регрессии.** Если новый top-10 потерял больше 60% фильмов из прошлого — refresh признаётся нестабильным, рассылка отменяется. В логах: `notify skipped — regression detected removed_pct=N%`.
- **C-lite. Consensus.** Push отправляется только для тех `kp_id`, которые присутствовали И в текущем top-10, И в предыдущем top-10 (поле `prev_top10_kp_ids` в кэше). Это значит фильм, впервые появившийся в топе, ждёт **один цикл** (12 ч) для подтверждения. Гарантия от транзиентных фильмов «один цикл живут и уходят».

Цена защиты: реальная новинка попадает в push с задержкой одного refresh-цикла (по умолчанию 12 ч). На практике это совпадает с моментом, когда фильм действительно стабилизировался в топе.

Управление подпиской также доступно через `/subs` — там отображается строка «🎬 Подписка на новинки: включена» с кнопкой отписки.

Количество подписчиков отображается в `/admin` в разделе «Новинки».

### Jackett (расширенный поиск)

```env
JACKETT_URL=http://192.168.1.103:9117
JACKETT_API_KEY=ваш_ключ
JACKETT_INDEXERS=all
JACKETT_MAX_RESULTS=20
JACKETT_FETCH_LIMIT=100
```

Jackett — основной поисковый движок бота когда настроен. Агрегирует результаты со всех сконфигурированных индексеров (Rutracker, NNMClub, BigFanGroup, RuTor и др.) в единый список.

> ⚠️ **Важная настройка для Rutracker-индексера в Jackett**
>
> В настройках индексера RuTracker.org (Jackett → шестерня рядом с RuTracker.org → Edit) есть опция **Strip Russian Letters**. По умолчанию она включена и обрезает кириллическую часть заголовка, из-за чего бот получает только английское название или вовсе «`[2026, WEB-DL 1080p] Original Rus`» без названия фильма.
>
> **Снимите галочку Strip Russian Letters** — тогда Jackett будет передавать полные двуязычные заголовки вида «`Невеста! / The Bride! [2026, ...]`», и бот сможет правильно группировать раздачи и показывать русские названия в блоке новинок.

Для включения нужно задать и `JACKETT_URL`, и `JACKETT_API_KEY`; если указать только одну переменную, бот остановится при старте с понятной ошибкой.

Когда Jackett включён, быстрый поиск и расширенный поиск используют Jackett по умолчанию. Если дополнительно настроен Rutracker, в начале поиска появляется кнопка **🔍 Rutracker (прямой)** для прямого поиска минуя Jackett.

Результаты из разных источников помечаются аббревиатурой трекера: `[RT]` — Rutracker, `[NNM]` — NNMClub, `[BFG]` — BigFanGroup, `[RuTor]` — RuTor и т.д.

`JACKETT_INDEXERS` — список индексеров через запятую или `all` для всех настроенных. По умолчанию выбирается Rutracker, если он есть в Jackett.

`RUTRACKER_MAX_RESULTS` — сколько результатов Rutracker передать в рейтинг бота. По умолчанию используется 50, это максимум одной страницы Rutracker и помогает не потерять подходящую раздачу до сортировки.

`JACKETT_FETCH_LIMIT` — сколько результатов запросить у Jackett для оценки рейтингом.

`JACKETT_MAX_RESULTS` — сколько лучших Jackett-результатов добавить к результатам Rutracker при расширенном поиске. Отдельный поиск через Jackett показывает весь полученный и отсортированный список.

В общей диагностике админ-панели Jackett отображается отдельным блоком: бот проверяет API и показывает список доступных индексеров.

### Plex Media Server

```env
PLEX_URL=http://192.168.1.103:32400
PLEX_TOKEN=xxxxxxxxxxxxxxxxxxxx
PLEX_MOVIE_SECTION=           # опционально; если пусто — секция фильмов определяется автоматически
```

Интеграция с Plex активируется, когда заданы оба `PLEX_URL` и `PLEX_TOKEN`. Без этих переменных бот работает полностью без изменений.

**Что даёт интеграция:**

1. **✅ значок в `/new`** — напротив фильмов, которые уже есть в вашей библиотеке Plex, появляется `✅ 1080` (или `4K`, `720` — в зависимости от качества в Plex). Это сразу видно в общем списке новинок.

2. **Проверка дубликата перед скачиванием** — при попытке скачать фильм или сезон сериала, который уже есть в Plex, бот показывает предупреждение:
   ```
   ⚠️ «Dune» уже есть в Plex (1080).
   Скачать всё равно?
   [⬇️ Скачать]  [✖️ Отмена]
   ```
   Для сериалов сообщение учитывает номер сезона: `⚠️ Сезон 3 «Клиника» уже есть в Plex (1080)`. Сообщение адаптируется к ситуации: «уже есть в лучшем качестве», «есть в худшем — скачать для замены?» и т.д. Работает для поиска через Rutracker/Jackett, а также для magnet-ссылок и `.torrent`-файлов (в последних двух случаях качество и название определяются по имени файла). Проверка для сериалов требует чтобы у Plex была настроена TV-секция (`type="show"`).

3. **Иконки `✅` в пикере сезонов** — после добавления одного сезона при нажатии «🔎 Другой сезон» бот показывает кнопки 1…N с пометкой `✅` для тех сезонов, которые уже есть в Plex. В тексте над кнопками — строка «В Plex: 1, 2, 3» (или «Все сезоны уже в Plex»). Сезон с галочкой остаётся кликабельным — можно перекачать для апгрейда качества.

4. **Уведомление когда контент появился в Plex** — после завершения скачивания (`finished`) бот опрашивает Plex каждые 30 секунд до 10 минут. Когда файл индексируется, приходит новое сообщение-пуш:
   ```
   ✅ Dune: Part Two добавлен в Plex.
   [▶️ Смотреть в Plex]
   [✖️ Закрыть]
   ```
   Для сериалов — `✅ Сезон 3 «Клиника» добавлен в Plex`. Если контент не появился за 10 минут — приходит предупреждение. Если Plex всё это время был недоступен — приходит уведомление с другим текстом («сервер был недоступен»), чтобы не сваливать причину на индексацию.

   **Lookup для сериалов.** Plex кэширует show под годом премьеры (например, «Благие знамения» — 2019), а `meta.year` бот извлекает из release-имени файла (год эпизода, например 2026). Поэтому year-bounded lookup с ±1 tolerance может не сработать. В этом случае `_plex_show_find` делает **fallback по нормализованному title** (без учёта года) — этого достаточно если name в Plex и `series_query` бот совпадают. Без этого fallback push-уведомление «✅ Сезон N «X» добавлен в Plex» вообще никогда не приходил бы для сериалов с разрывом более года между премьерой и эпизодом.

   **Куда ведёт кнопка «Смотреть в Plex».** Поведение зависит от env-переменной `PLEX_DEEPLINK_BASE_URL`:

   - **Не задано (по умолчанию)** — кнопка ведёт на `https://app.plex.tv/desktop/#!/server/{machineId}/details?key=...`. Plex Web показывает контент с вашего сервера. **На iOS это открывается в Safari**, не в нативном приложении: Plex не публикует [Universal Links](https://support.plex.tv/articles/218168898-universal-links/) для `app.plex.tv` (проверил их AASA — для пути `/desktop` ничего не зарегистрировано). Telegram при этом не принимает `plex://` в inline-кнопках с мая 2026 (`HTTP 400: unsupported url protocol`), поэтому прямо открыть приложение из кнопки невозможно — нужен redirect.

   - **`PLEX_DEEPLINK_BASE_URL=https://your-host/plex.html`** — бот формирует кнопку с URL `https://your-host/plex.html?key=/library/metadata/{ratingKey}&server={machineId}`. По этому адресу вы хостите крошечную HTML-страничку, которая JS-ом делает `location.href = "plex://..."`. Safari/Chrome принимают `plex://` через `location.href` (это разрешено) → iOS/Android запускают **нативное Plex-приложение**.

     Минимальный `plex.html`:
     ```html
     <!DOCTYPE html>
     <html><body>
     <script>
       const p = new URLSearchParams(location.search);
       const key = p.get('key'), server = p.get('server');
       if (key && server) {
         location.href = `plex://preplay/?metadataKey=${encodeURIComponent(key)}&metadataType=1&server=${server}`;
       } else {
         location.href = 'plex://';
       }
     </script>
     <p>Открываю Plex…</p>
     </body></html>
     ```

     Хостить можно где угодно: Synology DSM Web Station, GitHub Pages, Cloudflare Pages, отдельный nginx на NAS. Главное — публичный https URL который iOS Safari сможет открыть.

   **Где бот берёт точное название для сверки.** При добавлении задачи в Download Station (через поиск, magnet, `.torrent`, KP-ссылку, подписку Rutracker/Jackett) бот сохраняет канонические метаданные в `state/task_meta.json`: `kind` (movie/series), `title`, `year`, `quality`, `series_query`, `season_num`. При завершении задачи polling использует именно эти данные для лукапа в Plex — а не сырое имя торрента из DS. Это значит что фильмы с кириллицей в имени файла или сериалы с переименованными Plex-агентом файлами тоже корректно находятся.

5. **Диагностика в `/admin`** — блок Plex в разделе «Диагностика»: доступность сервера, количество фильмов и сериалов в кэше (`Фильмов: N · Сериалов: M`), время последнего обновления. Если в библиотеке есть несматченные файлы — добавляется строка `Не сматчено: N фильмов, M сериалов`. Если фоновый refresh библиотеки последний раз падал — отображается **тип** ошибки (`ошибка авторизации` → проблема с `PLEX_TOKEN`, `таймаут` → сервер не отвечает, `не удалось подключиться` → проверьте `PLEX_URL`/сеть, `некорректный ответ` → не-XML), счётчик подряд неудач и время последнего успешного refresh — это позволяет понять что именно сломалось без проверки логов.

6. **Радар несматченных файлов в Plex** — отдельный блок в `/admin` который показывает файлы, для которых ни один Plex-агент (для фильмов это обычно Kinopoisk, для сериалов — Plex TV) не смог найти метаданные. Такие файлы попадают в библиотеку без обложки, года и описания.
   
   - **Кнопка `📋 Plex: без матча (N)`** в `/admin` — открывает полный список: имена файлов из обеих секций (фильмы + сериалы), до 25 в каждой категории, дальше «и ещё K». Список доступен всегда, по запросу, независимо от настроек авто-сообщений.
   - **Кнопка `🔔 Подписка: вкл / 🔕 Подписка: выкл`** — переключатель авто-сообщений в Telegram о новых несматченных файлах. При первом включении админу прилетает сводка обо всех текущих несматченных. Дальше — только при появлении новых файлов (после следующего рефреша библиотеки). Дедупликация хранится в `state/movie_discovery_settings.json` (поле `plex_unmatched_seen`), так что временное выключение не вызывает спам при возврате.
   - Критерий «несматчено»: атрибут `guid` записи в Plex равен `local://…` или пуст. Точность ~99% — Plex однозначно помечает такие записи, как только обратно сматчите файл вручную (через Plex Web UI → «Match…») — он исчезает из списка после следующего рефреша библиотеки бота (раз в 30 мин).

**Как получить Plex Token:**

Откройте браузер, войдите в Plex Web (`app.plex.tv`), откройте DevTools → Network, найдите любой запрос к `plex.tv` или к вашему серверу — в параметрах URL будет `X-Plex-Token=xxxx`.

**Кэш библиотеки** обновляется раз в 30 минут в фоне — значок ✅, проверка дубликатов и радар несматченных используют именно его.

**KP-квота при включённой интеграции с Plex.** Если у вас в Plex настроен Kinopoisk-агент, он использует тот же ежедневный лимит kinopoiskapiunofficial.tech (по умолчанию ~500 запросов/день) что и бот. Plex-агент тратит ~1–2 KP-вызова на каждый новый фильм при первом скане. Если активно добавляете медиатеку через бота — рекомендуется уменьшить `KP_API_DAILY_LIMIT` в `.env` с запасом (например 400 вместо 500), чтобы оставить квоту для Plex.

## Админ-панель `/admin`

Команда `/admin` открывает компактную панель администратора. Главный экран разбит на три блока:

**📊 Состояние** — оперативные данные прямо сейчас:
- сводка по задачам Download Station: всего, активных, завершённых и с ошибкой
- количество подписок Rutracker и Jackett, время следующей проверки

**⚙️ Правила и интеграции** — конфигурация из `.env`:
- компактная строка состояния интеграций с цветовыми точками: `🟢 Rutracker · 🟢 Jackett · 🟢 Кинопоиск · 🟢 Plex · 🟢 Трекеры` (🔴 — не настроен)
- автоудаление, уведомления, параметры поиска по умолчанию
- *Живой статус сервисов — в разделе «Диагностика»*

**🎬 Новинки** — короткая сводка блока `/new` (компактный вид для мобильника):
- статус (включены / выключены)
- кэш с датой обновления и количеством карточек
- строка бюджета KP API (только если Кинопоиск включён)
- подписки на `/new` (только если есть)
- *Полная конфигурация фильтров, трекеров рейтинга и управление KP кэшем — по кнопке «🎬 Новинки»*

**📀 Хранилище** — занятость видеохранилища (показывается только если в `compose.yaml` есть bind-mount `/storage`):
- занято из общего объёма + % — иконка 📀 в норме, ⚠️ при превышении `STORAGE_ALERT_PERCENT` (default 90%)
- прогноз «~N дней до порога» по линейной экстраполяции от снепшота недельной давности (появляется после нескольких суток наблюдений)
- при пересечении порога вверх админам приходит **один** push-алерт; счётчик сбрасывается когда usage падает ниже — следующая граница снова триггернёт уведомление

Кнопки панели:

- **🔄 Обновить** — перечитать главную сводку
- **🧭 Диагностика** — полная диагностика всех сервисов с живыми проверками
- **👥 Пользователи** — управление доступом
- **🔔 Подписки** — список подписок всех пользователей с владельцами и кнопками удаления
- **📋 Загрузки** — перейти к общей панели загрузок
- **🎬 Новинки** — drill-down экран с полной конфигурацией: источники, фильтры качества, минимальный рейтинг КП, интервал автообновления, диапазон Rutracker, ограничения Jackett, разбивка трекеров рейтинга 🟢/🔴, KP API статистика (бюджет сегодня + размер кэша). Внутри: кнопки **🔄 Обновить KP кэш** и **🗑 Очистить KP кеш** — показываются только при настроенном `KINOPOISK_API_KEY`
- **🎬 Трекеры новинок** — экран выбора трекеров Jackett для рейтинга `/new`; чекбоксы ✅/☐ на каждый трекер (включённые вверху); переключение вступает в силу мгновенно без перезапуска; кнопка «✅ Включить все» сбрасывает фильтр
- **✖️ Закрыть** — удалить сообщение панели из чата; появляется краткое уведомление «Закрыто», которое исчезает через 3 секунды (аналогично — «Отменено» при нажатии «❌ Отмена» в диалогах поиска)

**Диагностика** проверяет только сервисы, к которым бот реально обращается:
- Download Station, Rutracker, Jackett — живое соединение
- Public-трекеры — состояние кэша: **кэш готов** (свежий), **кэш доступен (устарел)** (рабочий, но давно не обновлялся — это норма при on-demand режиме), **кэш пуст** (загрузится при первом BT-торренте), **кэш недоступен** (ошибка чтения)

Кинопоиск не входит в диагностику — его состояние уже отражено в строке интеграций на главном экране (`🟢/🔴`). Plex — отдельный блок в диагностике: доступность сервера (`GET /identity`), количество фильмов в кэше, время последнего обновления.

Plex deep-link — проверяет доступность redirect-страницы, указанной в `PLEX_DEEPLINK_BASE_URL` (`GET` с таймаутом 5с, ищет в теле маркер `plex://`). От неё зависят все кнопки «▶️ Открыть/Смотреть в Plex» в чате: если redirect упал, ссылки становятся мёртвыми. Статусы: **ok** (доступен), **warn** (200, но контент не похож на redirect-страницу), **error** (HTTP-ошибка / таймаут / сеть), **disabled** (`PLEX_DEEPLINK_BASE_URL` пуст → используется публичный `https://app.plex.tv/desktop`, проверка не нужна).

Отдельной команды для диагностики нет, только через `/admin`.
